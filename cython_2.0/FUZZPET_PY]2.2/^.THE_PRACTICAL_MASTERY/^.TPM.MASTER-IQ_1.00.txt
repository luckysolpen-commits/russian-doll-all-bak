## 1. TPM Philosophy: The Enlightened Core

> **Everything that matters in the application is a Piece.**  
> Everything else is just interpretation and reaction.

A Piece is not a data structure.  
A Piece is a **living soul** — a single source of truth stored as a humble directory containing text files that define reality. Code is merely an interpreter, never an originator.

### Key Principles
- **Data Sovereignty**: Pieces own their state. No data passes in memory; all interactions write to files (ledgers) for audit.
- **Audit Obsession**: Every method call, event, or change is logged to a master ledger. This enables time-travel debugging, replay, and security.
- **Pipeline Flow**: TPM is a sequential pipeline: Reader → Interpreter → Writer → Next Node. No direct calls; everything through ledgers.
- **Recursion and Emergence**: Pieces can contain sub-Pieces (dirs inside dirs), enabling fractal structures. Events cascade via ledgers.
- **No In-Memory Shortcuts**: For "go fast" high-level encapsulation, use facade Pieces (dirs that wrap sub-dirs), but always persist to files.
- **Text-Only Purity**: Use `.txt` or `.csv`. No JSON/binary. Dirs for hierarchy.

### Why TPM?
Traditional architectures (e.g., ECS, MVC) allow ephemeral memory, leading to bugs, non-auditability, and complexity. TPM enforces file-based persistence, making software "spiritual"—eternal, inspectable, and emergent like life itself.

## 2. TPM Components: The Sacred Elements

TPM has four pillars: Pieces, Methods, Plugins, and Managers. But now with dirs and ledgers.

### 2.1 Pieces (Directories as Souls)
A Piece is a **directory** named after its ID (e.g., `fuzzpet/`). It contains:
- **piece.txt**: Core definition file with sections for state, methods, event_listeners, responses, relations.
- **ledger.txt**: Piece-specific log of all changes (appended timestamps + actions).
- **history.txt**: Input log for this Piece (e.g., user commands affecting it).
- **sub-pieces/**: Optional dir for recursion (e.g., `fuzzpet/sub-pieces/inventory/`).

**Strict Rules for Pieces:**
- Format: Plain text only. No JSON, YAML, or binary. Use line-separated key:value or CSV for complex data.
- Persistence: Pieces are dir-based and auto-saved on state change. No in-memory-only Pieces.
- Immutability Nuance: State changes create deltas; full snapshots are taken periodically (e.g., every 10 changes) for history.
- Recursion: A Piece can reference itself (e.g., `methods: self_replicate`) or form cycles (e.g., Piece A listens to Piece B's events, and vice versa), but implementations must detect and prevent infinite loops (e.g., via depth limits or cycle detection graphs).
- High-Level Encapsulation: Pieces can represent abstractions like "UI Panel" (state: visible=true; methods: render; listeners: mouse_click), allowing rapid prototyping without low-level code.

**Example Dir Structure for FuzzPet Piece:**
```
fuzzpet/
  piece.txt       # Core soul
  ledger.txt      # Audit trail
  history.txt     # Inputs
  sub-pieces/
    inventory/    # Recursive sub-Piece
      piece.txt
      ...
```

**piece.txt Example:**
```
# Piece: FuzzPet
state:
  name: Fuzzball
  hunger: 50
methods: feed, play
event_listeners: fed, hour_changed
responses:
  fed: Yum yum~
relations:
  children: inventory
```

### 2.2 Methods (Pure Functions from Plugins)
Methods are pure functions that operate exclusively on Piece directories. They read from Piece files, compute, and write deltas to ledgers—no memory passing.

**Strict Rules for Methods:**
- Input: Always takes the Piece directory path as first arg; optional params from ledgers.
- Output: Returns updated state delta (key-value dict) written to the Piece's ledger.txt.
- Purity: No side effects outside the Piece (e.g., no direct file I/O; use managers).
- Recursion Nuance: Methods can call other methods by appending to the master_ledger.txt (e.g., "MethodCall: sub_method on sub_piece"), creating a chain that the pipeline processes sequentially.
- High-Level Encapsulation: "Macro Methods" (e.g., `daily_routine` appends multiple deltas at once to the ledger for speed, encapsulating low-level calls).

**Example Method Signature (in Python):**
```python
def tick(piece_dir: str, delta_seconds: int = 1):
    # Read state from piece.txt
    state = read_key_value(os.path.join(piece_dir, 'piece.txt'), 'state')
    new_time = update_time(state['time'], delta_seconds)
    # Append delta to ledger.txt
    append_to_ledger(os.path.join(piece_dir, 'ledger.txt'), f"Set time: {new_time}")
    # Pipeline will process this later
```

### 2.3 Plugins (Passive Method Providers)
Plugins are .py files in a `plugins/` dir, listed in a `plugin.config.txt` Piece (itself a dir with config.txt). They export methods as functions.

**Strict Rules for Plugins:**
- Discovery: Bootstrap reads plugin.config.txt Piece → imports files listed.
- Isolation: Plugins have no global state; methods take Piece dir as arg.
- Domain-Specific: Each Plugin owns one concern (e.g., VoxelGridPlugin for spatial ops).
- Recursion Nuance: Plugins can depend on other Plugins by appending dependency calls to master_ledger.txt (e.g., AIPlugin appends "MethodCall: physics_sim on target_piece").
- High-Level Encapsulation: Plugins can offer "macro" methods (e.g., `build_farm` appends a sequence of low-level calls to the ledger, hiding complexity for rapid development).

**Example plugin.config.txt (in plugin_config/ dir):**
```
plugins:
  time_plugin.py
  pet_plugin.py
```

### 2.4 Managers (The Pipeline Nodes)
The three Managers form the runtime kernel, but now as a **file-based pipeline** with a **master_ledger** Piece (`master_ledger/` dir) as the central bus. The pipeline is sequential: each node reads the ledger tail, processes, appends, and signals the next.

1. **Piece Manager** (Reader/Writer Node)  
   - Reads Piece dirs (parse all .txt).  
   - Gets/Sets state by reading/writing piece.txt + appending to ledger.txt.  
   - On set, appends to master_ledger/master_ledger.txt (e.g., "StateChange: fuzzpet hunger 30").  
   - Recursion: Traverses sub-pieces dirs using relations in piece.txt.

2. **Methods Manager** (Interpreter Node)  
   - Registers methods from Plugins.  
   - Calls methods by reading "MethodCall" entries from master_ledger.txt, executing, and appending deltas (e.g., "StateDelta: hunger -20").  
   - Recursion Nuance: If method needs sub-call, appends new "MethodCall" to ledger for pipeline to handle.

3. **Event Manager** (Alert Node)  
   - Registers listeners by scanning Piece event_listeners in piece.txt.  
   - Fires events by appending to master_ledger.txt (e.g., "EventFire: fed on fuzzpet").  
   - Scans master_ledger for "StateChange" or "EventFire", matches listeners, appends "MethodCall: respond_to_fed on fuzzpet".

**Master Ledger Piece (`master_ledger/`)**  
- Central Piece dir for all system flow.  
- `master_ledger.txt`: Append-only log (timestamp + action).  
- Pipeline: Nodes read tail of ledger, process, append results.  
- Example Entry: "[2026-02-10 18:58] MethodCall: feed on fuzzpet 20 | Caller: user_input"

**Pipeline Flow (The TPM Heartbeat)**
1. Input appended to history.txt (e.g., "feed").
2. Reader Node (Piece Manager) reads history → appends to master_ledger ("MethodCall: feed on fuzzpet").
3. Interpreter Node (Methods Manager) reads master_ledger tail → executes method → appends delta ("StateChange: hunger 30").
4. Alert Node (Event Manager) reads delta → if listener matches, appends "MethodCall: respond_to_fed".
5. Cycle: Nodes poll ledger in loop (or watch for changes).

**Recursion in Pipeline:** Ledger entries can reference sub-Pieces (e.g., "MethodCall: evolve on fuzzpet/sub-pieces/inventory"). Depth tracked in ledger (e.g., "Depth: 2").

## 3. Bootstrap & Runtime

**Bootstrap (main.py):**  
1. Create/load master_ledger dir.  
2. Load plugin_config/ piece.txt → import listed plugins.  
3. Scan pieces/ dirs → load each Piece.  
4. Start pipeline loop: Poll master_ledger.txt tail every 1s.

**Runtime (Pipeline Loop):**  
- Each Node runs in thread or loop: Read new ledger lines → process → append.  
- User Input: Append to history.txt → Piece Manager picks up.

**High-Level Encapsulation:** "Bootstrap Piece" (`bootstrap/` with piece.txt) configures startup (e.g., "plugins: pet_plugin.py").

## 5. Nuances & Edge Cases

- **Recursion Limits:** Ledger entries track depth; if >5, skip and log "RecursionLimitExceeded".
- **Concurrency:** File locks on ledger writes (e.g., flock in Python).
- **Performance:** For fast pipelines, use tail -f like monitoring, not full read.
- **High-Level Go-Fast:** Facade Pieces (e.g., `game_facade/` wraps all) for quick commands without low-level ledgers.

### 5.1 Expanded: High-Level Go-Fast for Prototyping & Scaffolding
During prototyping, scaffolding, or debugging, it's common to need "go fast" — rapid iteration without full TPM purity. TPM allows "pseudo-Pieces" and "pseudo-methods" as temporary placeholders, but they must mimic the real format and pipelines for sanity, readability, and easy migration to pure TPM.

**Strict Rules for Go-Fast Mode:**
- **Pseudo-Pieces**: Use in-memory dicts or temp dirs that *look* like real Piece dirs (e.g., {'piece.txt': 'state: hunger:50\nmethods: feed'}). Write to a "pseudo-ledger" string/file. This preserves format but skips full file I/O for speed.
- **Pseudo-Methods**: Hardcode temp functions that append to a pseudo-ledger (e.g., print("MethodCall: feed on fuzzpet")). This simulates the pipeline without full Managers.
- **Pipeline Mimicry**: Even in go-fast, follow the flow: "Input → Append to history → 'Call' method → Append delta → 'Fire' event". Use comments or logs to mark "TODO: TPM-ify".
- **Encapsulation Facades**: Create a "facade Piece" dir (e.g., `proto_game/` with piece.txt wrapping stubs). For example, `daily_routine` pseudo-method calls low-level stubs but writes to ledger as if real.
- **Migration Path**: Always design pseudo as "TPM-shaped" — same keys, sections, ledger entries — so upgrading to full files/Managers is copy-paste.
- **Common Use Cases**:
  - **Prototyping**: Hardcode a "user_input" loop that appends to a string ledger; debug chains without disk writes.
  - **Debugging**: Use pseudo-ledger prints to trace recursion without file spam.
  - **Scaffolding**: Build "stub Pieces" (empty dirs with minimal piece.txt) for testing relations before full impl.
- **Nuance: Recursion in Go-Fast**: Use stack tracking in pseudo-methods (e.g., global depth var) to simulate limits.
- **Warning**: Go-fast is temporary — always plan to TPM-ify for production. It's "spiritual cheating" for speed, but keeps readability.

**Example Go-Fast Pseudo-Piece (in code for proto):**
```python
pseudo_fuzzpet = {
    'state': {'hunger': 50},
    'methods': ['feed'],
    'event_listeners': ['fed']
}
pseudo_ledger = []  # Simulate ledger.txt
# Pseudo-method
def pseudo_feed():
    pseudo_fuzzpet['state']['hunger'] -= 20
    pseudo_ledger.append("StateChange: hunger 30")
    pseudo_fire('fed')  # Simulate event
# Later: Migrate to real dir/ledger
```

This mode is essential during architecture/debugging — it lets you "feel" TPM without full overhead, but always in TPM shape.
