## 1. HARDCODED METHODS/TRAITS

### Category: High Severity (blocks modularity)

[file: playrm_module.c:163-188] - Hardcoded method dispatch via strcmp chain: "scan", "collect", "place", "inventory", "interact", "inspect", "toggle_emoji"
  | Each method name is hard-compared against a string literal, then routed to a hardcoded .+x path.
  | Should be PDL-driven because: the method->op mapping should be resolved from the piece's PDL method table, not from an if/else chain. Adding a new method requires editing C code and recompiling. [file: playrm_module.c:164-180] - Hardcoded trait paths: `pieces/apps/playrm/ops/+x/scan_op.+x`, `collect_op.+x`, `place_op.+x`, `inventory_op.+x`, `interact.+x`, `inspect_tile.+x`

### Top 3 files to refactor first:

1. **`/home/jb/Desktop/ele-desk+cal/apr_5/TPMOS_c_+rmmp_62.01/pieces/chtpm/plugins/chtpm_parser.c`** -- 20+ violations. The single largest source of hardcoded project routing, module path resolution, and active_target assignment. 2. **`/home/jb/Desktop/ele-desk+cal/apr_5/TPMOS_c_+rmmp_62.01/pieces/apps/playrm/plugins/playrm_module.c`** -- 12+ violations. Hardcoded method dispatch table, hardcoded op paths, hardcoded project type checks. 3. **`/home/jb/Desktop/ele-desk+cal/apr_5/TPMOS_c_+rmmp_62.01/pieces/os/project_routes.kvp`** -- While a config file, this manual project registry is the gatekeeper for project loading. Replacing it with auto-discovery (scan projects/ for project.pdl, read `entry_layout` from each) would eliminate 404 errors for unregistered projects and remove the synchronization burden between adding a project and registering its route.

### Additional recommendations:

- **Unify `piece_manager.+x` path resolution**: 7+ files hardcode this path. Define it once in a system config or resolve it relative to project_root.
- **Replace `call_op()` hardcoded ops path**: The `call_op()` function in playrm_module.c should accept a trait path prefix that is resolved from PDL or project config, not hardcoded to `pieces/apps/playrm/ops/+x/`.
- **Automate project_routes.kvp generation**: A build-time or startup-time script that scans `projects/*/project.pdl` for `entry_layout` entries and regenerates the KVP would eliminate manual registration.
- **Type registry PDL**: The hardcoded glyph<->type mappings in render_map.c and create_piece.c should move to a `types.pdl` that maps `type_name = glyph` bidirectionally.
