3. THE REVISED PLAN: "OPS FIRST"
================================================================================

Instead of Prisc scripts, we have **Op Libraries**:

    pieces/apps/playrm/
    ├── playrm_module.c          # The Bridge (like fuzzpet_manager.c)
    └── ops/                     # Reusable Ops
        ├── move_player.c        # w/a/s/d movement
        ├── move_z.c             # x/z floor changes
        ├── interact.c           # ENTER on entities
        ├── place_tile.c         # Editor: paint tile
        ├── pick_tile.c          # Editor: select tile
        └── render_map.c         # Compose view.txt

Each Op is a +x binary that:
- Takes arguments (piece_id, direction, etc.)
- Reads/writes piece state.txt
- Returns success/failure

Example Op call from playrm_module.c:
    // On 'w' key:
    system("./pieces/apps/playrm/ops/+x/move_player.+x selector up");
    
    // On ENTER:
    system("./pieces/apps/playrm/ops/+x/interact.+x selector");

================================================================================
4. HOW THIS IS DIFFERENT FROM FUZZPET
================================================================================

FUZZPET (Current):
- Traits are HARDCODED in manager (switch-case calls specific traits)
- Only works for fuzzpet entities

PLAYRM (Revised):
- Ops are GENERIC (work for any piece with pos_x, pos_y, pos_z)
- Module reads piece.pdl to discover available methods
- Methods point to Ops dynamically

Example piece.pdl:
    # Piece: chest_01
    STATE: open=0, pos_x=5, pos_y=3
    METHOD: open_chest → +x/chest_opener.+x
    METHOD: examine → +x/examine_chest.+x

The module doesn't hardcode "chest" - it just calls whatever Op the piece defines.
