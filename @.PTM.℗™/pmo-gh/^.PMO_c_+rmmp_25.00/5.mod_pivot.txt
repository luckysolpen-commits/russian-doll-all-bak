1. WHAT WENT WRONG (POST-MORTEM)
================================================================================

My previous implementation failed due to three critical misunderstandings:

### 1. The Suspension Fallacy
I assumed the `chtpm_parser` would suspend (block) while a module ran. In
reality, the `<module>` tag implementation (v4.0) allows side-by-side
concurrent execution. My previous PAL integration strategy would have caused
"Ghost Input" because both the parser and the module would be competing for
the same history stream without clear ownership.

**Lesson:** Parser and module run CONCURRENT. Input must be routed exclusively
to the active module. Parser only handles navigation when NOT in INTERACT mode.

### 2. Path Resolution Ignorance
I used hardcoded relative paths (e.g., `./pieces/...`) which are brittle.
The correct "fixed" code uses dynamic resolution via `location_kvp` to find
the `project_root`, ensuring the system works regardless of where it's
launched from.

**Lesson:** Always use `build_path_malloc()` with `location_kvp` resolution.
Never hardcode `./pieces/...` paths.

### 3. State Persistence Failure
I overlooked that `emoji_mode` and `clock_mode` need to be synced to the
Mirror (`state.txt`) AND the DNA (`.pdl`) to survive restarts and be visible
to the renderer.

**Lesson:** Every state change MUST sync both DNA (.pdl) and Mirror (state.txt).
The Mirror Sync Mandate is not optional.

### 4. The Over-Engineering Trap (NEW)
I proposed adding a Prisc VM layer between the module and the Ops. This was
unnecessary complexity. FuzzPet's trait system ALREADY works perfectly -
traits ARE Ops, just called directly from C.

**Lesson:** If it works (FuzzPet traits), extend it. Don't add layers of
indirection (Prisc VM) unless absolutely necessary. KISS.

2. THE GUIDELINE: v4.0 Sovereign App Pattern
================================================================================

The fixed FuzzPet implementation provides the template for how PLAYR/RMMP
should function:

### Asynchronous Launch
Apps are launched via `<module>` tags in the `.chtpm` layout. The parser
continues running concurrently.

### IPC via Filesystem
Input is "injected" into the module's specific `history.txt`. The module
polls this file continuously.

### Mirror Sync Mandate
Every state change must be flushed to the piece's `state.txt` for the
renderer to see it immediately. DNA (.pdl) is the truth, Mirror (state.txt)
is for speed.

### Sovereign Signaling
Modules hit `view_changed.txt` to trigger a specific re-render, avoiding
global menu flickering. Also hit `frame_changed.txt` for global sync.

### Ops Over VM
Instead of Prisc scripts calling Ops, the module calls Ops directly.
This is simpler, faster, and proven by FuzzPet.

3. THE CORRECT PLAN (The Sovereign Pivot - Ops-First)
================================================================================

### [ENGINE] Shared Foundation
- **Refactor `chtpm_player.c`** to fully adopt the `<module>` tag pattern.
  It will no longer try to "be" the manager; it will be the **Theater**
  that hosts the Ops Bridge.
- **Dynamic Pathing:** Update all custom Ops to resolve paths via
  `location_kvp`.
- **Op Library:** Create `pieces/apps/playrm/ops/` with reusable +x binaries.

### [EDITOR] RMMP (Ops-Based)
- **Editor Module:** `playrm_module.c` reads piece.pdl to discover available
  methods, then calls appropriate Ops directly.
- **Paint Op:** `place_tile.+x` writes to map.txt and hits `view_changed.txt`
  so the map updates instantly.
- **Picker Op:** `pick_tile.+x` reads tile at cursor, sets selected_tile.

### [PLAYER] PLAYR (Ops-Based)
- **Project Selection:** `project_loader.c` overwrites the `project_id` in
  `manager/state.txt` using robust buffer-concatenation.
- **Ops-Driven UI:** Module calls Ops directly based on piece.pdl methods.
  No Prisc VM layer needed.
- **Transition:** Title Screen → Gameplay via Op calls (no state machine).

4. OPS LIBRARY (Planned)
================================================================================

### Movement Ops
- `move_player.+x` - Generic movement (any piece with pos_x/pos_y)
- `move_z.+x` - Z-level changes (up/down floors)
- `teleport.+x` - Absolute position set (map, x, y, z)
- `set_spawn.+x` - Set player spawn point

### Interaction Ops
- `interact.+x` - Generic interaction (checks adjacent entities)
- `open_chest.+x` - Opens chest, shows treasure
- `talk_npc.+x` - Shows dialogue from NPC
- `use_item.+x` - Uses item from inventory

### Editor Ops
- `place_tile.+x` - Writes char to map.txt at (x, y)
- `pick_tile.+x` - Reads char from map.txt at (x, y)
- `place_piece.+x` - Creates new piece .pdl at (x, y)
- `delete_piece.+x` - Removes piece .pdl

### Rendering Ops
- `render_map.+x` - Composes view.txt from map + entities
- `render_minimap.+x` - Creates mini map view
- `render_status.+x` - Shows HP/MP/gold status

### System Ops
- `save_game.+x` - Writes state to save file
- `load_game.+x` - Reads state from save file
- `show_message.+x` - Displays message to user
- `log_event.+x` - Logs to master_ledger.txt

Each Op:
- Is a +x binary (compiled from .c source)
- Accepts piece_id as first argument
- Uses dynamic path resolution (location_kvp)
- Hits frame_changed.txt after state change
- Logs to master_ledger.txt
