THE CONFUSION: #.main vs #.docs
================================================================================

**#.main/ (System Specifications)**
Purpose: Technical specifications for SYSTEM COMPONENTS
Contains:
  - fondu-how2.txt, fondu-kpis.txt ← Fondu system specs
  - migration-status.txt ← System migration tracking
  - tpm-functional-principles.txt ← Architecture principles
  - gl-os-architecture.txt ← GL-OS system spec
  - gl-os-layout-fix.txt ← Technical fix documentation
  - p2p-ops.txt ← P2P system specification (1491 lines)
  - op-ed-ops-section.txt ← Op-ed ops spec

**Rule:** If it's a TECHNICAL SPEC for a system component → #.main/

**#.docs/ (User Documentation)**
Purpose: Documentation for HUMANS using the system
Contains:
  - AGENT_ONBOARDING.txt ← "How to start as a developer"
  - TPMOS_APP_REFERENCE.txt ← "What each app does"
  - OPS_DICTIONARY.txt ← "What ops are available"
  - COMPILE_SYSTEM.md ← "How to compile"
  - INDEX.md ← "Documentation hub"
  - PITFALLS_ACTIVE_*.txt ← "What not to do"
  - investigation-report.txt ← "What we found"

**Rule:** If it's for PEOPLE learning/using the system → #.docs/

================================================================================
DOCUMENTATION TYPE DECISION TREE
================================================================================

"Where does this doc go?"

1. Is it a TECHNICAL SPEC for a system component?
   → YES: #.main/SYSTEM-SPECS/<component>/
   → NO: Continue

2. Is it for PEOPLE learning/using the system?
   → YES: #.docs/<category>/
   → NO: Continue

3. Is it about FUTURE plans/roadmap?
   → YES: #.plans/
   → NO: Continue

4. Is it a SCRIPT for building/testing?
   → YES: #.tools/
   → NO: Continue

5. Is it a TEST plan or result?
   → YES: #.testing/
   → NO: Continue

6. Is it REFERENCE CODE?
   → YES: #.ref/ (once populated)
   → NO: Where did this come from?!
