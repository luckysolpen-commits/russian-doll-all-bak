# 18.man-pal.txt - The PAL-Ops Pipeline Plan (Track 2)
# Date: 2026-03-12
# Goal: Prove the "Brain" layer by delegating decision-making to a PAL script.

## 1. THE HIERARCHY OF CONTROL
Unlike `man-ops` where the C module decides which Op to call, `man-pal` introduces a middleman:
**Module (Relay) -> Prisc VM (Brain) -> move.asm (Script) -> move_entity.+x (Muscle)**

## 2. THE INFRASTRUCTURE (C Module)
`pieces/apps/man-pal/plugins/man-pal_module.c`:
- **Thinness:** The module will have ZERO knowledge of movement or keys.
- **Polling:** It polls `pieces/apps/player_app/history.txt` (the unified buffer).
- **Execution:** For every key `K`, it executes:
  `PRISC_KEY=K PRISC_PROJECT_ID=man-pal ./prisc+x move.asm`
- **Compile Clean:** All paths and commands constructed via `asprintf`.

## 3. THE BRAIN (PAL Script)
`projects/man-pal/scripts/move.asm`:
- **Retrieval:** Reads the `PRISC_KEY` environment variable (standard Prisc interface).
- **Logic:** Uses assembly branches (`beq`, `bne`) to decide direction.
- **Delegation:** Calls the standardized muscle via the `exec` instruction:
  `exec ./move_entity.+x selector up man-pal`
- **Signaling:** Hits the frame marker using the `hit_frame` instruction.

## 4. THE LAYOUT (CHT-PM)
`pieces/apps/man-pal/layouts/man-pal.chtpm`:
- Identical to `man-ops.chtpm` but points to the `man-pal_module.+x`.
- Uses `<interact src="pieces/apps/player_app/history.txt" />`.

## 5. PROJECT STANDARDIZATION
- Create `projects/man-pal/` directory.
- Instantiate `project.pdl` with `starting_map=map_01`.
- Ensure `pieces/selector/state.txt` is present and correct.

## 6. WHY THIS MATTERS
If `man-pal` works, we have proven that the entire logic of a project can be swapped or modified by simply editing a text-based `.asm` file, without ever touching or recompiling the C code. This is the foundation for User Moddability and the "Sovereign Mind" pieces.

"The C module provides the pulse; the PAL script provides the purpose."
