## 2. TOP 3 HYPOTHESES

### Hypothesis 1: Instruction Mismatch (Prisc VM)
- **Details:** The current `move.asm` uses `read_env_key`. Upon inspection of `prisc+x.c`, this instruction does not exist in the switch/case or custom op handlers.
- **Solution:** Revert `move.asm` to use `read_history r1` (which is supported and consumes input) or ensure the VM is compiled with the latest instructions.

### Hypothesis 2: State Sync Race (Module vs. Op)
- **Details:** `man-pal_module.c` calls `write_pal_state()` and `trigger_render()` immediately after the `system()` call to Prisc. If Prisc launches background ops, the module might be syncing state BEFORE the op actually writes to disk.
- **Solution:** Ensure the PAL script hits the frame marker and that the module performs a final sync AFTER the `system()` call returns.

### Hypothesis 3: Project Buffer Routing
- **Details:** `chtpm_parser.c` injects into `player_app/history.txt`. The `prisc+x` VM's `OP_READ_HISTORY` might be defaulting to a different path if `PRISC_PROJECT_ROOT` isn't perfectly resolved.
- **Solution:** Explicitly pass the history path to the `read_history` instruction in the `.asm` file.
