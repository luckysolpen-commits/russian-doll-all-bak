## 2. IDENTIFIED BUGS

### BUG #1: Missing `last_key` Update in MAN-PAL Module
**Severity:** CRITICAL  
**Location:** `pieces/apps/man-pal/plugins/man-pal_module.c:process_key()`  
**Symptom:** UI shows "KEY: None" even when keys are pressed

**Root Cause:**
The man-pal module calls Prisc VM but never updates `last_key_str` for the UI display.
The `write_pal_state()` function writes `last_key_str` to state files, but it's never 
populated with the actual key value.

**Fix:**
Add key string conversion BEFORE calling Prisc:
```c
void process_key(int key) {
    // ADD THIS: Convert key to string for UI display
    if (key >= 32 && key <= 126) snprintf(last_key_str, sizeof(last_key_str), "%c", (char)key);
    else if (key == 10 || key == 13) strcpy(last_key_str, "ENTER");
    else if (key == 27) strcpy(last_key_str, "ESC");
    else if (key == 1000) strcpy(last_key_str, "LEFT");
    else if (key == 1001) strcpy(last_key_str, "RIGHT");
    else if (key == 1002) strcpy(last_key_str, "UP");
    else if (key == 1003) strcpy(last_key_str, "DOWN");
    
    // Then call Prisc VM
    char *cmd = NULL;
    if (asprintf(&cmd, "cd %s && PRISC_PROJECT_ID=%s PRISC_PROJECT_ROOT=%s ./pieces/system/prisc/prisc+x projects/man-pal/scripts/move.asm > /dev/null 2>&1",
                 project_root, current_project, project_root) != -1) {
        system(cmd); free(cmd);
    }
    // ... rest of code
}
```

### BUG #2: PAL Script `read_history` Path Resolution
**Severity:** HIGH  
**Location:** `projects/man-pal/scripts/move.asm` + `pieces/system/prisc/prisc+x.c`  
**Symptom:** Selector doesn't move despite keys being injected

**Root Cause:**
The PAL script uses `read_history r1` which relies on `PRISC_PROJECT_ROOT` environment 
variable. Looking at `prisc+x.c:604-620`:

```c
} else if (i.op == OP_READ_HISTORY) {
    char path[512];
    if (strlen(i.literal_arg) > 0) strcpy(path, i.literal_arg);
    else {
        char *proj_root = getenv("PRISC_PROJECT_ROOT");
        if (proj_root) snprintf(path, sizeof(path), "%s/pieces/apps/player_app/history.txt", proj_root);
        else strcpy(path, "pieces/apps/player_app/history.txt");
    }
    // ... reads and consumes key
}
```

The man-pal module DOES set `PRISC_PROJECT_ROOT` correctly, BUT the history file 
may already be consumed by the time Prisc reads it (race condition), OR the file 
path resolution fails silently.

**Evidence from history.txt:**
```
1003
```
The file contains key 1003 (DOWN), but the selector position remains at (5,5,0).

**Fix Option A (Recommended):** Add explicit path to PAL script:
```asm
start:
    # Explicitly specify the history buffer path
    read_history r1, "pieces/apps/player_app/history.txt"
```

**Fix Option B:** Ensure module passes environment correctly and add debug logging.

---

### BUG #3: Missing `hit_frame` in PAL Script Execution Path
**Severity:** MEDIUM  
**Location:** `projects/man-pal/scripts/move.asm`  
**Symptom:** Renderer may not update after movement

**Root Cause:**
The PAL script has `hit_frame` in each movement branch, but if `read_history` returns 0
(no key), it jumps to `end:` which only has `halt` without hitting the frame marker.

**Current Code:**
```asm
start:
    read_history r1
    beq r1, x0, end    # If no key, jump to end (NO hit_frame!)
    # ... movement logic
end:
    halt
```

**Fix:**
```asm
end:
    hit_frame    # Always hit frame marker before halt
    halt
```

**Key Insight:** The C module is a "Relay" that:
1. Polls history buffer
2. Triggers Prisc VM (Brain)
3. Syncs state for UI display
4. **MUST update last_key for UI debugging**

## 8. CONCLUSION

The man-pal pipeline has **three fixable bugs**:
1. Missing `last_key` string update (causes "KEY: None" display)
2. Missing `hit_frame` in no-key path (causes render sync issues)
3. Potential race condition in history consumption

All bugs are in the **Relay Layer** (C module), not the **Brain Layer** (PAL script).
The PAL architecture is sound; the implementation just needs these critical fixes.

**Quote from Monolith:** *"The Brain (PAL) is only as smart as the Relay (C) allows it to be."*
