## 1. Introduction to TPM for C (+x)

True Piece Method (TPM) is a software architecture where everything meaningful is a "piece" - a directory containing text files that define state, behavior, and relationships. In the C (+x) context, this means:

- **Pieces**: Directories with `piece.txt`, `ledger.txt`, `history.txt`
- **C Code**: System() calls to standalone executables
- **+x Pattern**: Executables with `.+x` extension in dedicated directories
- **File-Based Flow**: All communication through filesystem, no in-memory passing

## 2. Core Philosophical Principles

### 2.1 Everything Is a Piece
Every application component becomes a directory:
```
components/
├── player/
│   ├── piece.txt
│   ├── ledger.txt
│   └── history.txt
├── inventory/
│   ├── piece.txt
│   ├── ledger.txt
│   └── history.txt
└── game_master/
    ├── piece.txt
    ├── ledger.txt
    └── history.txt
```

### 2.2 Audit Obsession
Every state change, method call, and event is logged with timestamps:
```
[2026-02-10 18:58:25] MethodCall: feed on fuzzpet | Caller: user_input
[2026-02-10 18:58:26] StateChange: fuzzpet hunger 25 | Trigger: feed_method
```

### 2.3 No In-Memory Shortcuts
All data passes through files for complete traceability and reproducibility.

### 2.4 Data Sovereignty
Each piece owns its state files exclusively.

## 3. Directory-Based Architecture

### 3.1 Piece Directory Structure
Each piece follows this pattern:
```
piece_name/
├── piece.txt          # Core definition (state, methods, listeners, responses)
├── ledger.txt         # Piece audit trail
├── history.txt        # Input history for this piece
└── sub_pieces/        # Recursive pieces (optional)
```

### 3.2 Piece.txt Format
```
# Piece: piece_name
state:
  key1: value1
  key2: value2
methods: method1, method2, method3
event_listeners: event1, event2, event3
responses:
  event1: Response message for event 1
  event2: Response message for event 2
  default: Default response
```

## 4. The +x Pattern Explained

### 4.1 What is +x?
The `.+x` pattern is an executable naming convention meaning "executable" or "runnable".
- `action.+x` - Standalone executable performing a single function
- `manager.+x` - Module managing specific responsibilities
- All in a dedicated `+x/` directory

### 4.2 Why +x?
- Distinguishes executables from other files
- Maintains consistency in system() calls
- Clear separation of code from data
- Scalable execution model

### 4.3 +x Directory Structure
```
pieces/
├── plugins/           # Contains source and executables
│   ├── *.c            # Source files
│   └── +x/            # Compiled executables
│       ├── action.+x
│       ├── manager.+x
│       └── ...
```

## 5. File-Based Communication Architecture

### 5.1 The System() Call Pattern
Instead of function calls, use system() to execute standalone modules:
```c
system("./pieces/plugins/+x/action_module.+x arg1 arg2");
```

### 5.2 Inter-Module Communication
- Input: Command line arguments
- Output: File writes to piece files or stdout
- State changes: Modify piece.txt through manager
- Events: Separate modules handle event firing

### 5.3 The Communication Flow
1. **Input Received** → Write to command queue/historical log
2. **System() Calls** → Execute standalone modules  
3. **File Operations** → Read/modify piece state files
4. **Event Firing** → Trigger responses through separate modules
5. **Output** → Display to user or log to appropriate files

## 6. Three-Manager Pipeline Design

### 6.1 Piece Manager
Handles piece definitions and state:
```c
// Example system() call to piece manager
system("./pieces/plugins/+x/piece_manager.+x piece_id get-state key");
system("./pieces/plugins/+x/piece_manager.+x piece_id set-state key value");
```

### 6.2 Methods Manager  
Executes piece behaviors:
```c
// Execute action via system() call
system("./pieces/plugins/+x/feed.+x");
```

### 6.3 Event Manager
Triggers responses and reactions:
```c
// Fire events via system() call
system("./pieces/plugins/+x/event_fired.+x event_name");
```

### 6.4 Pipeline Flow
```
Input → Piece Manager → Methods Manager → Event Manager → Output
```

## 7. Master Ledger System

### 7.1 Central Audit Trail
Single file logging all system operations:
```
pieces/master_ledger/master_ledger.txt
```

### 7.2 Event Logging Module
Standalone executable for logging:
```c
system("./pieces/plugins/+x/log_event.+x EventType \"Description\" \"Source Info\"");
```

### 7.3 Complete Traceability
Every operation gets timestamped entry enabling:
- Time travel debugging
- Replay capability
- Security auditing
- Behavior verification

## 8. Enhanced Event Response Architecture

### 8.1 Decoupled Response Handling
Proper TPM implementation requires separating EventFire from ResponseRequest to allow asynchronous processing:
- **EventFire**: Occurs immediately after actions (feed, play, sleep, etc.)
- **ResponseRequest**: Separate entries in master ledger for triggering responses
- **ResponseHandler**: Dedicated module that monitors master ledger for response requests

### 8.2 Response Request Format
Standard format for response triggering in master ledger:
```
[2026-02-14 23:06:39] ResponseRequest: event_type on piece_name | Source: source_component
```

### 8.3 Event Parsing in Response Handlers
Important to correctly extract event data from log entries:
- Use proper string offsets when parsing: "ResponseRequest: " = 16 characters (15 + space)
- Skip leading whitespace after extraction
- Extract event_type before " on " and piece_name between " on " and " | "

### 8.4 Response Handler Thread
Response handling should be a separate thread in the orchestrator to process events asynchronously:
```c
// Response handler thread function
void* response_thread_func(void* arg) {
    system("./+x/response_handler.+x");
    pthread_exit(NULL);
}
```

```c
    char piece_file[500];
    snprintf(piece_file, sizeof(piece_file), 
             "pieces/%s/piece.txt", piece_id);
    
    // Parse piece file and perform requested action
    // Output result to stdout for calling module to use
}
```

### 10.2 Action Module Pattern
```c
// feed.c - Perform feeding action
int main() {
    // Get current state
    FILE *fp = popen("./pieces/plugins/+x/piece_manager.+x fuzzpet get-state hunger", "r");
    char hunger_str[10];
    fgets(hunger_str, sizeof(hunger_str), fp);
    pclose(fp);
    
    int hunger = atoi(hunger_str);
    int new_hunger = (hunger > 20) ? hunger - 20 : 0;
    
    // Update state
    char cmd[200];
    sprintf(cmd, "./pieces/plugins/+x/piece_manager.+x fuzzpet set-state hunger %d", new_hunger);
    system(cmd);
    
    return 0;
}
```

### 10.3 Event Handler Pattern
```c
// event_fired.c - Handle events and responses
int main(int argc, char *argv[]) {
    const char *event_name = argv[1];
    
    char response_cmd[500];
    char response[500];
    FILE *fp;
    
    sprintf(response_cmd, "./pieces/plugins/+x/piece_manager.+x fuzzpet get-response %s", event_name);
    fp = popen(response_cmd, "r");
    if (fp) {
        fgets(response, sizeof(response), fp);
        pclose(fp);
        printf("[Pet] %s\n", response);
    }
    
    return 0;
}
```

### 10.4 Orchestrator Pattern

## 11. Best Practices and Common Pitfalls

### 11.1 Best Practices
- **Always use file-based communication** between modules
- **Keep executables focused** on single responsibilities
- **Log everything** to the master ledger
- **Design pieces hierarchically** for complex systems
- **Use consistent naming** schemes (+x, piece.txt, etc.)
- **Handle errors gracefully** with fallback mechanisms
- **Implement proper parsing** for string manipulation operations
- **Test piece_manager independently** before integrating with other components

### 11.2 Common Pitfalls to Avoid
- **In-memory data passing** - Breaks audit trail
- **Direct file manipulation** without managers - Bypasses safeguards
- **Complex inter-module dependencies** - Breaks modularity  
- **Ignoring error cases** - Reduces reliability
- **Hardcoding paths** - Reduces portability
- **Incorrect string parsing** - Causes silent failures in event handling
- **Off-by-one errors in pointer arithmetic** - Critical for event parsing
- **Missing intermediate ResponseRequest step** - Bypasses proper response handling

### 11.3 Testing Strategy
- Test individual executables in isolation
- Verify file-based communication works correctly
- Validate audit trails contain expected entries
- Test system behavior with missing/wrong input
- Test response handlers with specific events separately

## 12. Advanced Implementation Patterns and Nuanced Tips

### 12.1 String Parsing Nuances
When parsing structured text from files, especially when using pointer arithmetic:
- Always count characters including spaces, colons, and punctuation
- For "ResponseRequest: event on piece | Source: component", the complete string is 16 characters to skip: "ResponseRequest" (13) + ":" (1) + " " (1) + " " (1) = 16 characters
- Use `strstr()` to find substrings, then advance pointer by correct offset
- Skip leading whitespace when extracting tokens

### 12.2 Debugging Strategies
- Create temporary debug versions of parsers that print entire lines and parsed elements
- Test piece_manager executables directly to ensure they return expected values
- Monitor master ledger files directly to verify proper event formatting
- Use print statements to show the exact strings being processed

### 12.3 Event Chain Design
Consider the complete chain of events in your system:
- **Trigger**: User input or system event occurs
- **Log**: EventFire or MethodCall written to master ledger
- **Process**: MethodsManager or other components respond
- **Update**: State changes logged to master ledger
- **Notify**: EventFire for state changes
- **Respond**: ResponseRequest logged to trigger feedback
- **Feedback**: ResponseHandler triggers response messages

### 12.4 Modular Thread Architecture
In orchestrator design, use separate threads for each responsibility:
- **Keyboard input thread**: Captures and logs user input
- **Game logic thread**: Processes events and updates state
- **Render thread**: Monitors state and updates display
- **Response handler thread**: Processes event responses and provides feedback

### 12.5 State Persistence and Synchronization
- Ensure in-memory state changes are saved to persistent files
- Synchronize piece state files with any visualization/representation state
- When multiple state representations exist (piece file and map file), keep them in sync
- Save updated state files frequently, especially after user actions
- Consider that the system will restart from what's in the files, not in-memory state
- Always test restart scenarios to verify persistence works correctly

### 12.6 Error Recovery and Fallbacks
- Always verify file operations succeed before proceeding
- Provide default responses when specific ones are not found
- Log error conditions to master ledger for audit trail
- Maintain system operation even when individual components fail

## Conclusion

The TPM +x architecture provides a robust, auditable, and scalable approach to building any application type. By following the principles of directory-based pieces, file-based communication, and standalone executables with the +x pattern, you can build complex systems that remain transparent, traceable, and maintainable.

Advanced implementations require careful attention to:
- Proper implementation of the full event-response lifecycle
- Accurate string parsing for event handling
- Decoupled components that can operate independently
- Complete audit trails in the master ledger
- Thread-safe file operations
- State persistence and synchronization across all representations

The key insight is that complexity emerges from simple interactions between simple components - pieces that own their state and communicate through standardized file-based protocols via system() calls to independent executables. This approach scales from simple pet simulators to complex production systems.

Start with the basic patterns shown here, then expand by adding more pieces and connecting them through events and responses. The architecture will naturally evolve to accommodate your application's needs while maintaining complete auditability and transparency.
