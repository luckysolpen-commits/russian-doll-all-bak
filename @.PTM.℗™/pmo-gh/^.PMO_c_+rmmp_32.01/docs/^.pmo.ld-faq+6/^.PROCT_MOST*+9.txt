1. THE GLOSSARY: DEFINING TPM AND PMO
=====================================

### TRUE PIECE METHOD (TPM)
TPM is a philosophy of software construction where every entity is a "Piece"—a self-contained directory on the filesystem that owns its own data (DNA), its own high-speed view (Mirror), its own behavior (Plugins), and its own history. 

### PIECE-MODULE-OS (PMO)
PMO is the hierarchical framework built upon TPM principles:
- **PIECE**: The atomic unit of state (The Soul).
- **MODULE**: The logic engine that animates pieces (The Brain).
- **OS**: The shell/theater that provides navigation and layout (The World).

2. CORE TPM PRINCIPLES
======================

PRINCIPLE 1: EVERYTHING IS A PIECE
- Every application component is a directory with a piece.txt or piece.pdl file.
- Examples: users, items, UI panels, clocks, ledgers, input handlers.

PRINCIPLE 2: DATA SOVEREIGNTY
- Pieces own their state - no shared global state. All state changes persist to disk immediately.

PRINCIPLE 3: AUDIT OBSESSION  
- Every operation logs to the master ledger. Complete traceability enables time-travel debugging.

PRINCIPLE 4: PIPELINE FLOW
- Sequential processing: Reader → Interpreter → Writer → Next Node. No direct calls.

PRINCIPLE 5: NO IN-MEMORY SHORTCUTS
- All data passes through files for auditability. behavior must be reproducible from the starting file state.

8. INPUT/OUTPUT SYSTEMS & THE 10-STEP PIPELINE
==============================================

This is the definitive flow of data in TPM v3.0:

1.  **INPUT**: User types 'w' or moves a joystick.
2.  **CAPTURE**: `keyboard_input` writes code 119 to `keyboard/history.txt`.
3.  **TRANSLATION**: `chtpm_parser` reads 119, sees it's 'w', and detects `INTERACT` mode.
4.  **RELAY**: Parser calls `inject_raw_key(119)` to append to the Module's `history.txt`.
5.  **TICK**: `fuzzpet_manager` reads 119 from its local history.
6.  **LOGIC**: Manager calls `move_entity.+x pet_01 w`.
7.  **STATE**: `move_entity` updates `pos_y` in `pet_01.pdl`.
8.  **MIRROR**: `piece_manager` immediately flushes `pos_y` to `pet_01/state.txt` (Mirror Sync).
9.  **VIEW**: `map_render` reads `state.txt` mirrors directly and writes to `view.txt`.
10. **RENDER**: Parser substitutes `view.txt` into the layout and writes the final frame to `current_frame.txt`.

11. FAQ & THE 7 DEADLY PITFALLS
===============================

#### PITFALL 1: THE TRIM TRAP
Filesystem mirrors often have trailing spaces or newlines (e.g., `on_map=1 `).
```c
// THE FIX:
char* trim(char *str) {
    while(isspace(*str)) str++;
    char *end = str + strlen(str) - 1;
    while(end > str && isspace(*end)) end--;
    *(end + 1) = '\0';
    return str;
}
```

#### PITFALL 2: popen OVERHEAD
Calling `popen` 100 times per frame will drop FPS to 5.
**FIX**: Bypass the manager for READS; use Direct Mirror Access (parsing `state.txt` in C).

#### PITFALL 3: mtime RESOLUTION
mtime has 1-second resolution. Updates within the same second are missed.
**FIX**: Watch the **SIZE** of `frame_changed.txt`.

#### PITFALL 4: BUFFER TRUNCATION
Expanding large maps into tag attributes causes corruption.
**FIX**: Hybrid Expansion (Expand large variables during final render).

#### PITFALL 5: RELATIVE PATH CONFUSION
**WRONG**: `../pieces/history.txt` (creates nested folders).
**RIGHT**: Calculate paths from project root via `location_kvp`.

#### PITFALL 6: OVERWRITING STRUCTURE
**WRONG**: `write_piece_state()` overwrites responses/methods.
**RIGHT**: Preserve structure using a section-aware parser.

#### PITFALL 7: REDUNDANT RESPONSES
**ISSUE**: Background processes printing to stdout are lost.
**FIX**: Log responses to `last_response.txt` and bake them into the frame.

#### PITFALL 8: THE MEMORY CORRUPTION PULSE (v3.1)
**ISSUE**: `fopen()` using a pointer that has already been `free()`'d.
**SYMPTOM**: Files with corrupted names (e.g., `D-]`) appearing in the root directory.
**FIX**: Ensure string lifetimes exceed their usage in I/O operations.

#### PITFALL 9: THE STALE ROOT DRIFTER (v3.1)
**ISSUE**: `location_kvp` hardcoding a `project_root` from a previous version/folder.
**SYMPTOM**: System writes data to old versions or the current root instead of the active `pieces/` directory.
**FIX**: Always resolve `project_root` dynamically using `SCRIPT_DIR` in `run_chtpm.sh`.
