## 1. PROBLEM STATEMENT

**Observed Symptom:** RMMP-Editor (`editor_module.+x`) randomly crashes the CPU or causes system instability.

**User Hypothesis:** "Is it not quitting all its processes on Ctrl+C kill maybe?"

---

## 2. ROOT CAUSE ANALYSIS

### 2.1 The Smoking Gun

After analyzing `editor_module.c`, the root cause has been identified:

**CRITICAL DEFECT:** `editor_module.c` has **NO SIGNAL HANDLER** for `SIGINT` (Ctrl+C).

When the user presses Ctrl+C:
1. The default signal handler terminates the main `editor_module.+x` process.
2. Any child processes spawned via `system()` calls **become orphaned**.
3. These orphaned processes continue running with no parent to reap them.
4. Over time, this creates a **zombie process accumulation** or **runaway process storm**.
5. CPU usage spikes, leading to crashes or throttling.

### 2.2 Evidence from Code

```c
// editor_module.c - NO signal handler defined anywhere
int main() {
    resolve_paths(); scan_projects(); scan_maps();
    // ... initialization ...
    while (1) {
        // Main loop with NO cleanup on interrupt
        if (!is_active_layout()) { usleep(100000); continue; }
        // ... polling logic ...
    }
    // Never reaches here on Ctrl+C
    free(hist_path); return 0;
}
```

**Comparison with Prisc VM (which HAS signal handling):**
```c
// prisc+x.c - Line 14-16
void handle_sigint(int sig) {
    _exit(130);  /* Immediate exit, no cleanup */
}

int main(int argc, char **argv) {
    signal(SIGINT, handle_sigint);  // Registered in main
    // ... rest of code ...
}
```

### 2.3 Why Legacy Fuzzpet Doesn't Crash

`fuzz_legacy_manager.c` doesn't have an explicit signal handler either, BUT:
1. It uses `fork()/exec()/waitpid()` pattern exclusively.
2. Child processes are waited on **synchronously** (blocking).
3. When parent dies, children are already complete (not orphaned).
4. No `system()` calls that could spawn detached processes.

### 2.4 Why Editor Module IS Vulnerable

`editor_module.c` uses `system()` extensively:
```c
// Line 152: trigger_render()
system(cmd);  // Spawns shell + command

// Line 340: Delete piece
system(cmd);  // Another detached process

// Line 378: Ctrl+Z undo
system(cmd);  // More detached processes

// Line 390: Move entity
system(cmd);  // And more...
```

**The Problem with `system()`:**
- Spawns a **subshell** (`/bin/sh -c <command>`)
- If parent is killed mid-execution, subshell may continue
- Grandchild processes (like `render_map.+x`) become orphaned
- These orphans inherit the terminal and can't be cleanly killed

## 3. THE FIX

### 3.1 Immediate Fix: Add Signal Handler

Add this to `editor_module.c`:

```c
#include <signal.h>

/* Global flag for graceful shutdown */
volatile sig_atomic_t shutdown_requested = 0;

void handle_sigint(int sig) {
    shutdown_requested = 1;
}

// In main():
signal(SIGINT, handle_sigint);

// In main loop:
while (1) {
    if (shutdown_requested) {
        // Cleanup code here
        free(hist_path);
        break;
    }
    // ... rest of loop ...
}
```

### 3.2 Better Fix: Process Group Management

For complete cleanup, set up a process group:

```c
#include <signal.h>

void handle_sigint(int sig) {
    // Kill entire process group
    kill(-getpgrp(), SIGTERM);
    _exit(130);
}

int main() {
    // Make this process a group leader
    setpgid(0, 0);
    
    signal(SIGINT, handle_sigint);
    signal(SIGTERM, handle_sigint);
    
    // ... rest of initialization ...
}
```

### 3.3 Best Fix: Mirror Legacy Fuzzpet Pattern

Replace `system()` calls with `fork()/exec()/waitpid()`:

```c
// Instead of:
system(cmd);

// Use:
pid_t pid = fork();
if (pid == 0) {
    // Child process
    freopen("/dev/null", "w", stdout);
    freopen("/dev/null", "w", stderr);
    execl(cmd_path, cmd_path, arg1, arg2, NULL);
    exit(1);
} else if (pid > 0) {
    // Parent waits for child
    waitpid(pid, NULL, 0);
}
```

This is exactly what `fuzz_legacy_manager.c` does (see `route_input()` function).

## 5. PREVENTION STRATEGY

### 5.1 Mandatory Signal Handler Template

All future PMO modules MUST include this boilerplate:

```c
#include <signal.h>

static volatile sig_atomic_t g_shutdown = 0;

static void sig_handler(int sig) {
    (void)sig;
    g_shutdown = 1;
}

int main() {
    signal(SIGINT, sig_handler);
    signal(SIGTERM, sig_handler);
    
    while (!g_shutdown) {
        // Main loop
    }
    
    // Cleanup
    return 0;
}
```

### 5.2 Process Group Cleanup

For modules that spawn many children:

```c
// At start of main:
setpgid(0, 0);

// In signal handler:
kill(-getpgrp(), SIGTERM);
```

### 5.3 Prefer Fork/Exec/Waitpid Over System()

| Pattern | CPU Safety | Process Control | Recommendation |
|---------|-----------|-----------------|----------------|
| `system()` | ❌ Poor | ❌ None | Avoid for critical paths |
| `fork()/exec()/waitpid()` | ✅ Excellent | ✅ Full | Use everywhere |
| `popen()/pclose()` | ⚠️ Moderate | ⚠️ Partial | OK for reading output |

## 6. VERIFICATION CHECKLIST

Before considering a module "CPU Safe":

- [ ] Signal handler for `SIGINT` implemented
- [ ] Signal handler for `SIGTERM` implemented
- [ ] Process group set (`setpgid(0, 0)`)
- [ ] All child processes are waited on or tracked
- [ ] No unbounded `system()` calls in tight loops
- [ ] Focus throttling implemented (`is_active_layout()`)
- [ ] History polling uses `stat()` before `fopen()`
- [ ] Main loop has reasonable sleep interval (≥16ms)
