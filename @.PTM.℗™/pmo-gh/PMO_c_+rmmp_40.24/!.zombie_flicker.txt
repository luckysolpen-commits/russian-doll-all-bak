# Zombie & Flicker Analysis - FuzzPet App
Date: 2026-03-17

## 1. Zombie AI Logic Failure
The zombie in `fuzzpet_app` stopped following the player due to hardcoded project paths in `zombie_ai.c`. 

**Technical Root Cause:**
The `zombie_ai.c` source code prioritized paths in `projects/fuzz-op/` for entity state detection. When running in `fuzzpet_app` mode, the zombie was attempting to chase the position of a "ghost" player from a previous `fuzz-op` session rather than the active player in `pieces/world/map_01/`.

**Collision Bug:**
The collision detection logic was also failing because it used inconsistent path resolution, often leading it to believe tiles were occupied by non-existent entities or entities at (0,0) due to failed state reads.

## 2. Screen Flickering (Double Writing)
The flickering observed during movement and actions is caused by "Multi-Pulse Contention".

**The Render Pipeline Leak:**
1. Manager receives input.
2. Manager calls `move_entity.+x`.
3. `move_entity.+x` updates state and **immediately hits `frame_changed.txt`**.
4. The Renderer sees this signal and prints the **OLD frame** (because the Parser hasn't updated it yet).
5. Manager then calls `trigger_render()` which calls `map_render.+x`.
6. `map_render.+x` updates `view.txt` and hits `view_changed.txt`.
7. The Parser sees `view_changed.txt`, composes the **NEW frame**, and hits `frame_changed.txt`.
8. The Renderer sees the second signal and prints the **NEW frame**.

This rapid sequence of OLD -> NEW frame rendering within milliseconds causes the visible flicker.

## 3. Initial Map "Ghost" Frame
On startup, `fuzzpet_app` shows one frame of the previously loaded project's map.

**Cause:**
The `view.txt` file is shared. The `chtpm_parser` reads the existing `view.txt` content immediately upon the first render pulse. If `fuzzpet_manager` triggers a render before it has properly initialized the project context or cleared the old view, the stale map is baked into the first frame.

## 4. Project Switching Failures
Switching projects (e.g., `fuzz-op` -> `fuzzpet_app`) breaks input routing and movement.

**Cause:**
`fuzzpet_manager` was not project-aware. It didn't refresh its path resolution or project ID from the global player state (`pieces/apps/player_app/manager/state.txt`). This caused it to continue looking at stale history buffers or attempting to update pieces in the wrong project directory.
