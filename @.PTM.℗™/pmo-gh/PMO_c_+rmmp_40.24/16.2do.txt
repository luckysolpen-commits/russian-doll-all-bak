# 16.2do.txt - Fuzz-Op Parity Plan & Understanding (Revised)

I have looked deeper into `fuzzpet_app` and `chtpm_parser.c`. Here is my refined understanding:

## Understanding of the Project Goal
We are implementing the **Index-Based Method Response** system. 
- **Pre-Interact Mode (Nav Mode)**: The user can navigate the menu indices (1. Control Map, 2. scan, 3. collect, etc.). Selecting an index and pressing ENTER should trigger the corresponding logic.
- **Interact Mode (Active Mode)**: Triggered by index 1 (Control Map). Nav is disabled, WASD moves the entity, but number keys (2-8) still act as hotkeys for the same methods.
- **Dynamic Population**: The Nav menu must update its choices based on the `active_target_id` (e.g., selector's methods vs. pet's methods).

## Identified Gaps in fuzz-op
1. **Naming Trap**: `pdl_reader` and `piece_manager` are looking for `piece.pdl`, but `selector` only has `selector.pdl`. This breaks the method lookup for the selector.
2. **Logic Block**: `fuzz-op_manager.c` explicitly blocks method hotkeys for the "selector" target, which prevents scan/collect from working.
3. **Response Path**: `chtpm_parser.c` is looking for responses in `pieces/apps/fuzzpet_app/...` but `fuzz-op` is writing them to its project folder. This makes the UI feel "dead" as responses don't show up.
4. **Layout/State Desync**: The layout uses `${pet_name}` but the manager might be writing to a different variable set or prefix.

## Implementation Plan
1. **Fix PDL Consistency**: Standardize on `piece.pdl` within piece directories so all tools can read them consistently.
2. **Enable Selector Methods**: Modify `fuzz-op_manager.c` to allow indices 2-8 to work regardless of whether the target is "selector" or an entity.
3. **Index-to-Method Mapping**: Ensure the manager correctly maps index `n` to the `n-th` method in the PDL (skipping built-ins), matching the Parser's indexing.
4. **Dynamic Response Integration**: Update `chtpm_parser.c` to load `${last_response}` from the project-specific piece path so the "Found: ..." and pet responses are visible.
5. **Mirror Sync Alignment**: Ensure `perform_mirror_sync` writes to the variables the layout actually expects (e.g., `pet_name` vs `name`).
