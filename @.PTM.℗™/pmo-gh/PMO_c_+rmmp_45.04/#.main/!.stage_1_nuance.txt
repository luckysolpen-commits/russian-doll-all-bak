# Stage 1 Nuance - Recursive Editor/Player Logic

## 1. The "Active Layout" Trap
Current Issue: `op-ed_module.c` uses `is_active_layout()` to gate key processing. 
Nuance: When switching to `file_menu.chtpm` or `load_menu.chtpm`, the module currently thinks it is inactive because `my_layout_path` only points to the main editor layout.
Fix needed: `is_active_layout` must modularly detect if the current layout belongs to the `op-ed` app suite (e.g., by checking the directory or reading `op-ed.pdl`).

## 2. Play Mode Transition
Philosophy: "Play" must be an execution of the current piece state using the engine (`playrm_module.c`), not just a preview.
Nuance: `playrm_module.c` reads `pieces/apps/player_app/manager/state.txt` to find the `project_id`. 
Workflow:
1. User in `op-ed` hits "PLAY".
2. `op-ed` writes `project_id` (current or cloned) to the manager state.
3. `op-ed` triggers a layout change to `pieces/apps/playrm/layouts/game.chtpm`.
4. `playrm_module.c` wakes up, sees it is the active layout, and begins processing inputs against the pieces in `projects/<id>/pieces/`.

## 3. Scroller Responsiveness
Issue: The load menu scroller appears but doesn't move.
Cause: Confirmed to be the "Active Layout" trap. The module stops reading `history.txt` as soon as the layout file name changes to `load_menu.chtpm`.

## 4. Local Piece Resolution
Nuance: Shared Ops (`move_entity`, `render_map`) must continue to prioritize `projects/<id>/pieces/world_xx/map_xx/` to maintain the world-architecture mandate.
Status: Op-ed saving is now world-aware, writing to `project.pdl` metadata to preserve this context.
