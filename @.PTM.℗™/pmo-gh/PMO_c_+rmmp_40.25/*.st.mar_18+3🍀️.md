## 1. Executive Summary
The codebase remains aligned with PMO/TPM principles (Piece -> Module -> OS), but has shifted from older flat piece assumptions toward explicit world/map containment. Legacy docs accurately describe the philosophy and pipeline, but were missing current operational changes: project-specific method dispatch fixes, root/path hardening, selector state integrity fixes, world-map migration staging, op-ed world-targeted creation, and loader path corrections for op-ed launch.

The key architectural direction is now:
- Canonical containment: `projects/<project>/pieces/world_<id>/map_<id>/<piece_id>/...`
- Transitional compatibility: root fallback remains temporarily for legacy modules
- Container model generalized: maps can contain pieces; pieces can contain pieces (inventory/interior)
- Delivery order updated: local authoring/play loop -> AI ops intelligence layer -> pet expansion + voice accessibility -> multiplayer civilization/market foundation game -> network layer -> gl-os/render platform -> PMOS install/versioning/remote-use platform -> long-horizon RISC-V/x86 kernel path

## 3. PMO/TPM Core Model (Consolidated)
### 3.1 Hierarchy
1. **PIECE**: sovereign data unit (directory + state + optional PDL + plugins)
2. **MODULE**: orchestration/behavior layer (input routing, invoking ops, sync)
3. **OS/CHTPM**: view/theater layer (layout, nav/interact routing, substitution)

### 3.2 Operational Principles
- File-based IPC, not in-memory coupling
- Auditability via ledgers/history/state files
- Delegation to ops (`+x` binaries) for reusable logic
- Parser composes views; modules decide behavior

### 3.3 Historical continuity
Legacy docs around KISS/TPM/PMO remain valid at principle-level. Main drift happened in pathing, project-specific behavior, and migration staging details.

## 4. Runtime Pipeline (Input to Render)
Current practical flow:
1. Input device writes key events
2. CHTPM parser resolves active context and injects into interact target
3. Active module reads app history
4. Module calls ops (`move_entity`, `render_map`, `create_piece`, etc.)
5. Ops update piece state / map files
6. Render op writes stage view
7. Parser composes final frame with layout variables
8. Display renderer prints frame

Important nuance:
- Loader/project selection is dynamic via loader module and generated loader methods
- For game/editor layouts, `pieces/apps/player_app/history.txt` remains central input relay

## 5. State Model: PDL vs state.txt (Current Reality)
### 5.1 Practical truth in codebase
- `state.txt` is the runtime mirror for fast reads/writes
- PDL is partially used for method capabilities and optional dispatch
- Some modules/ops still rely on direct state/method assumptions

### 5.2 Current best interpretation
- Use PDL for method capability declarations and handler mapping where available
- Use `state.txt` for mutable runtime values
- Keep both synchronized where behavior depends on both

### 5.3 Mar 18 correction
A major source of bugs was inconsistent PDL filename/path conventions (`piece.pdl` vs `<piece_id>.pdl`) across parser vs reader. This was normalized with fallback resolution logic.

## 8. World Architecture Clarification (Current Direction)
### 8.1 Canonical structure
Preferred now:
- `projects/<project>/pieces/world_<id>/map_<id>/<piece_id>/`

No mandatory extra `/pieces/` layer is required under map containers.

### 8.2 Container-generalized model
Containment is recursive:
- map contains piece
- piece can contain pieces (inventory/interior)
Examples:
- `house/dog/`
- `player_1/bag/crayon_box/crayon1/`

### 8.3 Transitional compatibility
Until migration completes, renderer/module fallback may still read root-level piece dirs.

## 10. Known Gaps and Migration Risks
1. Not all modules are world/map-native yet; some still assume root piece paths
2. Fallback paths intentionally remain in several places to keep system usable during migration
3. Risk of duplicate semantics during migration if same logical piece appears in multiple locations without dedup strategy
4. World/map container creation and migration tooling still needs full UX and operational integration
5. AI runtime standardization is not yet formalized (FSM/RL/LLM piece APIs, model profile contracts, distillation workflow)
6. Network design must account for AI-generated state/events to avoid desync and incompatible payload schemas
7. Exchange/trading mechanics can unintentionally create financial-regulatory exposure if positioned as investment products
8. Accessibility parity risk: voice control/readback must remain behaviorally identical to joystick/keyboard action paths
9. Install/versioning complexity risk: remote app use and trunk-save semantics can fragment if package/version contracts are not standardized
10. Portability risk: kernel migration to RISC-V/x86 can force costly rewrites if piece/module/app contracts are not isolated early

## 12. MVP Legal and Early-Adopter Go-To-Market Guardrails (U.S.)
This section is operational guidance, not legal advice. Before public launch, retain U.S. counsel for securities, payments/MTL, sanctions/AML, privacy, tax, and consumer-protection review.

### 12.1 Product-positioning guardrails for MVP
1. Market as a game/simulation platform, not an investment vehicle.
2. Do not promise, imply, or benchmark toward guaranteed returns, passive income, or appreciation from team/host efforts.
3. Treat "Play Exchange" as in-game utility/accounting first; if external value transfer is added, run legal review before launch.
4. Keep early-adopter messaging factual and test-oriented (alpha/beta), with clear risk disclosures and no earnings claims.

### 12.2 Regulatory workstreams to open before external launch
1. **Securities review (SEC/Howey):** evaluate token/asset design, distribution, and marketing language for investment-contract risk.
2. **Money transmission/AML (FinCEN + states):** determine whether any flow is money transmission and whether MSB registration/state licensing obligations are triggered.
3. **Sanctions screening (OFAC):** build sanctions controls before handling value transfer or hosted exchange flows.
4. **Consumer protection/marketing (FTC):** ensure ad, influencer, and testimonial disclosures are clear and not deceptive.
5. **Children/privacy (COPPA + general privacy):** if under-13 use is possible, implement COPPA-compliant flows before collection.
6. **Tax operations (IRS):** implement digital-asset transaction logging and user tax reporting support from day one.

### 12.3 Minimum compliance controls for a 2-user MVP
1. KYC-lite identity and geoblocking policy consistent with counsel guidance.
2. Terms of service + risk disclosures + clear acceptable-use policy.
3. Basic AML/sanctions screening and incident response path.
4. Immutable event/audit logs for gameplay trades and exchange operations.
5. Kill-switches for trading/exchange features and suspicious account activity.
6. Marketing approval checklist so no public copy implies investment returns.

### 12.4 Recommended rollout sequencing
1. Private pre-alpha (closed users, no real-value transfer).
2. Limited alpha with Play Exchange in simulation-only mode.
3. Compliance gate review with counsel before enabling any cash-out/real-value bridge.
4. Public beta only after securities/payments/sanctions/privacy sign-off.

## 13. Glossary
- **CHTPM**: Layout/parser/render layer (theater)
- **DNA**: Structured canonical piece definition (PDL concept)
- **Mirror**: Fast mutable state file (`state.txt`)
- **Op / +x**: Reusable compiled executable used by modules/scripts
- **PMO**: Piece-Module-OS hierarchy
- **TPM**: True Piece Method philosophy
- **World Container**: `world_<id>` directory representing a world scope
- **Map Container**: `map_<id>` directory under a world, containing map-local pieces
- **Container Piece**: Any piece directory that contains child piece dirs (inventory/interior)
- **Fallback Path**: Legacy-compatible path resolution used while migration is incomplete
- **Dedup Precedence**: Rule to prefer canonical nested piece instances over fallback/root duplicates
