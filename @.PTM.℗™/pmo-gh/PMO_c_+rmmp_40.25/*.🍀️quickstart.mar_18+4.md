# PMO/TPM Quickstart Roadmap + Integration Notes (March 18, 2026)

## Purpose
This is the technical quickstart layer for execution sequencing, interface contracts, and synergy opportunities across:
- local no-code creation
- AI ops
- pet AI expansion (chat/management/self-training)
- voice accessibility (voice control + voice readback)
- foundation simulation/economy game
- networking
- gl-os and multi-view rendering
- PMOS install/versioning/remote-use platform features
- long-horizon kernel portability (RISC-V/x86)

## Priority Stack (Execution Order)
1. **Task 1:** `op-ed` save-to-file, save/load, local play + `pal`/op-event no-code piece workflows.
2. **Task 2:** AI ops intelligence layer (FSM, RL, small LLM via ollama + CLI-coder workflows, scalable model profiles).
3. **Task 3 (Pet Expansion Before Full Sim):** pet chat, pet self-management, self-supervised research/training, multi-pet AI management menus, and a pet generator.
4. **Task 4 (Accessibility Track, starts now):** voice control and voice readback for navigation menus, pet chat, and core interactions; start in pet layer and carry forward into all phases.
5. **Task 5:** Turn-based foundation game (evolution/civilization/market/war/space exploration/chem-physics R&D) in text, 2D, and 3D.
6. **Task 6:** Network layer (PvP, P2P trade, chat, profiles) with Play Exchange on hybrid hosted/P2P blockchain architecture.
7. **Task 7:** `gl-os` desktop process manager + unified terminal/2D/3D/3D-ascii surfaces.
8. **Task 8 (May start before kernel):** PMOS install/distribution system with remote app use and trunk-save versioning.
9. **Task 9 (Long-horizon):** RISC-V/x86 kernel path using the same piece architecture, desktop OS model, and app layer contracts.

## Core Integration Contracts

### 1) Piece Runtime Contract
1. Every piece must expose canonical identity and location:
- `piece_id`
- `world_id`
- `map_id`
- `owner_id` (optional in local mode)
- `container_id` (optional inventory/interior nesting)
2. Runtime state remains in `state.txt`; behavior capability remains in PDL/method mapping.
3. Event handlers must be deterministic per turn tick when running in turn-based simulation.

### 2) Event Contract (Shared Across All Phases)
1. Standard envelope:
- `event_id`
- `turn_id` and `tick_seq`
- `source_piece`
- `target_piece` or `target_scope`
- `event_type`
- `payload`
- `signature_hash` (for audit/replay)
2. All modules (local, AI, pet manager, accessibility layer, simulation, network) consume and emit the same envelope.
3. Foundation-game economy and later network reconciliation both depend on this single event shape.

### 3) Intelligence Contract (AI Ops)
1. AI op adapters must present the same call shape regardless of backend (FSM/RL/LLM).
2. Required interfaces:
- `decide(context) -> action`
- `learn(feedback) -> model_delta`
- `explain(action) -> rationale` (for debugging/mod tools)
3. Profile-aware model routing:
- `profile_embedded`
- `profile_desktop`
- `profile_gpu`
4. Distillation pipeline: larger-model offline guidance -> compact model artifact for runtime pieces.

### 4) Accessibility Contract (Voice)
1. Voice is an input/output adapter over the same menu/event system, not a separate logic path.
2. Required interfaces:
- `voice_command -> canonical_action`
- `ui_state -> voice_readback`
- `chat_text <-> speech_stream`
3. Command grammar should mirror joystick/menu semantics for consistency.
4. Accessibility modes:
- `voice_only`
- `voice_plus_keyboard`
- `readback_only`

### 5) Economy/Exchange Contract
1. Separate simulation value from external-value bridge.
2. Play Exchange baseline entities:
- `asset`
- `order`
- `fill`
- `inventory_position`
- `exchange_rate_snapshot`
3. Turn-based clearing engine produces deterministic fills per turn for replay and anti-desync.
4. Foundation game becomes benchmark source for multiverse exchange references.

## Technical Details by Phase

### Task 1: Local No-Code Authoring + Play Loop
1. Persist op-ed project artifacts in world/map-native structure.
2. Local play bootstrap reads project manifest and loads pieces without manual edits.
3. `pal` + op-event editor must support create/bind/test for non-coders.
4. Save/load round trip must preserve:
- event bindings
- piece state
- container nesting

### Task 2: AI Ops Layer
1. Implement pluggable adapters:
- `ai_fsm.+x`
- `ai_rl.+x`
- `ai_llm.+x`
2. Maintain shared feature schema for state featurization so models can be swapped.
3. Add model registry metadata per piece/tool:
- `model_type`
- `model_version`
- `profile`
- `resource_budget`
4. Add guardrails for bounded inference time per turn.

### Task 3: Pet Expansion Layer (Before Full Simulation)
1. Add pet chat with conversation memory controls and configurable personality/disposition profiles.
2. Add pet self-management loops (needs, routines, goals, autonomous tasks).
3. Add self-supervised research/training pipeline for pet-owned models with budget controls.
4. Build multi-pet management menus for assigning tasks, priorities, and AI dispositions.
5. Build pet generator templates (species/archetype/traits) that emit world/map-native piece bundles.

### Task 4: Voice Accessibility Layer
1. Add voice navigation for primary menus and submenu traversal.
2. Add voice readback for active menu item, context hints, errors, and confirmations.
3. Add voice input/output path for pet chat.
4. Add per-user voice settings:
- readback speed/verbosity
- confirm-before-action policy
- wake/action phrase mappings
5. Add accessibility regression tests for menu parity (keyboard/joystick/voice should trigger same canonical actions).

### Task 5: Foundation Simulation Game (Before Networking)
1. Build as canonical simulation substrate, not a side project.
2. Systems included in turn loop:
- civ/evolution progression
- stock/commodity exchange mechanics
- war/conflict resolution
- space exploration progression
- chemistry/physics R&D chains
3. Multi-view output must be renderer-only transformation:
- text view
- 2D view
- 3D view
4. Same source-of-truth events/states drive every view.

### Task 6: Network + Play Exchange
1. Network protocol transmits signed event envelopes and state deltas.
2. Reconciliation strategy:
- deterministic turn authority
- conflict resolution policy
- replay from event log
3. Hybrid hosted/P2P design:
- hosted relay for reliability/discovery
- P2P channels for direct session exchange where possible
- blockchain anchor for critical exchange/audit checkpoints
4. Social systems:
- profile identity graph
- chat events tied to moderation pipeline
- PvP/trade actions bound to same turn/event ledger
5. Voice extension over network:
- optional voice-chat transport separate from command/readback events
- command events remain text/canonical for deterministic replay

### Task 7: gl-os + Unified Surfaces
1. Process manager treats terminal apps and graphical apps as first-class processes.
2. Desktop context menu launches scoped terminals and app surfaces.
3. Multiple terminals per virtual desktop; process introspection from gl-os panel.
4. “3D ASCII in terminal” is a renderer mode, not a separate simulation engine.
5. System-wide accessibility panel controls voice mode and readback behavior.

### Task 8: PMOS Install + Versioning + Remote App Use
1. Add install flows for PMOS apps/components with dependency-aware manifests.
2. Add remote app use support (remote execution/session linkage) with consistent piece/app contracts.
3. Add trunk-save versioning model:
- branch/trunk snapshots for apps/projects
- rollback/replay checkpoints
- compatibility metadata for shared app artifacts
4. Keep install/versioning APIs aligned with local-first event/state architecture.
5. If needed for delivery velocity, this task can start in parallel with late Task 6 or early Task 7 work.

### Task 9: Kernel Portability (RISC-V/x86)
1. Define kernel-facing contracts that preserve piece/module/app semantics.
2. Build portability layer so existing app/runtime contracts survive architecture shift.
3. Stage hardware targets:
- emulated bring-up
- RISC-V target
- x86 target
4. Keep this as long-horizon after platform/runtime distribution foundations are stable.

## Synergistic Opportunities (Nuanced Integrations)
1. **AI + Pet Layer:** pet chat/self-management is a low-risk proving ground for disposition control and autonomous task planning.
2. **Pet Layer + Voice:** menus and chat become accessible early, and this validates voice grammar before sim/network complexity.
3. **Voice + Joystick/Menu Parity:** one canonical action bus means voice control can reuse existing navigation logic.
4. **Pet Layer + No-Code Authoring:** users can tune pet behavior through menus/events before touching larger simulation systems.
5. **Pet Layer + Foundation Game:** multi-pet orchestration patterns map directly to factions/agents in civ and war systems.
6. **AI + Foundation Economy:** RL policies tune market/civ balancing; FSM keeps safety rails for deterministic fallback.
7. **Foundation Game + Network:** deterministic turn simulation drastically reduces sync complexity and cheat surface.
8. **Exchange + Multiverse:** single benchmark exchange-rate model allows user games to interoperate economically with shared references.
9. **gl-os + Development Velocity:** integrated process visibility shortens debug loops across parser/module/op layers.
10. **Text/2D/3D parity:** one simulation core with multiple renderers prevents gameplay drift across modalities.
11. **Model scaling + deployment:** same piece API supports tiny embedded models and larger desktop/GPU models without game-logic rewrites.
12. **Install/Versioning + Ecosystem Scale:** trunk-save versioning and remote app use accelerate user-created app reuse across the multiverse.
13. **Kernel Portability + App Continuity:** stable piece contracts reduce rewrite cost when moving toward RISC-V/x86-native PMOS.

## MVP-to-Scale Guardrails
1. Keep world/map hierarchy canonical; avoid new hardcoded root-path assumptions.
2. Do not fork event schemas by phase; extend one envelope.
3. Enforce deterministic turn boundaries before enabling full distributed play.
4. Gate external-value exchange features behind compliance review and staged rollout.
5. Keep AI modules swappable and budget-capped to prevent runtime instability.
6. Keep voice commands mapped to canonical actions only; no hidden side-path behavior.
7. Treat install/versioning metadata as first-class state with migration/version compatibility checks.
