CHTPM Module Hardcode Audit (M18)
Date: 2026-03-18
Scope: pieces/chtpm support modules, excluding chtpm_parser.c
Status: No code changes made. Deferred for later implementation.

Summary
-------
Audit focused on hardcoded assumptions in CHTPM support modules (`chtpm_player`, `orchestrator`, render helpers).
No obvious project-specific gameplay hardcoding was found in these non-parser modules (no direct fuzz-op/op-ed gameplay coupling).
Main findings are infrastructure defaults that should eventually be configurable.

Findings
--------
1) chtpm_player default layout fallback
- File: pieces/chtpm/plugins/chtpm_player.c:48
- Current: `current_layout = "pieces/chtpm/layouts/player_test.chtpm"`
- Risk: Medium
- Why: test-biased startup fallback can mislead future agents and create brittle behavior assumptions.

2) orchestrator parser startup layout hardcoded
- File: pieces/chtpm/plugins/orchestrator.c:202
- Current: launches parser with `pieces/chtpm/layouts/os.chtpm`
- Risk: Medium
- Why: startup layout should be configurable, not embedded in orchestration code.

3) orchestrator core service paths are static
- File: pieces/chtpm/plugins/orchestrator.c:198-201
- Current: hardcoded response/render/gl/clock service paths.
- Risk: Low-Medium
- Why: infra wiring is acceptable short-term, but config-driven paths would improve modularity and portability.

4) shell command hardcoding in orchestrator
- File: pieces/chtpm/plugins/orchestrator.c:47, 50, 109, 216
- Current: `pkill`, `kill_all.sh`, `mkdir -p` via `system()`.
- Risk: Low
- Why: operationally brittle and less controlled than direct process/file APIs.

Current Decision
----------------
No modifications performed now (accepted as stable enough for current priorities).

Deferred Modularization Plan (Later)
------------------------------------
1) Move startup layout defaults to config/state (with safe fallback chain).
2) Move core service executable paths to centralized config.
3) Keep CHTPM support modules infrastructure-generic and avoid app-specific hardcoding.
4) Replace brittle shell control paths where practical.

Guardrail Reminder
------------------
In `pieces/chtpm` support modules (excluding parser), keep logic as generic plumbing.
Do not add project-specific gameplay assumptions (piece IDs, app rules, world/map semantics).
