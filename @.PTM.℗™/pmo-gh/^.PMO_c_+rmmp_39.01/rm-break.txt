# RM-BREAK: op-ed Display Issue Analysis
**Date:** 2026-03-16  
**Status:** DIAGNOSED  

---

## THE SYMPTOM

After removing fuzzpet_v2 and selecting op-ed from the loader:
- Map palette shows empty: `[ ] 2. MAPS      :`
- Glyph palette shows empty: `[ ] 3. GLYPHS    :`
- `[RESP]:`, `[POS]:`, `[ARMED]:` all empty
- Map display shows garbage text overlapping

---

## ROOT CAUSE

**NOT related to fuzzpet_v2 removal.** The removal was successful.

**Actual issue:** op-ed module is NOT running or NOT writing state properly.

### Evidence:

1. **op-ed process not running:**
   ```bash
   $ ps aux | grep op-ed_module
   # Only shows grep, no op-ed_module process
   ```

2. **player_app/state.txt shows wrong data:**
   ```
   project_id=template
   active_target_id=loader
   ```
   Should show:
   ```
   project_id=op-ed
   gui_focus=1
   map_palette_view=ADD [map_0001_z0.txt] map_01.txt
   glyph_palette_view=[#] . R T @ & Z X ? !
   armed_glyph=#
   ```

3. **Maps exist and are valid:**
   ```bash
   $ ls projects/op-ed/maps/
   map_0001_z0.txt  map_01.txt
   
   $ cat projects/op-ed/maps/map_0001_z0.txt
   ####################
   #                  #
   #      DEMO-0      #
   ...
   ```

---

## WHY THIS HAPPENED

The issue is a **timing/state synchronization problem**, not caused by fuzzpet_v2 removal:

1. User selected op-ed from loader
2. Loader set `project_id=op-ed` in manager state
3. op-ed started but exited (or never properly initialized)
4. Display is showing stale/corrupted frame

### Possible Causes:

1. **op-ed crashed on startup** - need to check stderr
2. **Layout switched but op-ed didn't start** - orchestrator issue
3. **op-ed is running but not writing state** - bug in write_editor_state()
