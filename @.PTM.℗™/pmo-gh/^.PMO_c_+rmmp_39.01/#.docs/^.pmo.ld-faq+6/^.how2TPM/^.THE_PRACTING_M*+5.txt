1. INTRODUCTION TO TRUE PIECE METHOD (TPM)
=========================================

True Piece Method (TPM) is an architectural approach where everything in your application is represented as "pieces" - simple text file directories that define all application state and behavior. Unlike traditional architectures that mix code and data, TPM separates concerns completely:

- Data/Persistence: Represented as text files in directory structure
- Logic: Implemented as functions in plugins
- Flow: Managed through a master ledger audit trail

TPM treats all application components as "living souls" stored as humble directories containing text files that define reality. Code is merely an interpreter, never an originator.

2. CORE TPM PRINCIPLES
======================

PRINCIPLE 1: EVERYTHING IS A PIECE
- Every application component is a directory with a piece.txt file
- Examples: users, items, UI panels, clocks, ledgers, input handlers
- Each piece maintains its own state, methods, event listeners, and responses

PRINCIPLE 2: DATA SOVEREIGNTY
- Pieces own their state - no shared global state
- All state changes persist to disk immediately
- Files serve as the single source of truth

PRINCIPLE 3: AUDIT OBSESSION  
- Every operation logs to the master ledger
- Complete traceability of all actions
- Enables time-travel debugging and replay

PRINCIPLE 4: PIPELINE FLOW
- Sequential processing: Reader → Interpreter → Writer → Next Node
- No direct calls - everything through ledger files
- Deterministic execution order

PRINCIPLE 5: NO IN-MEMORY SHORTCUTS
- All data passes through files for auditability
- No ephemeral memory states
- Reproducible behavior from starting file state

4. PIECE DEFINITION FORMAT
=========================

A piece.txt file has this structure:
```
# Piece: piece_name
state:
  key1: value1
  key2: value2
  key3: value3
methods: method1, method2, method3
event_listeners: event1, event2, event3
responses:
  event1: Response message for event1
  event2: Response message for event2
  default: Default response when no specific match
relations:
  parent: parent_piece_name
  children: child1, child2, child3
```

Example user piece:
```
# Piece: alice_user
state:
  name: Alice
  age: 25
  email: alice@example.com
  active: true
  login_count: 0
methods: increment_login, update_email, activate, deactivate
event_listeners: user_logged_in, user_updated, account_created
responses:
  user_logged_in: Alice just logged in successfully
  user_updated: User profile updated for Alice
  account_created: New account created for Alice
  default: Unknown user event occurred
relations:
  parent: user_manager
  children:
```

5. THE THREE-MANAGER PIPELINE
=============================

TPM implements three core managers that form the system pipeline:

A) PIECE MANAGER
- Responsibilities: Load, store, and manage piece data
- Reads piece.txt files and maintains in-memory representations
- Handles state get/set operations with automatic file persistence
- Tracks piece relationships and directory structures

```python
class PieceManager:
    def __init__(self, pieces_dir='pieces'):
        self.pieces_dir = pieces_dir
        self.pieces = {}  # {piece_id: piece_data}
        self.piece_paths = {}  # {piece_id: directory_path}
        self.load_all()
    
    def load_all(self):
        for piece_dir_name in os.listdir(self.pieces_dir):
            piece_path = os.path.join(self.pieces_dir, piece_dir_name)
            if os.path.isdir(piece_path):
                piece_txt_path = os.path.join(piece_path, 'piece.txt')
                if os.path.exists(piece_txt_path):
                    self.piece_paths[piece_dir_name] = piece_path
                    self.pieces[piece_dir_name] = self.parse_piece(piece_txt_path)
    
    def get_state(self, piece_id, key=None):
        state = self.pieces.get(piece_id, {}).get('state', {})
        if key:
            return state.get(key)
        return state
    
    def set_state(self, piece_id, key, value):
        if piece_id not in self.pieces:
            return
        self.pieces[piece_id]['state'][key] = str(value)
        self.save_piece(piece_id)
        
        # Log to master ledger for audit
        import time
        timestamp = time.strftime("%Y-%m-%d %H:%M:%S")
        log_entry = f"[{timestamp}] StateChange: {piece_id} {key} {value}\n"
        with open('pieces/master_ledger/master_ledger.txt', 'a') as ledger:
            ledger.write(log_entry)
        
        # Trigger state change event
        event_manager.fire('state_changed', piece_id)
```

B) METHODS MANAGER
- Responsibilities: Maintains registry of callable methods
- Loads methods from plugins dynamically
- Executes methods on pieces with proper context
- Routes method calls through the master ledger

```python
class MethodsManager:
    def __init__(self):
        self.methods = {}  # {method_name: function_object}
    
    def register(self, method_name, func):
        self.methods[method_name] = func
    
    def call(self, method_name, piece_id, *args, **kwargs):
        if method_name in self.methods:
            return self.methods[method_name](piece_id, *args, **kwargs)
        print(f"ERROR: Method '{method_name}' not found for {piece_id}")
        return None
```

C) EVENT MANAGER
- Responsibilities: Manages event registrations and dispatching
- Matches piece event listeners to available methods
- Triggers response methods based on events
- Ensures event propagation through the system

```python
from collections import defaultdict

class EventManager:
    def __init__(self):
        self.listeners = defaultdict(list)  # {event_name: [(piece_id, method_name)]}
    
    def register_all_listeners(self):
        self.listeners.clear()
        for piece_id, data in piece_manager.pieces.items():
            for event_name in data.get('event_listeners', []):
                respond_method = f"respond_to_{event_name}"
                if respond_method in methods_manager.methods:
                    self.listeners[event_name].append((piece_id, respond_method))
    
    def fire(self, event_name, piece_id=None):
        # Log event to master ledger
        import time
        timestamp = time.strftime("%Y-%m-%d %H:%M:%S")
        log_entry = f"[{timestamp}] EventFire: {event_name} on {piece_id or 'global'}\n"
        with open('pieces/master_ledger/master_ledger.txt', 'a') as ledger:
            ledger.write(log_entry)
        
        matching_listeners = self.listeners.get(event_name, [])
        for listener_piece_id, respond_method_name in matching_listeners:
            if listener_piece_id in piece_manager.pieces:
                methods_manager.call(respond_method_name, listener_piece_id)
```

6. MASTER LEDGER SYSTEM
=======================

The master ledger is the heart of TPM - a centralized audit trail that processes all system operations:

```
# Master ledger file: pieces/master_ledger/master_ledger.txt
[2026-02-10 18:58:25] MethodCall: feed on fuzzpet | Caller: user_input
[2026-02-10 18:58:26] StateChange: fuzzpet hunger 25 | Trigger: feed_method  
[2026-02-10 18:58:27] EventFire: fed on fuzzpet | Trigger: hunger_change
[2026-02-10 18:58:28] Response: fuzzpet fed | Message: Yum yum~ Feeling happy!
```

MASTER LEDGER MONITOR THREAD:
```python
import threading
import time

class MasterLedgerMonitor:
    def __init__(self):
        self.last_position = 0
        self.monitoring = False
        self.file_path = 'pieces/master_ledger/master_ledger.txt'
    
    def start_monitoring(self):
        self.monitoring = True
        monitor_thread = threading.Thread(target=self._monitor_loop, daemon=True)
        monitor_thread.start()
    
    def _monitor_loop(self):
        """Continuously monitor ledger for new entries"""
        while self.monitoring:
            self.process_new_entries()
            time.sleep(0.1)  # Brief pause to prevent busy-waiting
    
    def process_new_entries(self):
        """Process any new entries in the master ledger"""
        try:
            with open(self.file_path, 'r') as ledger:
                ledger.seek(self.last_position)
                new_lines = ledger.readlines()
                self.last_position = ledger.tell()
                
            for line in new_lines:
                line = line.strip()
                if not line or line.startswith('#'):
                    continue
                self.parse_and_execute_line(line)
        except FileNotFoundError:
            time.sleep(0.1)  # Wait for file to be created
    
    def parse_and_execute_line(self, line):
        """Parse a ledger entry and execute appropriate action"""
        if 'MethodCall:' in line:
            self._handle_method_call(line)
        elif 'EventFire:' in line:
            self._handle_event_fire(line)
        elif 'StateChange:' in line:
            self._handle_state_change(line)
    
    def _handle_method_call(self, line):
        # Parse: [TIMESTAMP] MethodCall: method_name on piece_id | Caller: source
        try:
            parts = line.split('MethodCall: ')[1].split(' |')[0]
            method_and_target = parts.split(' on ')
            if len(method_and_target) >= 2:
                method_name = method_and_target[0].strip()
                piece_id = method_and_target[1].strip()
                delta = methods_manager.call(method_name, piece_id)
                if delta:
                    for k, v in delta.items():
                        piece_manager.set_state(piece_id, k, v)
        except IndexError:
            pass
```

```python
def feed(piece_id, amount=20):
    from main import piece_manager
    hunger = piece_manager.get_state(piece_id, 'hunger')
    new_hunger = max(0, hunger - amount)
    happiness_boost = min(20, amount // 2)
    happiness = piece_manager.get_state(piece_id, 'happiness') + happiness_boost
    return {
        'hunger': new_hunger,
        'happiness': min(100, happiness)
    }

def play(piece_id, intensity=30):
    from main import piece_manager
    energy = piece_manager.get_state(piece_id, 'energy')
    happiness = piece_manager.get_state(piece_id, 'happiness')
    new_energy = max(0, energy - intensity)
    new_happiness = min(100, happiness + intensity // 2)
    return {
        'energy': new_energy,
        'happiness': new_happiness
    }

def respond_to_fed(piece_id):
    from main import piece_manager
    responses = piece_manager.pieces[piece_id].get('responses', {})
    message = responses.get('fed', responses.get('default', 'Unknown response'))
    print(f"[Pet] {message}")
    
    # Log response to master ledger
    import time
    timestamp = time.strftime("%Y-%m-%d %H:%M:%S")
    log_entry = f"[{timestamp}] Response: {piece_id} fed | Message: {message}\n"
    with open('pieces/master_ledger/master_ledger.txt', 'a') as ledger:
        ledger.write(log_entry)
```

8. INPUT/OUTPUT SYSTEMS
=======================

INPUT HANDLING follows the pattern of logging all input to history files first:

USER INPUT PROCESS:
1. User types command
2. Command written to master ledger as MethodCall
3. Master ledger monitor processes MethodCall
4. Appropriate method executes on target piece
5. State changes get logged to master ledger
6. Events fire based on state changes
7. Response messages get displayed

INPUT CAPTURE CODE:

# Log command to master ledger before processing

9. EVENT MANAGEMENT
===================

Events propagate through the system using the master ledger:

EVENT PROPAGATION FLOW:
1. Something causes state change
2. State change logged to master ledger
3. Event manager detects state change
4. Event manager fires appropriate event
5. Event logged to master ledger
6. Event handler methods execute
7. More state changes may result

EVENT RESPONSE PATTERN:
```python
def respond_to_event(piece_id, event_name):
    from main import piece_manager
    responses = piece_manager.pieces[piece_id].get('responses', {})
    message = responses.get(event_name, responses.get('default', 'Default message'))
    print(f"[Response] {message}")
    
    # Log response to maintain audit trail
    import time
    timestamp = time.strftime("%Y-%m-%d %H:%M:%S")
    log_entry = f"[{timestamp}] Response: {piece_id} {event_name} | Message: {message}\n"
    with open('pieces/master_ledger/master_ledger.txt', 'a') as ledger:
        ledger.write(log_entry)
```

11. FAQ, TROUBLESHOOTING & COMMON PITFALLS
==========================================

### COMMON PITFALLS (Lessons from Production Implementation)

#### Pitfall 1: Centralized Plugins Directory
**WRONG**: All plugins in root `plugins/` folder
**RIGHT**: Plugins inside `pieces/<piece>/plugins/`

```
❌ project/plugins/game_logic.c
✅ pieces/world/plugins/game_logic.c
```

**Why**: TPM requires pieces to own their behavior, not just data. Centralized plugins break modularity and discoverability.

**Fix**: Move each plugin to its piece directory:
```bash
mv plugins/keyboard_input.c pieces/keyboard/plugins/
mv plugins/game_logic.c pieces/world/plugins/
mv plugins/renderer.c pieces/display/plugins/
```

#### Pitfall 2: Overwriting piece.txt Structure
**WRONG**: `write_piece_state()` overwrites entire file with just state values
**RIGHT**: Preserve piece.txt structure (responses, methods, event_listeners)

```python
# WRONG - loses responses section, breaks modularity
def write_piece_state(self, piece_name, state_data):
    with open(state_path, 'w') as f:
        for key, value in state_data.items():
            f.write(f"{key}={value}\n")

# RIGHT - preserve structure
def write_piece_state(self, piece_name, state_data):
    # Read existing content
    with open(state_path, 'r') as f:
        lines = f.readlines()
    
    # Write back, updating only state section
    with open(state_path, 'w') as f:
        for line in lines:
            if line.strip().startswith('state:'):
                f.write('state:\n')
                for key, value in state_data.items():
                    f.write(f'  {key}={value}\n')
            # Skip old state lines, preserve rest
```

**Why**: Responses and methods are defined in piece.txt for modularity. Overwriting them breaks TPM's customization principle.

#### Pitfall 3: Using mtime for Frame Detection
**WRONG**: Check file modification time (1-second resolution)
**RIGHT**: Append to marker file, check SIZE

```c
// WRONG - misses updates within same second
time_t current_time = get_file_mod_time("pieces/display/current_frame.txt");
if (current_time != last_time) { render_display(); }

// RIGHT - catches every update
FILE *marker = fopen("pieces/master_ledger/frame_changed.txt", "a");
fprintf(marker, "X\n");  // Append-only, size always grows

// In renderer:
struct stat st;
stat("pieces/master_ledger/frame_changed.txt", &st);
if (st.st_size != last_size) {
    render_display();
    last_size = st.st_size;
}
```

**Why**: mtime has 1-second resolution. Multiple updates in same second are missed. Marker file size is exact.

#### Pitfall 4: Hardcoded Response Messages
**WRONG**: `strcpy(last_command_output, "Yum yum~");`
**RIGHT**: Fetch from piece.txt via piece_manager

```c
// WRONG - not modular, can't customize per installation
strcpy(last_command_output, "Yum yum~ Feeling happy!");

// RIGHT - modular, customizable without code changes
char response_cmd[300], response[500];
sprintf(response_cmd, "./+x/piece_manager.+x fuzzpet get-response fed");
FILE *rsp_fp = popen(response_cmd, "r");
if (rsp_fp) {
    fgets(response, sizeof(response), rsp_fp);
    sprintf(last_command_output, "[FUZZPET] %s", response);
    pclose(rsp_fp);
}
```

**Why**: Responses in piece.txt allow customization without code changes. Hardcoding defeats TPM modularity.

#### Pitfall 5: Relative Path Confusion
**WRONG**: `../pieces/keyboard/history.txt` (creates nested directories)
**RIGHT**: Calculate paths from project root

```python
# WRONG - creates pieces/pieces/... nested directories
HISTORY_PATH = os.path.join(script_dir, '../pieces/keyboard/history.txt')

# RIGHT - from pieces/world/plugins/, go up 2 levels to project root
PIECES_ROOT = os.path.abspath(os.path.join(script_dir, '../../'))
HISTORY_PATH = os.path.join(PIECES_ROOT, 'pieces/keyboard/history.txt')
```

**Why**: Incorrect relative paths create nested `pieces/pieces/` directories and file not found errors.

#### Pitfall 6: Response Handler Printing Instead of Updating Frame
**WRONG**: response_handler prints to stdout (lost in background)
**RIGHT**: game_logic sets `last_command_output` before render

```c
// WRONG - response_handler runs in background, stdout redirected to /dev/null
printf("[FUZZPET] Yum yum~");  // Never seen by user

// RIGHT - included in next frame render
strcpy(last_command_output, "[FUZZPET] Yum yum~");
render_print();  // Includes message in frame
```

**Why**: Background processes have stdout redirected. Responses must be in frame file to be visible.

### FAQ

Q: Why use directories for pieces instead of single files?
A: Directories allow for multiple related files per piece (piece.txt, ledger.txt, history.txt, other data). It also makes the system more modular and organized.

Q: What happens if multiple threads try to write to the master ledger?
A: Use file locking or atomic write operations. Consider using file locking mechanisms like flock() on Unix systems or similar approaches on Windows.

Q: How do I handle recursive calls without infinite loops?
A: Track call depth in the ledger and implement cycle detection. Limit recursion depth (e.g., max 5 levels deep).

Q: How do I ensure data consistency with file-based storage?
A: Write to temporary files first, then atomically rename. Use checksums or validation to ensure file integrity.

Q: What if the master ledger grows too large?
A: Implement log rotation, archival, or compression. Periodically archive older entries to separate files.

Q: How do I debug when nothing seems to be happening?
A: Check the master ledger file - all operations should be logged there. Also check that your event listeners are registered properly.

Q: Why isn't my event response appearing?
A: Ensure the event name in responses matches the listener name, and that the respond_to_event_name method exists in your plugin.

Q: My piece data isn't saving between runs?
A: Make sure your set_state() method writes back to the piece.txt file each time state changes.

Q: How do I ensure state persists across system restarts?
A: Always save state files after changes (not just in memory), keep multiple state representations in sync, and verify that the files contain the expected state when the system starts up.

Q: How do I handle complex data structures?
A: Keep it simple in piece.txt (flat key-value pairs). Store complex data in separate files within the piece directory if needed.

Q: What's the performance like with file-based operations?
A: For most applications, file I/O is fast enough. For high-performance needs, consider caching frequently accessed pieces in memory with automatic sync policies.

14. NEW: FRAME CHANGE DETECTION (MARKER-FILE PATTERN)
=====================================================

PROBLEM: File modification time (mtime) has 1-second resolution, causing missed frame updates when multiple changes occur within the same second.

SOLUTION: Append-only marker file that grows with every frame update:

GAME_LOGIC.C - After writing frame:
```c
void render_print() {
    FILE *fp2 = fopen("pieces/display/current_frame.txt", "w");
    // ... write frame content ...
    fclose(fp2);
    
    // Append marker to signal frame change
    FILE *marker = fopen("pieces/master_ledger/frame_changed.txt", "a");
    if (marker) {
        fprintf(marker, "X\n");
        fclose(marker);
    }
}
```

RENDERER.C - Check marker SIZE (not mtime!):
```c
off_t last_marker_size = 0;

void timer(int value) {
    struct stat st;
    if (stat("pieces/master_ledger/frame_changed.txt", &st) == 0) {
        if (st.st_size != last_marker_size) {
            // Frame was updated!
            render_display();
            last_marker_size = st.st_size;
        }
    }
    glutPostRedisplay();
    glutTimerFunc(16, timer, 0);  // 60 FPS
}
```

WHY THIS WORKS:
- Marker file only grows, never shrinks
- Every write increases size by at least 1 byte
- No resolution issues - size is exact
- Works even with 100+ updates per second
- Same pattern works for both ASCII and GL renderers

15. NEW: DUAL RENDERER ARCHITECTURE (ASCII + GL)
================================================

Both renderers read from the SAME frame file and check the SAME marker:

ASCII RENDERER (renderer.c / renderer.py):

- Output: Terminal stdout

16. NEW: ORCHESTRATOR PATTERN WITH THREADS
==========================================

The orchestrator launches and manages all components as concurrent threads:

ORCHESTRATOR.C:
```c
#include <pthread.h>
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

void* keyboard_thread_func(void* arg) {
    system("./+x/keyboard_input.+x");
    pthread_exit(NULL);
}

void* game_thread_func(void* arg) {
    system("./+x/game_logic.+x");
    pthread_exit(NULL);
}

void* render_thread_func(void* arg) {
    system("./+x/renderer.+x");
    pthread_exit(NULL);
}

void* gl_render_thread_func(void* arg) {
    system("./+x/gl_renderer.+x");
    pthread_exit(NULL);
}

void* response_thread_func(void* arg) {
    system("./+x/response_handler.+x");
    pthread_exit(NULL);
}

int main() {
    pthread_t kb_thread, game_thread, render_thread, gl_render_thread, response_thread;
    
    // Create threads for each component
    pthread_create(&kb_thread, NULL, keyboard_thread_func, NULL);
    pthread_create(&game_thread, NULL, game_thread_func, NULL);
    pthread_create(&render_thread, NULL, render_thread_func, NULL);
    pthread_create(&gl_render_thread, NULL, gl_render_thread_func, NULL);
    pthread_create(&response_thread, NULL, response_thread_func, NULL);
    
    // Wait for all threads
    pthread_join(kb_thread, NULL);
    pthread_join(game_thread, NULL);
    pthread_join(render_thread, NULL);
    pthread_join(gl_render_thread, NULL);
    pthread_join(response_thread, NULL);
    
    pthread_exit(NULL);
    return 0;
}
```
