## 2. HISTORICAL CONTEXT

### Why PDL Was Created
The PDL (Piece Definition Ledger) format was an **experiment** in creating a structured, human-readable piece definition format that could:
- Serve as a "source of truth" for piece properties
- Define method bindings (which ops handle which events)
- Potentially support hot-reloading or scripting

### What Actually Happened
- `state.txt` remained the **actual** source of truth (easier to parse/update)
- PDL files are written by `create_piece.c` but **never read**
- PDL files become **stale** when state changes (no sync mechanism)
- No code in the codebase reads PDL files for runtime data

---

## 3. PROS AND CONS

### state.txt

| Pros | Cons |
|------|------|
| ✅ Simple key=value format | ❌ No schema validation |
| ✅ Easy to read/write in C | ❌ No type information |
| ✅ Easy to append/update fields | ❌ No built-in documentation |
| ✅ Directly used by all runtime code | ❌ Can become malformed |
| ✅ Easy to grep/parse in shell scripts | ❌ No support for nested data |
| ✅ Backward compatible (unknown fields ignored) | |

### piece.pdl

| Pros | Cons |
|------|------|
| ✅ Structured, tabular format | ❌ **NOT USED BY ANY CODE** |
| ✅ Clear column alignment (readable) | ❌ Duplicates state.txt (redundant) |
| ✅ METHOD section for op bindings | ❌ Becomes stale immediately |
| ✅ META section for documentation | ❌ Extra I/O during piece creation |
| ✅ Could support advanced features | ❌ Confusing for new developers |
| | ❌ Wastes disk space (~200 bytes per piece) |

## 6. WHAT ABOUT METHOD BINDINGS?

The PDL format has a `METHOD` section:
```
METHOD       | on_interact        | ./pieces/apps/playrm/ops/+x/interact.+x npc_7_5
```

### Current Reality
- Method bindings are **hardcoded** in ops anyway
- `interact.c` doesn't check PDL for what to run
- The PDL METHOD line is **documentation only**

### If You Want Method Bindings
Better approaches:
1. **Convention-based:** `npc_*` pieces always use `interact.+x`
2. **Type-based:** Store `handler=interact` in state.txt
3. **Future:** Add to project.pdl (project-level, not piece-level)

## 7. RECOMMENDATION

### Remove PDL Files

**Reasons:**
1. **Not used** - Zero runtime value
2. **Redundant** - Duplicates state.txt
3. **Confusing** - Developers wonder which to read/update
4. **Stale data** - Becomes wrong immediately after creation
5. **Easy removal** - Only 1 file to modify (`create_piece.c`)

### Migration Path

**Phase 1: Stop Creating PDL**
- Remove PDL writing from `create_piece.c`
- New pieces won't have PDL files

**Phase 2: Optional Cleanup**
```bash
# Delete all existing PDL files
find projects/ -name "piece.pdl" -delete
```

**Phase 3: Update Documentation**
- Remove PDL references from docs
- Clarify that `state.txt` is the single source of truth

## 8. COUNTERARGUMENTS (Why Keep PDL?)

### "It's Good Documentation"
- **Rebuttal:** Stale documentation is worse than no documentation
- **Better:** Comment in state.txt or separate README

### "We Might Use It Later"
- **Rebuttal:** YAGNI (You Aren't Gonna Need It)
- **Better:** Add back when there's an actual use case

### "It Looks Professional"
- **Rebuttal:** Form over function
- **Better:** Professional = working, maintainable code

### "Future Scripting System"
- **Rebuttal:** If needed, use a different format (JSON, Lua, etc.)
- **Better:** Design when requirements are known

## 12. DEVIL'S ADVOCATE: Why PDL Might Be The Real Source of Truth

**Contributor:** User (2026-03-15)

### The Original Vision (Not Yet Implemented)

The METHOD section in PDL was **never supposed to be documentation**. It was supposed to be the **runtime dispatch table**:

```
# piece.pdl
METHOD       | on_interact        | ./pieces/apps/playrm/ops/+x/interact.+x npc_7_5
METHOD       | on_turn            | ./pieces/apps/playrm/ops/+x/zombie_ai.+x npc_7_5
METHOD       | on_trade           | ./pieces/apps/playrm/ops/+x/trade.+x npc_7_5
```

### The Problem: Current Code Hardcodes Everything

**Current `render_map.c` / `interact.c`:**
```c
// HARDCODED - INFLEXIBLE
if (strcmp(type, "npc") == 0) {
    system("./pieces/apps/playrm/ops/+x/interact.+x npc_7_5");
}
```

**What It SHOULD Be:**
```c
// FLEXIBLE - Read from PDL at runtime
char handler[MAX_PATH];
get_method_from_pdl(piece_id, "on_interact", handler);
if (strlen(handler) > 0) {
    system(handler);  // Whatever the piece defines
}
```

---

### Future Use Cases That Require PDL

#### 1. Tradable Pieces with Unique Behaviors

```
# pieces/rare_merchant_01/piece.pdl
META         | piece_id           | rare_merchant_01
STATE        | type               | merchant
METHOD       | on_interact        | ./pieces/apps/playrm/ops/+x/rare_trade.+x rare_merchant_01
METHOD       | on_trade           | ./pieces/apps/playrm/ops/+x/negotiate_price.+x rare_merchant_01
```

```
# pieces/common_merchant_01/piece.pdl
META         | piece_id           | common_merchant_01
STATE        | type               | merchant
METHOD       | on_interact        | ./pieces/apps/playrm/ops/+x/interact.+x common_merchant_01
METHOD       | on_trade           | ./pieces/apps/playrm/ops/+x/standard_trade.+x common_merchant_01
```

**Same type, different handlers.** Hardcoding can't support this.

---

#### 2. Learnable/Forgettable Skills

A piece gains/loses capabilities as it levels up:

```
# pieces/hero/piece.pdl - Level 1
METHOD       | on_attack          | ./pieces/apps/playrm/ops/+x/basic_attack.+x hero

# pieces/hero/piece.pdl - Level 5 (learned fireball)
METHOD       | on_attack          | ./pieces/apps/playrm/ops/+x/basic_attack.+x hero
METHOD       | on_cast_fire       | ./pieces/apps/playrm/ops/+x/fireball.+x hero

# pieces/hero/piece.pdl - Level 10 (forgot basic, learned ultimate)
METHOD       | on_attack          | ./pieces/apps/playrm/ops/+x/ultimate_attack.+x hero
METHOD       | on_cast_fire       | ./pieces/apps/playrm/ops/+x/inferno.+x hero
```

**Skills change at runtime.** PDL would need to be **rewritten** when skills change, or we need a hybrid approach.

---

#### 3. Piece Trading Between Projects

```
# Export piece from project A
cp -r projects/rpg1/pieces/hero/ projects/rpg2/pieces/

# hero.pdl travels with the piece:
METHOD       | on_interact        | ./pieces/apps/playrm/ops/+x/hero_dialogue.+x hero
METHOD       | on_quest           | ./pieces/apps/playrm/ops/+x/hero_quests.+x hero
```

**Self-contained piece definition.** The receiving project doesn't need to know what handlers the piece needs - it's all in the PDL.

---

### The Real Problem: PDL Was Half-Implemented

**What Was Done:**
- ✅ PDL format designed
- ✅ `create_piece.c` writes PDL
- ❌ No code reads PDL at runtime
- ❌ No `get_method_from_pdl()` function
- ❌ Ops still hardcoded

**What Should Have Been Done:**
1. Create `read_pdl.c` utility
2. Add `get_method(piece_id, event_name, out_handler)` function
3. Modify ops to call `get_method()` instead of hardcoding
4. PDL becomes the **capability definition**
5. state.txt becomes **mutable runtime state**

---

### Proposed Architecture (PDL as Source of Truth)

```
┌─────────────────────────────────────────────────────────────┐
│                     PIECE DEFINITION                         │
├─────────────────────────────────────────────────────────────┤
│  piece.pdl (Immutable Capability Definition)                │
│  ├── META: piece_id, type, version                          │
│  ├── STATE: default values (pos, stats)                     │
│  └── METHOD: Event → Handler mappings                       │
└─────────────────────────────────────────────────────────────┘
                          │
                          │ Runtime reads
                          ▼
┌─────────────────────────────────────────────────────────────┐
│                     RUNTIME STATE                            │
├─────────────────────────────────────────────────────────────┤
│  state.txt (Mutable Runtime Data)                           │
│  ├── Current pos_x, pos_y, pos_z                            │
│  ├── Current on_map status                                  │
│  ├── Modified stats (hunger, happiness, etc.)               │
│  └── map_id association                                     │
└─────────────────────────────────────────────────────────────┘
```

**Key Distinction:**
- **PDL** = What this piece CAN do (capabilities, handlers)
- **state.txt** = What this piece IS doing right now (position, status)

### Revised Recommendation

**DON'T remove PDL.** Instead:

1. **Keep PDL** as the capability/skill definition
2. **Keep state.txt** as mutable runtime state
3. **Implement PDL reading** in runtime code
4. **Remove hardcoded handlers** from ops
5. **Document the distinction** clearly

**What to Remove Instead:**
- Hardcoded type→handler mappings in ops
- The assumption that "all NPCs use interact.+x"

**What to Build:**
- `pdl_reader.c` - Read PDL files at runtime
- `get_method()` - Lookup handler by event name
- `add_method()` / `remove_method()` - For skill changes

---

### When PDL Becomes Essential

| Feature | Requires PDL? |
|---------|---------------|
| Basic pieces with fixed behavior | ❌ No |
| Pieces with custom handlers | ✅ Yes |
| Tradable pieces with unique skills | ✅ Yes |
| Learnable/forgettable skills | ✅ Yes |
| Piece-level scripting | ✅ Yes |
| Mod support (custom piece types) | ✅ Yes |

**If you want ANY of the right-column features, PDL (or equivalent) is mandatory.**

### Files to Create

**`pieces/system/pdl/pdl_reader.c`**
```c
/*
 * pdl_reader.c - Read PDL files and extract method handlers
 */
#include <stdio.h>
#include <string.h>

#define MAX_PATH 4096
#define MAX_LINE 1024

/**
 * Get method handler for a piece's event
 * 
 * @param piece_id The piece name (e.g., "zombie_01")
 * @param event The event name (e.g., "on_turn")
 * @param out_handler Buffer to store handler path
 * @param project_root Root project path
 * @param current_project Current project name
 * @return 0 on success, -1 if not found
 */
int get_method(const char* piece_id, const char* event, 
               char* out_handler, size_t out_size,
               const char* project_root, const char* current_project) {
    
    char pdl_path[MAX_PATH];
    snprintf(pdl_path, sizeof(pdl_path), "%s/projects/%s/pieces/%s/piece.pdl",
             project_root, current_project, piece_id);
    
    FILE *f = fopen(pdl_path, "r");
    if (!f) return -1;
    
    char line[MAX_LINE];
    while (fgets(line, sizeof(line), f)) {
        // Look for: METHOD | event | handler
        if (strncmp(line, "METHOD", 6) == 0) {
            char parsed_event[64], parsed_handler[MAX_PATH];
            if (sscanf(line, "METHOD | %63[^|] | %4095s", parsed_event, parsed_handler) == 2) {
                // Trim whitespace
                char *p = parsed_event + strlen(parsed_event) - 1;
                while (p > parsed_event && *p == ' ') *p-- = '\0';
                p = parsed_handler + strlen(parsed_handler) - 1;
                while (p > parsed_handler && *p == ' ') *p-- = '\0';
                
                if (strcmp(parsed_event, event) == 0) {
                    strncpy(out_handler, parsed_handler, out_size - 1);
                    fclose(f);
                    return 0;
                }
            }
        }
    }
    
    fclose(f);
    return -1;  // Not found
}
```
