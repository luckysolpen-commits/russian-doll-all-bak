# Enter Key Parity Analysis & Fix Strategy (Windows)
Date: April 1, 2026
Status: FINAL RESEARCH / READY FOR EXECUTION

## 1. THE PHENOMENON
- **Numerical Jumps WORK:** Typing a number (e.g., "1") and pressing [ENTER] successfully jumps the focus to that item.
- **Direct Activation FAILS:** Pressing [ENTER] on a currently focused item (selected via 'w/s' or after a jump) does NOT trigger the button's action (e.g., LOAD_PROJECT or LAUNCH).

## 2. THE SMOKING GUN: WHY JUMPS WORK
Numerical jumps work because they hit the FIRST block of the Enter handler in `chtpm_parser.c`:
```c
else if (key == 10 || key == 13 || key == JOY_BUTTON_0) {
    if (digit_accum > 0) {
        // JUMP LOGIC: This branch is executing!
        if (do_jump(digit_accum)) { ... }
        digit_accum = 0;
    } else if (focus_index >= 0 && focus_index < element_count) {
        // BUTTON LOGIC: This branch is failing or being bypassed!
    }
}
```
**Conclusion:** The code 10/13 is definitely reaching the parser. The hardware/orchestrator pipeline is functional.

## 3. ROOT CAUSE HYPOTHESES

### Hypothesis A: The "Ghost" Double-Enter (CRLF)
On Windows, Enter often sends `\r\n` (13 followed by 10).
1. Orchestrator captures `13`, writes to history.
2. Orchestrator captures `10`, writes to history.
3. Parser reads `13`: `digit_accum` is > 0 (from the number you typed), JUMP happens, `digit_accum` reset to 0.
4. Parser reads `10`: `digit_accum` is now 0. It enters the `else if (focus_index >= 0)` block immediately.
**Impact:** It might be "double-clicking" or triggering an action so fast that the UI hasn't finished reloading the layout from the jump, leading to a race condition or a crash.

### Hypothesis B: `atoi` Fragility with `\r`
The parser uses `atoi(strstr(line, "KEY_PRESSED: ") + 13)`.
If a line ends in `13\r\n` (Windows CRLF), the `13` is at the end of the string.
While `atoi` is usually robust, if there's any hidden character or if the file pointer `last_history_position` is misaligned by 1 byte due to CRLF counting, the parser might be reading garbage.

### Hypothesis C: Focus Index Desync
In `chtpm_parser.c`, `do_jump` updates `focus_index`.
However, if the `Enter` is processed in the same "dirty" cycle as the jump, the `elements` array might be in a state of flux, or `is_navigable(focus_index)` might be returning false for the new target until the next `parse_chtm` call.

## 4. PROPOSED SOLUTION: "THE STABILIZER"

### 1. Robust Key Parsing
Replace the brittle `+ 13` offset with a colon-seeking pointer.
```c
char *p = strstr(line, "KEY_PRESSED: ");
if (p) {
    p = strchr(p, ':'); // Find the colon
    if (p) {
        int key = atoi(p + 1); // Skip colon and space
        process_key(key);
    }
}
```

### 2. Enter-Key Debouncing
Ignore a `10` if it immediately follows a `13` (and vice-versa) within a few milliseconds, or simply treat them as identical and suppress the second one if the first already triggered a "heavy" action like a jump.

### 3. State-Locked Execution
Ensure that `Enter` only triggers the `else` block (Button Logic) if `digit_accum` was ALREADY 0 at the start of the process cycle. This prevents a single "Enter" from being split across two codes and triggering both a jump and a button-press.

## 5. EDGE CASES
- **MSYS2/mintty:** Might only send `10`. The fix must support single-code Enters.
- **Joystick (JOY_BUTTON_0):** Does not send a sequence, only one code.
- **Fast Typists:** Must not drop legitimate double-enters (e.g., clicking a button twice).

## 6. GUARANTEE OF SUCCESS
By normalizing the input capture (handling 10/13 as a single logical "Submit" event) and ensuring the parser logic is state-aware, we eliminate the Windows-specific "Double-Enter" race condition while maintaining Linux parity.
