## 2. NEW WISDOMS DISCOVERED (v0.2 Additions)

### 2.1 The "Dual Injection Bug" (CRITICAL DISCOVERY)

**The Bug:** The C version's `send_command()` was injecting BOTH numeric AND joystick codes:

```c
// BROKEN - sends BOTH codes for single UI click
if (strcmp(cmd, "FEED") == 0) { 
    inject_raw_key('2');              // ASCII 50
    inject_raw_key(JOY_BUTTON_0 + 1); // 2001
}
```

**Why It's Wrong:** When user clicks "Feed Pet" with keyboard/mouse:
- Only '2' should be sent (keyboard input)
- J-B1 (2001) should ONLY be sent when actual joystick button is pressed

**The Fix:**
```c
// CORRECT - send only numeric code for UI click
if (strcmp(cmd, "FEED") == 0) { 
    inject_raw_key('2');  // Keyboard parity only
}
```

**The Wisdom:** **Input parity is about equivalence, not duplication.** Keyboard and joystick should be able to trigger the same action, but each input method sends its own code. Don't inject both for a single action.

---

### 2.2 The "Nav Buffer Persistence" Discovery

**The Pattern:** Nav buffer should persist until the next key press (for debugging):

```
User presses '2' → Nav > 2_
User presses Enter → Nav > ENTER_  (stays!)
User presses 's' → Nav > s_  (cleared by new key)
```

**The Anti-Pattern:** Auto-clearing nav buffer after each action:
```javascript
// WRONG - clears too early
parser.navBuffer = "";
clearNavOnNext = true;
```

**The Wisdom:** **Debug visibility trumps aesthetic cleanliness.** The nav buffer is a diagnostic tool - it should show what key was pressed, persisting until replaced by new input. This helps users understand the system state.

---

### 2.3 The "Wait-For-View Trap" (New Conditional Pulse Variant)

**The Bug:** We tried to wait for `view_changed.txt` before composing:

```javascript
// WRONG - skips pulse() when waiting!
if (waitForViewChange) {
    setTimeout(renderLoop, 100);
    return;  // Never pulses!
}
await module.pulse();  // Never reached!
```

**Why It's Wrong:** The orchestrator can't process history if `pulse()` isn't called. The view never changes because the module never reads the command.

**The Fix:**
```javascript
// CORRECT - ALWAYS pulse, even when waiting
await module.pulse();  // Always called

// Check marker, but don't skip pulse
if (waitForViewChange && !viewChanged) {
    setTimeout(renderLoop, 100);
    return;
}
```

**The Wisdom:** **The module's heartbeat is sacred.** Never skip `pulse()` - it's the module's only chance to process input. Waiting for markers is about *composition timing*, not *processing timing*.

---

### 2.4 The "Response Tracking" Discovery

**The Bug:** PHP orchestrator had hardcoded response logic:

```javascript
// WRONG - response based on state threshold
const responseText = this.state.hunger < 20 ? "Yum yum~" : "*moves around excitedly*";
```

**Why It's Wrong:** The response should reflect the *last action*, not the current state. After feeding, show "fed" response regardless of hunger level.

**The Fix:**
```javascript
// CORRECT - track which response to show
this.currentResponseKey = "default";

// In processKey()
if (code === 50) {
    this.currentResponseKey = "fed";  // Track the action
}

// In writeSovereignView()
const responses = {
    fed: "Yum yum~ Feeling so happy!",
    played: "Wheee~ Best playtime ever!",
    default: "*confused tilt* ...huh?"
};
const responseText = responses[this.currentResponseKey];
```

**The Wisdom:** **State and response are separate concerns.** State is the data (hunger=40), response is the narrative ("Yum yum~"). Track them separately.

---

### 2.5 The "Frame Spacing" Discovery

**The Bug:** C renderer was padding with full screen height (50-75 lines):

```c
// WRONG - excessive padding
for(int i = 0; i < w.ws_row; i++) printf("\n");  // 24-50 lines!
```

**The Fix:**
```c
// CORRECT - minimal spacing
printf("\n\n\n\n\n");  // 5 lines
```

**The Wisdom:** **Scroll fatigue is real.** Users shouldn't need to scroll 50 lines to see the current frame. 5-10 lines of separation is enough for visual clarity.

---

### 2.6 The "Marker File Initialization" Discovery

**The Bug:** PHP version didn't create `view_changed.txt` on startup:

```bash
# php_server++.sh was missing:
> pieces/apps/fuzzpet_app/fuzzpet/view_changed.txt
```

**Why It Matters:** If the file doesn't exist, the parser's `stat()` call fails, and `waitForViewChange` never resolves.

**The Wisdom:** **Initialize all marker files.** Every `.txt` file that will be monitored should be created (even if empty) during initialization.

```
1. INPUT CAPTURE:
   keyboard_input.+x OR joystick_input.+x writes to keyboard/history.txt
   - Keyboard: ASCII codes (97='a', 119='w', etc.)
   - Joystick: 2000-range (buttons), 2100-range (axes)

2. RELAY (OS LAYER - chtpm_parser):

   Reads fuzzpet/history.txt (ALWAYS pulsing)

4. COMPOSITION (OS LAYER - chtpm_parser):
   Reads fuzzpet/view.txt → substitutes into ${game_map}
   - Writes current_frame.txt
   - Appends to frame_changed.txt (marker)
   - Nav buffer persists until next key (debug visibility)

5. RENDERING:
   Detects frame_changed.txt size delta
   - Clears screen (or pads 5 lines)
   - Prints current_frame.txt
```

## 3. NEW WISDOMS DISCOVERED (v0.3 - Clock Integration Era)

**Date:** 2026-03-05  
**Context:** CHTPM Stage 2 - Centralized Clock Daemon Implementation

### 3.1 The "Silent Daemon" Pattern

**The Pattern:** Background processes that tick continuously without user input:

```c
while (1) {
    read_state(state_path, turn, time_str, mode);
    if (strcmp(mode, "auto") == 0) {
        increment_time_str(time_str);
        write_state(state_path, turn, time_str, mode);
        hit_frame_marker();  // CRITICAL: triggers re-render
    }
    sleep(1);
}
```

**The Wisdom:** **Daemons must hit the global frame marker.** Without this, the clock ticks but the UI never updates. The `frame_changed.txt` append is the daemon's "voice" to the renderer.

**The Anti-Pattern:** Updating state without signaling:
```c
// WRONG - state changes but no one knows
write_state(state_path, turn, time_str, mode);
// Missing: hit_frame_marker();
```

---

### 3.2 The "Location KVP" Discovery

**The Pattern:** All paths resolve through `pieces/locations/location_kvp`:

```bash
project_root=/absolute/path
clock_daemon_dir=${project_root}/pieces/system/clock_daemon
```

```c
void resolve_root() {
    FILE *kvp = fopen("pieces/locations/location_kvp", "r");
    // ... read project_root ...
}
```

**The Wisdom:** **Never hardcode relative paths.** Always resolve `project_root` from `location_kvp` first, then build absolute paths. This prevents `system()` calls from failing silently when the working directory changes.

**The Anti-Pattern:**
```c
// WRONG - fails if cwd changes
fopen("pieces/system/clock_daemon/state.txt", "r");

// RIGHT - always works
snprintf(path, sizeof(path), "%s/pieces/system/clock_daemon/state.txt", project_root);
```

---

### 3.3 The "Method Parity" Discovery

**The Pattern:** Adding a method requires only 2 files (plus optional response):

1. **`.pdl`** - Declare `METHOD | toggle_clock | void`
2. **`fuzzpet_manager.c`** - Handle the method in `route_input()`
3. **Layout** - Uses `${piece_methods}` - auto-generates button (no changes needed!)

**The Wisdom:** **Trust the DNA.** If the `.pdl` has the method and the manager handles it, the UI will show it. Don't hardcode redundant buttons.

**The Anti-Pattern:**
```html
<!-- WRONG - hardcoded button duplicates DNA -->
<button label="Toggle Clock" onClick="KEY:5" />
<!-- The .pdl already defines this - let the parser generate it -->
```

---

### 3.4 The "Buffer Size Discipline" Discovery

**The Pattern:** GCC's `-Wformat-truncation` warns about potential string truncation:

```c
// WRONG - triggers warning
#define MAX_PATH 2048
char path[MAX_PATH];
snprintf(path, sizeof(path), "%s/pieces/...", project_root);  // Warning!

// RIGHT - safe buffer sizes
#define MAX_PATH 4096  // Double what you think you need
char path[MAX_PATH];
snprintf(path, sizeof(path), "%s/pieces/...", project_root);  // No warning
```

**The Wisdom:** **Buffer warnings are technical debt.** Fix them at the source by using generous buffer sizes. Don't ignore warnings - they compound.

---

### 3.5 The "Clock Daemon Architecture" Discovery

**The Pattern:** Centralized time management via a system piece:

```
pieces/system/clock_daemon/
├── clock_daemon.pdl    # DNA: turn, time, mode, tick_rate
├── state.txt           # Mirror: key=value for fast reads
└── plugins/
    └── clock_daemon.c  # Daemon logic: tick, update, signal
```

**Integration Points:**
- **Parser:** Loads `${clock_time}`, `${clock_turn}`, `${clock_mode}` in `load_vars()`
- **Layout:** Displays via `System Time: ${clock_time} | Turn: ${clock_turn}`
- **Manager:** Handles `toggle_clock` method to flip mode
- **Orchestrator:** Launches daemon as background thread

**The Wisdom:** **Time is a Piece.** The clock follows the same DNA/Mirror/Plugin pattern as every other entity. This makes it auditable, toggleable, and replaceable.

---

### 3.6 The "Turn vs Time Mode" Discovery

**The Pattern:** Two clock modes with different semantics:

```c
// AUTO MODE: Time ticks every second, turn every 60 seconds
if (strcmp(mode, "auto") == 0) {
    increment_time_str(time_str);  // Every second
    auto_tick_count++;
    if (auto_tick_count >= 60) {
        increment_turn(turn);  // Every minute
    }
}
// TURN MODE: Daemon does nothing - waits for external trigger

// increment_turn.c - called by user action:
increment_turn(turn);       // Turn = action count
increment_time_str(time);   // Each action takes time
```

**The Wisdom:** **Turn ≠ Time in turn mode.** 
- **AUTO mode:** `turn` = elapsed time (1 turn = 1 minute)
- **TURN mode:** `turn` = action count, time advances with each action

**The Anti-Pattern:**
```c
// WRONG - turn mode should NOT tick automatically
if (strcmp(mode, "turn") == 0) {
    increment_time_str(time_str);  // Don't do this!
}
```

---

### 3.7 The "Precise Process Killing" Discovery

**The Bug:** `pkill -f "\.\+x"` kills ANY process with `.x` in the command line:
```bash
# WRONG - kills Chrome, Firefox, etc.
pkill -f "\.\+x"
```

**The Fix:** Match specific binary paths:
```bash
# RIGHT - precise matching
pkill -f "pieces/chtpm/plugins/\+x/orchestrator\.\+x"
pkill -f "pieces/system/clock_daemon/plugins/\+x/clock_daemon\.\+x"
```

**The Wisdom:** **Pattern matching is a double-edged sword.** Broad patterns kill innocent processes. Match specifically, kill surgically.

---

### 3.8 The "Clean Compilation" Discovery

**The Pattern:** GCC's `-Wformat-truncation` warns about potential truncation even with large buffers:

```c
// WRONG - triggers warning even with 8192 buffer
#define MAX_PATH 8192
char path[MAX_PATH];
snprintf(path, sizeof(path), "%s/pieces/...", project_root);

// RIGHT - use asprintf for dynamic allocation
#define _GNU_SOURCE
char *path;
asprintf(&path, "%s/pieces/...", project_root);
// ... use path ...
free(path);
```

**The Wisdom:** **Warnings are technical debt.** Use `asprintf()` for path concatenation - it allocates exactly what's needed and compiles clean.

---

### 3.9 The "Parser Sovereignty" Discovery

**The Mantra:** "The Parser is the mirror's eye, not the mirror's mind."

**The Pattern:** The Parser (`chtpm_parser.c`) has exactly three responsibilities:
1. Substitute `${variables}` from state files
2. Route input (nav vs interact mode)
3. Compose the final frame

**The Wisdom:** **The Parser should NEVER have app-specific logic.** It reflects state, it doesn't own business logic.

**The Anti-Pattern:**
```c
// WRONG - Parser shouldn't know about pets
if (strcmp(active_id, "pet_01") == 0) {
    hunger = parse_hunger(...);  // Don't do this!
}
```

**The Correct Pattern:**
```c
// RIGHT - Parser just loads variables generically
load_vars_from_state("clock_daemon");  // Any piece, same code
```

---

### 3.10 The "Flicker Bug" Discovery

**The Bug:** Screen flickers or shows garbled output.

**The Cause:** Two processes writing to `current_frame.txt`:
```bash
# WRONG - both renderer AND parser writing
parser writes to current_frame.txt
renderer ALSO writes to current_frame.txt
# Result: flickering, torn frames
```

**The Fix:** Only `chtpm_parser` writes the final frame. The `renderer` only reads and prints.

**The Wisdom:** **One writer, many readers.** The frame pipeline is:
```
Parser (writes) → current_frame.txt → Renderer (reads & prints)
```
Never reverse this flow.

---

### 3.11 The "Joystick Parity" Discovery

**The Pattern:** Joystick codes map to keyboard equivalents:
```
2000-2008 (buttons) → KEY:0-8 (UI actions)
2100-2103 (axes)    → w/a/s/d (movement)
```

**The Wisdom:** **Joystick and keyboard are equivalent input methods.** The Parser normalizes both to the same command stream. The app logic never knows (or cares) which device triggered the action.

**The Anti-Pattern:**
```c
// WRONG - duplicate logic for keyboard vs joystick
if (key == '2') feed_pet();
if (key == 2001) feed_pet();  // Don't do this!
```

**The Correct Pattern:**
```c
// RIGHT - Parser maps joystick to KEY:n, app handles KEY:n
if (key >= '0' && key <= '8') trigger_method(key - '0');
// Works for BOTH keyboard (2-8) and joystick (2001-2007)
```

### 3.12 The "Use-After-Free" Corruption Bug (CRITICAL)

**The Bug:** Freeing a string used for a file path BEFORE calling `fopen()`.
```c
// WRONG - creates files with corrupted names like 'D-]'
free(state_path);
f = fopen(state_path, "w"); // state_path is now garbage memory
```

**The Wisdom:** **Never free a path until the file is closed.** Memory safety is non-negotiable in file-based systems. A corrupted path results in a corrupted filesystem, which is a corrupted Reality in TPM.

### 3.13 The "Stale Root" Path Trap

**The Bug:** Hardcoding `project_root` in `location_kvp` or failing to resolve it dynamically in `orchestrator.c`.

**The Wisdom:** **Paths are relative to the current project instance.** Always generate `location_kvp` using `SCRIPT_DIR` at runtime. Ensure the Orchestrator resolves `project_root` with a safe fallback (like `getcwd()`) to prevent the system from "drifting" into the wrong version's folder.

**The Anti-Pattern:**
```bash
# WRONG - hardcoded path from previous version
project_root=/home/user/project_v13.00
```

**The Correct Pattern:**
```bash
# RIGHT - dynamic resolution
SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
echo "project_root=${SCRIPT_DIR}" > pieces/locations/location_kvp
```
