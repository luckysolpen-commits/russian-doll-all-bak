- Parser injects to: `pieces/apps/player_app/history.txt`

## MISCONCEPTIONS WE HAD

### ❌ "One History File Is Enough"
**Wrong:** Module should read keyboard/history.txt directly like legacy fuzzpet.

**Right:** Parser needs to inject to a SEPARATE file for INTERACT mode, otherwise nav and interact keys mix.

### ❌ "Parser Should Guess Inject Path"
**Wrong:** Parser can guess based on `project_id` or layout name.

**Right:** Layout should EXPLICITLY specify inject path with `<interact>` tag.

### ❌ "Module Can Hardcode Read Path"
**Wrong:** Module hardcodes where it reads.

**Right:** Layout's `<interact>` tag and module's read path must MATCH.

## WHAT WE LEARNED

### 1. FuzzPet Pipeline Is Solid
```
keyboard_input.+x → keyboard/history.txt
                        ↓
                  chtpm_parser.c
                        ↓ (injects in INTERACT mode)
           player_app/history.txt
                        ↓
                fuzzpet_module.+x
                        ↓ (calls ops)
                  state.txt
                        ↓
                render_map.+x
                        ↓
                current_frame.txt
```

**Why it works:**
- Clear separation: keyboard → parser → module → ops → state → render
- Each component has ONE responsibility
- File-based IPC = auditable, debuggable

### 2. TPM Works With Ops
- **TPM:** Everything is a Piece (directory with state files)
- **Ops:** Reusable muscles (move_entity, place_tile, etc.)
- **Together:** Module orchestrates ops, ops update pieces

**Pattern:**
```
Module (brain) → Op (muscle) → Piece State (data)
```

### 3. Keep It Simple
**Don't over-complicate:**
- No PAL scripts for simple movement
- No complex path resolution
- No guessing

**Do:**
- Parser writes to file
- Module reads from SAME file
- Ops update state
- Render displays state

## WHY FUZZPET PIPELINE IS THE TRUTH

### Legacy FuzzPet (SOLID)
```c
// game_logic.c - monolithic but WORKING
while (1) {
    fseek(history, last_pos, SEEK_SET);  // Read keyboard/history.txt
    while (fgets(line, ...)) {
        key = atoi(...);
        process_key(key);  // Direct function call
        update_state();
        render();
    }
}
```

### New Ops-Based (ALSO SOLID)
```c
// fuzzpet_v2_module.c - modular, same pipeline
while (1) {
    fseek(history, last_pos, SEEK_SET);  // Read player_app/history.txt
    while (fgets(line, ...)) {
        key = atoi(...);
        call_op("move_entity", ...);  // Same logic, extracted to op
        update_state();
        render();
    }
}
```

**Same pipeline, just modular!**
