## 1. THE PMO HIERARCHY (UPDATED)

### 1.1 PIECE (The Atomic Unit / Soul)
**Unchanged from TPM v5.0**

```
projects/{id}/pieces/{piece_name}/
├── piece.pdl          # DNA (Truth) - state, methods, responses
├── state.txt          # Mirror (Speed) - flat key=value
├── ledger.txt         # Audit trail
└── plugins/+x/        # Behavior binaries (optional)
```

**NEW:** Pieces can now be instantiated dynamically via `create_piece.+x` op.

---

### 1.2 MODULE (The Logic Agent / Brain)
**EVOLVED:** Modules now come in two flavors:

#### Type A: Monolithic Module (Legacy FuzzPet)
```c
// fuzzpet_manager.c - Does everything
while (1) {
    poll_history();
    route_input();      // Internal routing
    execute_logic();    // Direct function calls
    update_state();     // Direct file writes
    trigger_render();
}
```

#### Type B: Op-Driven Module (New Player/Editor)
```c
// playrm_module.c - Thin orchestrator
while (1) {
    poll_history();
    if (key == 'w')
        system("./ops/+x/move_entity.+x hero up");  // Delegate to op
    if (project == "fuzzpet_v2")
        system("prisc+x scripts/game_loop.asm");    // Delegate to PAL
}
```

**MANDATE:** New modules MUST be Type B (Op-Driven).

---

### 1.3 OS / CHTPM (The Theater / View)
**ENHANCED:** Now supports dynamic layout switching based on project.

```c
// chtpm_parser.c - Layout routing
if (project_id == "fuzzpet_v2")
    current_layout = "fuzzpet_v2.chtpm";
else if (strstr(layout, "editor"))
    current_layout = "editor.chtpm";
else
    current_layout = "playrm/layouts/game.chtpm";
```

### 1.4 PAL (The Script Layer / Nervous System)
**NEW TIER:** Sits between Module and Ops for event-driven behavior.

```
INPUT → MODULE → PAL SCRIPT → OPS → STATE
              (Prisc VM)   (Muscles)
```

**Purpose:**
- Allows users to create complex behavior without C compilation
- Provides conditional logic, loops, and state checks
- Enables "no-code" event editing via visual Op selector

## 2. THE 14-STEP PIPELINE (UPDATED FOR PAL/OPS)

### MODE 1: DIRECT C EXECUTION (Legacy FuzzPet Style)
```
1. INPUT: keyboard_input.+x → pieces/keyboard/history.txt
2. ROUTING: chtpm_parser.c reads key, injects to app history
3. TICK: fuzzpet_manager.c polls local history
4. TRAIT: Manager calls move_entity.c (direct C function)
5. SOVEREIGNTY: Trait updates piece.pdl
6. MIRROR SYNC: Update flushed to state.txt
7. STAGE: map_render.c reads mirrors, writes view.txt
8. SIGNAL: Manager hits frame_changed.txt
9. COMPOSITION: Parser substitutes ${game_map}, ${clock_time}
10. RENDER: Parser writes current_frame.txt
11. DISPLAY: renderer.+x prints to terminal
12. AUDIT: Response handler logs to master_ledger.txt
13. PULSE: Clock daemon ticks (background thread)
14. SYNC: Active GUI index exported for focus tracking
```

### MODE 2: OP-DRIVEN EXECUTION (New Player/Editor)
```
1. INPUT: keyboard_input.+x → pieces/keyboard/history.txt
2. ROUTING: chtpm_parser.c injects to pieces/apps/player_app/history.txt
3. TICK: playrm_module.c polls history
4. DELEGATE: Module calls system("./ops/+x/move_entity.+x ...")
5. OP EXEC: move_entity.+x runs as separate process
6. SOVEREIGNTY: Op reads/writes projects/{id}/pieces/{piece}/state.txt
7. MIRROR SYNC: Op updates state.txt directly
8. SIGNAL: Op hits frame_changed.txt
9. STAGE: render_map.+x reads all piece states
10. COMPOSITION: Parser substitutes ${game_map}
11. RENDER: Parser writes current_frame.txt
12. DISPLAY: renderer prints frame
13. AUDIT: Op logs to master_ledger.txt
14. PULSE: Clock daemon ticks (independent)
```

### MODE 3: PAL SCRIPT EXECUTION (Event-Driven)
```
1. TRIGGER: Player steps on tile / clicks NPC / timer fires
2. MODULE: playrm_module.c detects trigger
3. PAL BOOT: Module launches prisc+x scripts/event.asm
4. ENV SETUP: Module sets PRISC_PROJECT_ID, PRISC_ACTIVE_TARGET
5. SCRIPT: Prisc VM executes PAL instructions
6. OP CALL: PAL script calls ops via `exec` or custom op handlers
7. STATE: Ops update piece state
8. SIGNAL: hit_frame instruction triggers re-render
9. RETURN: Script halts, control returns to module
10. RESUME: Module continues polling for next input
```

## 4. PRISC VM & PAL SCRIPTING (CUTTING EDGE)

### 4.1 PRISC VM ARCHITECTURE

**Location:** `pieces/system/prisc/prisc+x.c`

**Instruction Set:**
| Opcode | Name | Description |
|--------|------|-------------|
| 0x00 | ADDI | Add immediate: `rd = rs1 + imm` |
| 0x01 | BEQ | Branch if equal: `if (rs1 == rs2) goto label` |
| 0x02 | LW | Load word: `rd = mem[rs1 + imm]` |
| 0x03 | SW | Store word: `mem[rs1 + imm] = rs2` |
| 0x04 | JALR | Jump and link register |
| 0x05 | J | Unconditional jump |
| 0x06 | HALT | Stop execution |
| 0x10 | CUSTOM | Custom op call |
| 0x11 | READ_HISTORY | Read from history file |
| 0x12 | EXEC | Execute shell command |
| 0x13 | HIT_FRAME | Trigger frame re-render |
| 0x14 | READ_STATE | Read piece state file |
| 0x15 | READ_ACTIVE_TARGET | Get active target from env |
| 0x16 | READ_ENV_KEY | Get key code from env |

### 4.2 PAL SCRIPT FORMAT

```asm
# game_loop.asm - FuzzPet V2 Main Loop
# Variables (parsed at load time)
int key 0
int menu_result 0
string piece_id "fuzzpet"

# Labels
start:
    # Read keyboard input
    read_history r1
    
    # Branch on input
    beq r1, x0, end          # r1 == 0? skip
    addi r2, r1, -119        # 'w' key
    beq r2, x0, move_up
    addi r2, r1, -115        # 's' key
    beq r2, x0, move_down
    
    # FuzzPet actions (keys 1-4)
    addi r2, r1, -49         # '1' = feed
    beq r2, x0, feed_pet
    addi r2, r1, -50         # '2' = play
    beq r2, x0, play_pet
    
    j end

move_up:
    exec ./ops/+x/move_entity.+x fuzzpet up
    hit_frame
    j end

move_down:
    exec ./ops/+x/move_entity.+x fuzzpet down
    hit_frame
    j end

feed_pet:
    exec ./ops/+x/fuzzpet_action.+x fuzzpet feed
    hit_frame
    j end

play_pet:
    exec ./ops/+x/fuzzpet_action.+x fuzzpet play
    hit_frame
    j end

end:
    halt
```

**Ops File Format:** `default_op.txt` or `player_ops.txt`

```

# FORMAT: op_name type handler {description}

fuzzpet_action op     ./ops/+x/fuzzpet_action.+x {Feed, play, sleep pet}

```

**Prisc Execution Flow:**
```c
// prisc+x.c - exec_custom_op()
void exec_custom_op(Inst *i) {
    CustomOp *op = &custom_ops[op_idx];
    
    // Handler is path to +x binary
    if (strncmp(op->handler, "+x/", 3) == 0) {
        char *cmd = NULL;
        
        // Pass literal arg if present
        if (strlen(i->literal_arg) > 0) {
            asprintf(&cmd, "%s \"%s\"", op->handler, i->literal_arg);
        } else {
            // Pass register value
            asprintf(&cmd, "%s %d", op->handler, regs[i->rs1]);
        }
        
        // Execute and capture output
        FILE *pipe = popen(cmd, "r");
        // ... read output ...
        pclose(pipe);
    }
}
```

### 5.3 THE TENSION POINTS

#### 5.3.1 Path Resolution Chaos

**Three competing systems:**
1. **Environment Variables** (Prisc): `PRISC_PROJECT_ID`, `PRISC_PROJECT_ROOT`
2. **Location KVP** (System): `pieces/locations/location_kvp`
3. **Manager State** (App): `pieces/apps/{app}/manager/state.txt`

**Problem:** Ops don't know which to trust. Editor ops use location_kvp, PAL ops use env vars, player ops use manager state.

**Proposed Resolution:**
```c
// Standardized resolution order (ALL ops must follow)
void resolve_paths_standard() {
    // 1. Environment (if set, use exclusively)
    if (getenv("PRISC_PROJECT_ROOT")) {
        use_env_resolution();
        return;
    }
    
    // 2. Location KVP (system standard)
    if (parse_location_kvp()) {
        return;
    }
    
    // 3. Manager state (app-specific fallback)
    parse_manager_state();
}
```

#### 5.3.2 State Fragmentation

**Current State Locations:**
```
pieces/apps/editor/state.txt          # Editor preferences
pieces/apps/editor/manager/state.txt  # Editor engine state
pieces/apps/player_app/state.txt      # Player preferences
pieces/apps/player_app/manager/state.txt  # Player engine state
projects/{id}/manager/state.txt       # Project-specific state
projects/{id}/pieces/{piece}/state.txt # Piece state
```

**Problem:** Which state file does an op read when called from editor vs player?

**Current Hack:** Ops check `PRISC_PROJECT_ID` env var to determine context.

#### 5.3.3 Undo/Ledger Complexity

**Ledger Format:**
```
PLACE_TILE|map_0001|5|5|.|#
CREATE_PIECE|npc|7|3
DELETE_PIECE|chest|10|8
```

**Problem:** Editor ops log to `projects/{id}/config/editor_history.ledger`, but player ops log to `master_ledger.txt`. Undo only works for editor actions.

**Proposed Unified Ledger:**
```
# projects/{id}/config/action_ledger.txt
TIMESTAMP|CONTEXT|OP|ARGS|PREV_STATE|NEW_STATE
2026-03-11T10:30:00|EDITOR|PLACE_TILE|map_001,5,5,#|.|#
2026-03-11T10:30:05|PLAYER|MOVE_ENTITY|hero,up|5,5|5,4
```

## 7. TECHNICAL PITFALLS (EXPANDED)

### 7.1 THE TRIM TRAP (v5.0)
**Unchanged:** Always `trim()` values from files.

### 7.2 popen OVERHEAD (v5.0)
**Unchanged:** Use direct mirror access for high-frequency reads.

### 7.3 mtime RESOLUTION (v5.0)
**Unchanged:** Use marker-file size detection.

### 7.4 THE DUAL INJECTION BUG (v5.0)
**Unchanged:** Don't inject both numeric and joystick codes.

### 7.5 THE ENV VAR DRIFT (NEW ⚠️)
**Problem:** Ops called from PAL scripts don't inherit environment variables if launched via `system()` without explicit export.

**Wrong:**
```c
// Module sets env var, but child process doesn't see it
setenv("PRISC_PROJECT_ID", "demo_1", 1);
system("./ops/+x/move_entity.+x hero up");  // Env var lost!
```

**Correct:**
```c
// Explicitly pass env vars in command
system("PRISC_PROJECT_ID=demo_1 PRISC_PROJECT_ROOT=/path ./ops/+x/move_entity.+x hero up");
```

### 7.6 THE LEDGER CONTEXT GAP (NEW ⚠️)
**Problem:** Undo ledger doesn't track which context (editor vs player) created the entry.

**Current:**
```
PLACE_TILE|map_001|5|5|.|#
```

**Should Be:**
```
EDITOR|PLACE_TILE|map_001|5|5|.|#
PLAYER|MOVE_ENTITY|hero|5,5|5,4
```

### 7.7 THE SELECTOR VISIBILITY CONFLICT (NEW ⚠️)
**Problem:** Editor renders selector as 'X', but player also has a selector piece. When switching contexts, selector may be invisible or in wrong position.

**Current Hack:** Editor writes to `pieces/apps/editor/pieces/selector/state.txt`, player reads from `projects/{id}/pieces/selector/state.txt`.

**Proposed:** Single selector piece at `projects/{id}/pieces/selector/` with context-aware rendering.

### 7.8 THE PAL BOOTSTRAP PARADOX (NEW ⚠️)
**Problem:** PAL scripts need ops to function, but ops are defined in `default_op.txt` which is parsed by Prisc. Circular dependency if ops file is missing.

**Resolution:** Prisc must have hardcoded fallback ops (`addi`, `beq`, `halt`, `exec`, `hit_frame`).

### 7.9 THE ASYMMETRIC HISTORY TRAP (NEW ⚠️)
**Problem:** The `chtpm_parser` injects keys into one file (e.g., `pieces/apps/player_app/history.txt`) while the Module polls another (e.g., `pieces/apps/editor/history.txt`).
**Symptoms:** Input is captured in `pieces/keyboard/history.txt` but the app doesn't react.
**Resolution:** Standardize on `pieces/apps/player_app/history.txt` for all unified player/editor contexts, or use project-specific history.

### 7.10 THE GHOST RENDERER (NEW ⚠️)
**Problem:** Graphical windows (like `gl_renderer`) may not receive `SIGINT` if the terminal session is closed or `Ctrl+C` is hit.
**Resolution:** The Orchestrator MUST handle `SIGHUP` and `SIGTERM` and call `kill(0, SIGTERM)` to clean up the entire process group.

### 7.11 THE TRUNCATION TERROR (NEW ⚠️)
**Problem:** The project enforces `-Wformat-truncation=1`. Using `snprintf` with fixed buffers for paths often triggers warnings.
**Resolution:** Use `asprintf` for all path constructions. MANDATORY: Define `#define _GNU_SOURCE` before includes and always `free()` the result.

## 8. ARCHITECTURAL DECISIONS (EXPLAINED)

### 8.1 WHY OPS ARE SEPARATE PROCESSES

**Rationale:**
1. **Crash Isolation:** Op crash doesn't bring down module
2. **Language Agnostic:** Ops can be C, Python, Bash, etc.
3. **Audit Trail:** Each op execution is a discrete event
4. **Memory Clean:** No state leakage between calls

**Trade-off:** Process spawn overhead (~5-10ms per op call)

### 8.2 WHY PAL EXISTS

**Rationale:**
1. **User Scripting:** Non-programmers can create game logic
2. **Hot Reload:** Scripts can be edited without recompilation
3. **Event-Driven:** Cleaner than polling for triggers
4. **Composable:** Scripts can call other scripts

**Trade-off:** VM overhead, debugging complexity

### 8.3 WHY PROJECTS USE PDL

**Rationale:**
1. **Human Readable:** Plain text, version-control friendly
2. **Self-Describing:** Metadata embedded with data
3. **Extensible:** New fields don't break parsers
4. **TPM Compliant:** DNA/Mirror pattern

**Trade-off:** Slower than binary formats, parsing complexity

### 8.4 WHY EDITOR AND PLAYER SHARE OPS

**Rationale:**
1. **Code Reuse:** Fix bug once, benefits both
2. **Consistency:** Same collision logic, same tile format
3. **Bootstrap:** Game inherits editor-tested ops
4. **TPM Compliant:** Ops are "muscles" - stateless, reusable

**Trade-off:** Op interface must satisfy both contexts

## 10. DEBUGGING GUIDE

### 10.1 COMMON ISSUES

**Issue:** Op doesn't update state
**Check:**
1. Is `project_root` resolved correctly?
2. Does piece directory exist?
3. Is state.txt writable?
4. Did op hit `frame_changed.txt`?

**Issue:** PAL script doesn't execute
**Check:**
1. Is `prisc+x` compiled?
2. Is ops file (`default_op.txt`) present?
3. Are environment variables set?
4. Does script path exist?

**Issue:** Input is captured but App is frozen
**Check:**
1. Run `ls -l pieces/apps/*/history.txt`. Which one is growing?
2. Check `chtpm_parser.c` `inject_raw_key` logic.
3. Check `module.c` `history_path` resolution. They MUST match.

**Issue:** Editor and player out of sync
**Check:**
1. Are both using same `project_id`?
2. Is selector state in correct location?
3. Is ledger being written to correct file?
4. Is render triggered after op execution?
