## THE REAL ISSUE IDENTIFIED

The problem wasn't just copying code; it was a mismatch between the **Shared Op** expectations and the **Project Structure**. Specifically:
- `move_entity.c` expected `maps/map_01.txt`.
- Modern projects (like `template`) use `maps/map_0001_z0.txt`.
- This caused a silent collision failure, making the selector appear "stuck".

## FINAL FIXES APPLIED

### 1. Robust Map Discovery
Updated `pieces/apps/playrm/ops/src/move_entity.c` to:
- Read `starting_map` from `project.pdl`.
- Sequentially attempt to find `_z0.txt`, `.txt`, and `_zX.txt`.
- This ensures any project map naming convention is supported.

### 2. Standardized Trait Usage
- Deprecated `move_selector.+x` in favor of the more robust and project-aware `move_entity.+x`.
- Refactored `editor_module.c` to call `move_entity selector <dir> <project_id>`.

### 3. Unified Interaction Buffer
- Verified all apps now correctly use `<interact src="pieces/apps/player_app/history.txt" />`.
- This eliminates the race conditions and path confusion caused by project-specific history files.

### 4. Compile Clean Mandate
- Refactored `editor_module.c` and `render_map.c` to use `asprintf` for all path concatenation.
- Result: 0 compilation warnings and robust memory management.
