# failure.txt - Selector Movement & Editor Logic Analysis

## 1. REASON FOR INITIAL FAILURE
The selector did not move initially because `move_entity.c` was hardcoded to look for `maps/map_01.txt`. However, the `template` project uses `maps/map_0001_z0.txt`. When the op failed to find the map, it defaulted to a "collision" state or failed to resolve boundaries correctly, effectively blocking movement.

## 2. FIXES APPLIED
- **Dynamic Map Resolution:** Updated `move_entity.c` to read `starting_map` from `project.pdl` and resolve the correct `.txt` filename.
- **Direction Parity:** Updated `move_entity.c` to support single-character direction keys ('w', 'a', 's', 'd') in addition to descriptive strings ('up', 'down', etc.).
- **History Path Standardization:** Updated `editor_module.c` to poll the shared `pieces/apps/player_app/history.txt` buffer, matching the `<interact>` tag routing.

## 3. PENDING "FINISHING" TASKS
- **Entity Selection:** The editor needs logic to scan for entities under the selector when ENTER is pressed.
- **Active Target Sync:** The `active_target_id` must be updated in `pieces/apps/player_app/manager/state.txt` so the UI and shared ops know which piece is being manipulated.
- **Mode Switching:** Proper ESC/9 handling to return focus to the selector.
