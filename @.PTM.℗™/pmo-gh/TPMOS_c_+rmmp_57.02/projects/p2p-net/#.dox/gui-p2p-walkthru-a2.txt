1. THE VISION: MULTI-AGENT ORCHESTRATION
================================================================================

Our goal is to evolve the P2P-NET app from a simple messaging client into a
"Multi-Agent Orchestration System". This means:

- ONE terminal can control MULTIPLE users/agents.
- Users can switch identities seamlessly within the same GUI.
- Different windows or regions can show state for different subnets/wallets.
- Agents can communicate locally (same terminal) or across the network.

================================================================================
2. CODE WALKTHROUGH (How it works today)
================================================================================

A. THE BRAIN: p2p_manager.c
---------------------------
- Path: projects/p2p-net/manager/p2p_manager.c
- Logic:
  1. Polls `pieces/apps/player_app/history.txt` for key codes.
  2. Map keys (1-6) to P2P operations.
  3. Uses `fork/exec` to run compiled `+x` ops (CPU-safe).
  4. Updates `manager/state.txt` for the GUI.
  5. Triggers frame refresh (via TPM renderer).

B. THE MUSCLES: Ops (+x)
-----------------------
- Path: projects/p2p-net/ops/+x/
- Action:
  1. Read configuration from `pieces/network/subnets/`.
  2. Perform network I/O (UDP/TCP).
  3. Write data to `pieces/network/peers.txt` and `pieces/chat/inbox/`.
  4. Write response to `manager/response.txt`.

C. THE FACE: p2p-net.chtpm
-------------------------
- Path: projects/p2p-net/layouts/p2p-net.chtpm
- Display:
  - Uses `${variables}` that the TPM parser substitutes.
  - Dynamically shows peer counts, unread counts, and message previews.

3. PLAN: GUI P2P FEATURES (Next Steps)
================================================================================

FEATURE 1: Account Switching (Multi-User same CLI)
--------------------------------------------------
- Action: Add `[7] Switch Account` to ACTIONS.
- Implementation: 
  - `p2p_manager.c` will load available wallets from `pieces/chat/inbox/`.
  - Pressing '7' cycles `current_wallet`.
  - **Selector Pattern:** Like `op-ed`'s glyph selector, the GUI will show a 
    visual list of accounts with the active one highlighted (e.g., `[Account1] Account2 Account3`).
  - **Intuitive UX:** The layout will prominently display the active account 
    and provide clear feedback (e.g., "Switched to Account2") to ensure 
    even automated agents can easily navigate and verify identity states.
  - Frame updates to show the NEW wallet's inbox, status, and balance.

FEATURE 2: Real-Time Peer Status
-------------------------------
- Action: Visual indicator of "Connected" vs "Offline" for subnets.
- Implementation:
  - `ping_peer.+x` runs in background (called by manager).
  - Updates `peers.txt` last_seen.
  - Layout shows `[●]` (green) or `[○]` (gray) next to subnets.

FEATURE 3: Rich Message Preview
------------------------------
- Action: Detailed view of the selected message.
- Implementation:
  - `read_message.+x` populates `message_preview` variable.
  - Layout expands the preview section when a message is selected.

4. TESTING WALKTHROUGH (How we'll verify)
================================================================================

SCENARIO 1: Single Terminal Multi-User
--------------------------------------
1. Launch P2P-NET.
2. Select "player1" wallet.
3. Compose message to "player2".
4. Press [7] to Switch Account to "player2".
5. Check Inbox -> See message from "player1".
6. VERDICT: Success if both "users" managed in same CLI instance.

SCENARIO 2: Multi-Terminal Discovery
------------------------------------
1. Open Terminal A (Wallet A, Port 8000).
2. Open Terminal B (Wallet B, Port 8001).
3. Terminal A: Join Subnet "chat".
4. Terminal B: Join Subnet "chat".
5. Terminal A: List Peers -> Should see Terminal B (127.0.0.1:8001).
6. VERDICT: Success if UDP broadcast connects separate instances.
