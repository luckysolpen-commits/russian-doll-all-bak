# Layout-OS-FuzzPet Pipeline: Canonical Architecture Analysis

**Document Type:** Canonical Architecture Specification  
**Date:** 2026-02-23  
**Status:** AUTHORITATIVE - Standard for PMO-Compliant CHTPM Systems  
**Reference:** Original FuzzPet Pipeline (`c+x_tpm_FUZPET+gl_12+PDLON/`)

---

## Executive Summary

The primary cause of visual lag and inconsistent command firing in the current system is a violation of the **Sovereign Piece Principle**. The system was incorrectly using "Lazy Composition" (Parser-led), where the OS Parser attempted to calculate Piece state and substitute variables.

**The Fix:** Transition to the **KISS Pipeline (Eager Composition)**.
1.  **Module** (not Parser) owns the View during active interaction.
2.  **Filesystem** (not Pipes) acts as the primary API.
3.  **Marker-File Signaling** (not mtime) ensures 60 FPS synchronization.

---

## Part 1: The Canonical PMO Pipeline

### 1.1 The "Perfect" Pipeline Flow

```
┌─────────────────────────────────────────────────────────────────┐
│                    THE KISS PIPELINE                            │
└─────────────────────────────────────────────────────────────────┘

1. INPUT CAPTURE:
   keyboard_input.+x writes to pieces/keyboard/history.txt

2. RELAY (OS LAYER):
   chtpm_parser reads keyboard history.
   - Mode == INTERACT: Parser calls inject_raw_key() -> writes to fuzzpet/history.txt
   - Mode == Navigation: Parser handles Focus movement.

3. LOGIC (MODULE LAYER):
   fuzzpet.c module reads fuzzpet/history.txt.
   - Calculates state changes (hunger, position).
   - Writes state to fuzzpet/state.txt (via piece_manager).
   - Logs to pieces/master_ledger/master_ledger.txt.

4. SOVEREIGN VIEW PRODUCTION (MODULE LAYER):
   fuzzpet.c module generates its ASCII stage (Map + Response).
   - Writes directly to pieces/apps/fuzzpet_app/fuzzpet/view.txt.
   - **MANDATE:** The Module DOES NOT write to display/current_frame.txt.

5. COMPOSITE COMPOSITION (OS LAYER):
   chtpm_parser reads fuzzpet/view.txt and substitutes it into ${game_map}.
   - Parser writes the final composite frame to pieces/display/current_frame.txt.
   - Parser appends "X\n" to master_ledger/frame_changed.txt.

6. RENDERING:
   renderer.+x detects size change in frame_changed.txt.
   - Prints pieces/display/current_frame.txt to terminal.
```

## Part 2: The Dual-Mode Interaction Model

### 2.1 Navigation Mode (`active_index == -1`)
*   **Composer:** The Parser.
*   **Input:** w/s/a/d moves the focus indicator `[>]`.
*   **Update:** Parser writes frame, hits marker, Renderer displays UI change.

### 2.2 INTERACT Mode (`onClick="INTERACT"`)
*   **Final Writer:** The Parser.
*   **Content Producer:** The Module.
*   **Input:** All keys are injected into the Module's `history.txt`.
*   **Update:** The Module calculates logic and writes its stage to `view.txt`. The Parser detects the change (via polling mirror state) and composes the final composite frame.
*   **KISS Rule:** Only the Parser writes to `current_frame.txt`. This eliminates flickering and ensures the OS Shell (Theater) is permanent.

---

## Part 3: Why Pipes/VARS are Forbidden (Mistakes Corrected)

Previous draft versions of this document suggested pipe-based IPC (`VARS:message=...`). This is an **anti-pattern** in PMO for the following reasons:

1.  **Audit Failure:** Data passing through a pipe leaves no trace on disk. 
2.  **Synchronization Lag:** Pipes introduce a "ping-pong" delay between Module and Parser.
3.  **KISS Violation:** Pipes require complex multi-process error handling. 

**The PMO Solution:** The Module writes its logic result to `view.txt`. The Parser reads this file during its standard variable-substitution cycle.

## Part 4: Implementation Specification

### 4.1 FuzzPet Module Refactor
-   Implement `write_sovereign_view()`.
-   This function generates the Map + Response and writes to `fuzzpet/view.txt`.
-   **DO NOT** hit the `frame_changed.txt` marker. The Parser handles signaling.

### 4.2 Parser Refactor
-   Map `${game_map}` to read from `fuzzpet/view.txt`.
-   **CHANGE-DRIVEN RENDER:** The Parser polls for state/view changes at 60 FPS.
-   **MARKER-PROTOCOL:** The Parser **ONLY** writes to `current_frame.txt` and appends to `frame_changed.txt` if a change is detected in Piece state or the Module's `view.txt`.
-   Only one writer (`chtpm_parser.c`) means no more flickering.
