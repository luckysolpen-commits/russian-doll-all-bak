# TPM Investigator Document

## 1. Overview of TPM (The Practical Master) FUZPET System

The project referred to as "TPM" appears to be an "Orchestrated-MVC" (Model-View-Controller) system designed for a virtual pet simulation or a simple game, specifically named "FuzzPet System".

**Core Concepts:**
*   **Orchestrated-MVC Architecture:** The system adheres to an Orchestrated-MVC pattern, implying a central orchestrator component manages the flow and interaction between different parts of the application. The primary architectural flow is described as: `Orchestrator -> Keyboard -> Game Logic -> Renderer`.
*   **Modular Design ("Pieces" and "Plugins"):** The system is highly modular, composed of distinct "pieces" and "plugins". Each "piece" (e.g., fuzzpet, keyboard, master_ledger, display, world) represents a specific functional unit within the system.
*   **File-Based Communication:** A fundamental aspect of TPM is its inter-component communication, which is explicitly file-based. Components interact by reading from and writing to designated files (e.g., `pieces/keyboard/history.txt` for input, `save_map.txt` for rendering). This is a critical design constraint.
*   **State Management ("piece.txt"):** Each "piece" has an associated `piece.txt` file that defines its internal `state` (e.g., fuzzpet's hunger, happiness, energy), available `methods` (actions it can perform), `event_listeners` (events it can react to), and `responses` (messages generated after certain actions/events). These `piece.txt` files essentially define the "Model" for each component.
*   **Input and Output:** User input is managed via the `keyboard` piece, which logs key presses to `pieces/keyboard/history.txt`. The `renderer` piece is solely responsible for producing console output to display the game's state.
*   **Master Ledger:** A central `master_ledger` piece is responsible for logging and auditing system events, method calls, and state changes across the entire application, providing an auditable history of operations.
*   **Game/Simulation Context:** The presence of a "FuzzPet", "world" (with a map), "game_logic", and controls for movement and actions strongly indicates a virtual pet simulation where users interact with a digital pet within a defined world.

**Initial Setup & Execution:**
The `run_fuzzpet.sh` script handles the initial setup:
1.  Creates necessary directory structures for `pieces` (fuzzpet, keyboard, master_ledger, display, world).
2.  Initializes default `map.txt` and `piece.txt` files for various components if they don't already exist, populating them with initial states and definitions.
3.  Clears previous session data from `pieces/keyboard/history.txt`, `pieces/master_ledger/master_ledger.txt`, and `pieces/master_ledger/ledger.txt` to ensure a clean start.
4.  Creates an empty `save_map.txt` file, likely used by the renderer.
5.  Finally, it executes the `orchestrator.+x` binary, which presumably kicks off the entire system.

## 4. Interaction and Communication Flow

The interaction and communication within the TPM FUZPET system are fundamentally driven by an **Orchestrated-MVC architecture** and rely heavily on **file-based communication**. Components generally do not directly invoke functions on each other; instead, they communicate by reading and writing to designated files, effectively making the filesystem the central message bus and state repository.

**4.1. Orchestrated Flow (High-Level):**

The `orchestrator` plugin serves as the central control unit, managing the overall application lifecycle and execution flow. The high-level data and control flow is described as:

`Orchestrator -> Keyboard -> Game Logic -> Renderer`

This implies the `orchestrator` initiates and sequences operations among these major functional areas, likely in a continuous loop.

**4.2. File-Based Communication Mechanisms:**

The core of inter-component communication is through shared files located primarily within the `pieces/` directories. This design choice has several implications:

*   **Decoupling:** Components are highly decoupled; they only need to know the path to the relevant file, not the internal workings of other components.
*   **Persistent State:** The file-based nature means that much of the system's state is inherently persistent between executions or across different components' lifecycles.
*   **Implicit Event Handling (Polling):** While `piece.txt` files define `event_listeners`, these are not handled via traditional direct event callbacks. Instead, plugins responsible for reacting to certain events likely **poll** or periodically read specific files (e.g., `history.txt`, `ledger.txt`) to detect new entries or changes, and then process them.

**Specific Communication Channels and Patterns:**

*   **User Input to Game Logic:**
    1.  The `keyboard` plugin captures user keystrokes.
    2.  It logs these keystrokes into `pieces/keyboard/history.txt`.
    3.  The `game_logic` plugin (or a component it controls) periodically reads `pieces/keyboard/history.txt` to retrieve new user commands.
*   **State Updates:**
    1.  When a plugin (e.g., `game_logic`, `fuzzpet`) needs to change the state of a "piece" (e.g., fuzzpet's hunger, world map), it writes these changes to the corresponding `piece.txt` file or a dedicated state file (e.g., `pieces/fuzzpet/piece.txt`, `pieces/world/map.txt`, `pieces/clock/state.txt`).
    2.  The `piece_manager` plugin plays a crucial role here, likely providing standardized functions to load, parse, update, and save the state of any given piece, abstracting the file I/O details.
*   **Rendering State:**
    1.  The `renderer` plugin is responsible for generating the visual output.
    2.  It reads the current state of the game world from files like `pieces/world/map.txt`.
    3.  It might also read from `pieces/display/current_frame.txt` or a common file like `save_map.txt` (mentioned in `run_fuzzpet.sh` and created by it) to get additional rendering instructions or data.
    4.  The final output is then typically printed to the console.
*   **Auditing and Logging:**
    1.  The `master_ledger_monitor` plugin is tasked with observing significant system activities, such as method calls, state changes, or other system events.
    2.  It then uses the `master_ledger` piece's `log_event` method (which internally writes to `pieces/master_ledger/ledger.txt` and `pieces/master_ledger/master_ledger.txt`) to record these events, creating a comprehensive audit trail.
*   **Responses and Feedback:**
    1.  Individual pieces define `responses` in their `piece.txt` files for various actions or events.
    2.  When a method is executed or an event occurs, the system (likely via the `response_handler` plugin) retrieves the appropriate response message from the relevant `piece.txt` and presents it to the user or uses it for internal feedback.

In essence, the file system acts as the central nervous system of TPM, enabling loosely coupled components to interact and maintain a consistent global state.

## 5. Primary Data Structures and State Management Mechanisms

The TPM system's state management is fundamentally file-based, reflecting its component communication strategy. Various files within the `pieces/` directories serve distinct roles in defining and maintaining the system's dynamic and static state. The `piece_manager` plugin is central to handling the loading, parsing, updating, and saving of these data structures, abstracting the underlying file I/O for other plugins.

Here's a breakdown of the primary data structures:

*   **`piece.txt` (Static Configuration and Schema):**
    *   **Location:** Found in each piece's directory (e.g., `pieces/fuzzpet/piece.txt`).
    *   **Purpose:** These files define the static configuration and operational schema for each component ("piece"). They act as blueprints, specifying:
        *   **`state`**: The initial attributes and properties of the piece. This defines *what kind* of data the piece holds.
        *   **`methods`**: The actions or functions that can be performed by or on this piece.
        *   **`event_listeners`**: The types of events this piece is designed to react to.
        *   **`responses`**: Pre-defined messages or outputs associated with certain actions or events.
    *   **Format:** YAML-like key-value structure.
    *   **Role in State Management:** Establishes the structure and initial values, but typically the dynamic, live state is stored in other files (like `state.txt`).

*   **`state.txt` (Dynamic Piece State):**
    *   **Location:** Found in some piece directories (e.g., `pieces/clock/state.txt`).
    *   **Purpose:** Stores the current, mutable, and operational state variables for a specific piece. This reflects the *live* data of the component as the system runs.
    *   **Format:** Typically simple key-value pairs or structured plain text, representing the current values of attributes like `turn`, `time`, or other piece-specific dynamic data.
    *   **Role in State Management:** This is the primary repository for a piece's active state. Plugins read from and write to these files to update the piece's condition as game logic progresses.

*   **`history.txt` (Event/Input Logs):**
    *   **Location:** Found in piece directories where a chronological log of events or inputs is necessary (e.g., `pieces/keyboard/history.txt`, `pieces/fuzzpet/history.txt`).
    *   **Purpose:** Functions as a sequential log of events, actions, or inputs that have occurred.
    *   **Format:** Typically line-delimited entries, where each line represents a distinct event or input.
    *   **Role in State Management:** Essential for event processing. For example, the `game_logic` plugin would read `keyboard/history.txt` to discover and process new user commands.

*   **`ledger.txt` and `master_ledger.txt` (System-Wide Audit Trails):**
    *   **Location:** Primarily within `pieces/master_ledger/` (`pieces/master_ledger/ledger.txt`, `pieces/master_ledger/master_ledger.txt`).
    *   **Purpose:** Serve as a comprehensive, centralized audit log for significant system activities, including method calls, state changes, system events, and errors. `master_ledger.txt` is likely the complete, immutable record, while `ledger.txt` might be a working or temporary log.
    *   **Format:** Line-delimited entries, each describing a logged event with relevant details.
    *   **Role in State Management:** Provides an invaluable historical record for debugging, understanding system evolution, and ensuring data integrity. The `master_ledger_monitor` plugin is responsible for feeding events into these logs.

*   **`map.txt` (World Map Data):**
    *   **Location:** `pieces/world/map.txt`.
    *   **Purpose:** Defines the layout and current content of the game world. This is where the FuzzPet moves and interacts with its environment.
    *   **Format:** Typically a grid-like ASCII representation, where different characters denote terrain, objects, or the FuzzPet's position.
    *   **Role in State Management:** Represents the environmental state of the game. Updates to the world (e.g., FuzzPet movement, object placement) directly modify this file.

*   **`current_frame.txt` (Display Buffer):**
    *   **Location:** `pieces/display/current_frame.txt`.
    *   **Purpose:** Holds the raw textual data that the `renderer` plugin should display to the user as the current frame.
    *   **Format:** Plain text, often representing a screen buffer.
    *   **Role in State Management:** Acts as an output buffer. The `display` piece manages this, and the `renderer` consumes it to update the console output.

*   **`save_map.txt` (Renderer Specific Data/Signal):**
    *   **Location:** Created by `run_fuzzpet.sh` in the main C implementation directory (e.g., `c+x_tpm_FUZPET_11]SHIP/save_map.txt`).
    *   **Purpose:** Potentially acts as a signal file or a temporary data dump specifically for the `renderer` to quickly access map-related information or trigger a re-render cycle. Its contents and exact usage would need further code investigation.
    *   **Format:** Unknown without code, likely simple text.
    *   **Role in State Management:** Secondary, specific to rendering optimization or signaling.

## 7. Guidance for Future Agents

This section provides guidelines for future agents interacting with the TPM FUZPET project, focusing on how to approach bug fixes, feature development, and ensuring code alignment with the project's unique architecture.

### 7.1. General Principles

*   **Embrace File-Based Communication:** The most crucial principle is to understand that almost all inter-component interaction occurs through file I/O. Any bug investigation or new feature design *must* respect and leverage this file-based paradigm. Direct memory sharing or traditional inter-process communication mechanisms are generally not used and should be avoided unless explicitly sanctioned and thoroughly documented.
*   **Modular "Pieces" and "Plugins" Approach:** The system's modularity into "pieces" (data/model) and "plugins" (logic/controller) is fundamental. Understand which piece owns which state and which plugins are responsible for its manipulation.

### 7.2. Approaching Bug Fixes

1.  **Start with the `piece.txt`:** To diagnose a problem with a specific component, always begin by examining its `piece.txt` file. This defines its expected state attributes, methods, and event listeners. Verify if the perceived issue aligns with the piece's defined capabilities.
2.  **Trace Plugin Logic:** Once the target `piece` and its relevant `methods` or `event_listeners` are identified, locate the corresponding plugin (`.c` or `.py` file in the `plugins/` directory). This is where the actual implementation logic resides.
3.  **Identify Communication Files:** Determine which specific files within the `pieces/` directories are involved in the interaction related to the bug. This could be a `state.txt` for dynamic state, a `history.txt` for input/event logs, `map.txt` for world state, or `current_frame.txt` for display output.
4.  **Debugging Strategies:**
    *   **File Monitoring:** The most effective debugging technique will be to actively monitor changes in the relevant files. Tools like `watch`, `tail -f`, or custom scripts that perform `diff` operations on file contents can reveal unexpected state transitions or data corruption.
    *   **`master_ledger` Inspection:** The `master_ledger` serves as a system-wide audit trail. Inspecting `pieces/master_ledger/ledger.txt` or `pieces/master_ledger/master_ledger.txt` can provide a chronological sequence of events, method calls, and state changes, helping to pinpoint the exact moment or interaction where a bug originates.
    *   **Isolate and Test:** For complex issues, try to isolate the problematic piece(s) and their associated plugins. Test them in a controlled environment by manually setting up initial file states and then executing only the relevant plugins. This can help narrow down the source of the bug.

### 7.3. Feature Development and Code Alignment

1.  **Adhere to the File-Based Model:** Any new feature must be designed to integrate seamlessly with the existing file-based communication and state management model. Resist the urge to introduce new communication paradigms (e.g., shared memory, network sockets) unless absolutely critical for performance and explicitly approved, as this would break architectural consistency.
2.  **Update `piece.txt` Definitions:** If a new feature requires changes to a piece's state, adds new methods, or introduces new event listeners, ensure that the corresponding `piece.txt` file is updated to reflect these changes. This maintains the declarative nature of piece definitions.
3.  **Consider Both Implementations:** If the project goal is to maintain functional parity between the C/C++ and Python versions, new features should be developed with both implementations in mind. This means creating analogous plugins (`.c` and `.py`) and ensuring consistent behavior through the file-based interface.
4.  **Leverage `piece_manager`:** For managing the state of pieces (loading from `piece.txt` and `state.txt`, updating, and saving back), always utilize the existing `piece_manager` plugin. Avoid reimplementing file parsing and serialization logic, as this ensures consistency and reduces errors.
5.  **Log to `master_ledger`:** All significant actions, state changes, and any potential errors introduced by new features should be logged to the `master_ledger`. This preserves the auditable history of the system and aids in future debugging and understanding.

### 7.4. Testing Considerations

*   **State-Based Assertions:** Given the file-based nature, testing primarily involves:
    1.  Setting up a known initial state by creating/modifying the relevant `piece.txt`, `state.txt`, `map.txt`, etc., files.
    2.  Executing the relevant plugin(s) or the orchestrator.
    3.  Asserting the final state by examining the contents of expected output files (`state.txt`, `map.txt`, `history.txt`, `ledger.txt`, `current_frame.txt`).
*   **Regression Testing:** Ensure that changes or new features do not inadvertently break existing functionality by running a suite of tests that verify the expected file states after standard interactions.
