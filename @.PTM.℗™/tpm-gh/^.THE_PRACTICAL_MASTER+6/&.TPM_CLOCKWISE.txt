# TPM Clock Piece Implementation Guide

## Overview
This document specifies the design and implementation of a Clock piece for TPM (True Piece Method) systems. The clock piece provides time tracking and turn management functionality, following the TPM principles of file-based communication, auditability, and event-driven architecture.

## Core Purpose
The Clock piece serves to:
1. Track and maintain game time in HH:MM:SS format
2. Maintain turn counter for game progression
3. Advance time based on player actions (movements, turn ends)
4. Provide time-based game mechanics integration
5. Enable other pieces to query current time and turn count
6. Follow TPM principles of passive components and file-based communication

## Piece Architecture

### Clock Piece File (`pieces/clock/piece.txt`)
```
# Piece: clock
state:
  time: 08:00:00
  time_increment_min: 5
  tick_counter: 0
  status: active
methods: tick, reset, pause, resume
event_listeners: turn_end, game_move
responses:
  tick: Time advances by one unit
  reset: Clock reset to initial time
  paused: Clock paused
  resumed: Clock resumed
  turn_end: Your turn has ended. Time advances.
  default: Clock updated successfully
```

### Required State Variables:
- `time`: Current time in HH:MM:SS format (string)
- `time_increment_min`: Minutes to advance per tick (integer, default: 5)
- `tick_counter`: Total number of time advances (integer)
- `status`: Operational status (string: active/paused)

### Required Methods:
- `tick`: Advances time by configured increment
- `reset`: Resets time to default value
- `pause`: Pauses time progression
- `resume`: Resumes time progression

### Required Event Listeners:
- `turn_end`: Triggers time advancement when turn ends
- `game_move`: Triggers time advancement when moves occur

### Response Messages:
- `tick`: "Time advances by one unit" or custom message
- `reset`: "Clock reset to initial time"
- `paused`: "Clock paused"
- `resumed`: "Clock resumed" 
- `turn_end`: "Your turn has ended. Time advances."
- `default`: "Clock updated successfully"

## Event-Driven Architecture

### Movement-Based Time Advancement
1. Player performs movement (w, a, s, d)
2. Game logic processes movement command
3. Game logic calls clock increment utilities directly
4. Time advances by configured increment
5. Updated time is available for display

### Turn-End Time Advancement
1. Player presses end-turn command ('6')
2. Game logic processes end-turn command
3. Game logic calls combined turn/time advancement utility
4. Both turn and time advance by configured amounts
5. Updated values are available for display

## Implementation Requirements

### File-Based Operations (TPM Compliant)
- All state changes persist to files immediately
- No in-memory state shortcuts
- Complete audit trail in master ledger
- Atomic file operations to prevent corruption

### No Continuous Polling
- Clock piece does NOT run continuous monitoring loops
- Clock responds ONLY to direct system calls from other pieces
- Event-driven architecture, not polling-based

### Cross-Language Compatibility
- Implementation patterns should work for both C and Python
- Standardized file formats and access patterns
- Consistent command-line interfaces for utilities

## Required Utilities

### 1. increment_time.py/c
**Purpose:** Advance time only (for movements)
**Interface:** `python3 plugins/increment_time.py` or `./+x/increment_time.+x`
**Effect:** Time advances by `time_increment_min` while turn stays same

### 2. increment_turn.py/c  
**Purpose:** Advance turn only (if needed separately)
**Interface:** `python3 plugins/increment_turn.py` or `./+x/increment_turn.+x`
**Effect:** Turn advances by 1 while time stays same

### 3. end_turn.py/c
**Purpose:** Advance both turn and time (for '6' command)
**Interface:** `python3 plugins/end_turn.py` or `./+x/end_turn.+x`
**Effect:** Both turn and time advance

## Architectural Principles
- **Passive Components:** Clock only responds to direct calls, no autonomous behavior
- **Event-Driven:** Time advances via events, not continuous processes
- **File-Based:** All communication through files for auditability
- **State Persistence:** All values preserved between sessions
- **Modular Design:** Separate utilities for different time operations
- **Auditable:** Complete log of all time-related events

This design ensures the clock functionality follows true TPM principles while providing the necessary game mechanics for time-based gameplay.
