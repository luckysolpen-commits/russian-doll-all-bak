# FuzzPet Codebase Pipeline Flow (Updated)

This document outlines the execution pipeline of the "tpm fuzzpet" codebase, detailing how keyboard input is processed, game logic is applied, and the display is rendered. The system is designed around a "Piece" architecture, where independent components (executables) communicate primarily through a file-based system using `.txt` files in the `pieces/` directory for state management and event logging.

## 1. System Initialization and Orchestration

The entire system is launched and managed by shell scripts and a central orchestrator.

*   **`run_fuzzpet.sh`**: This is the main entry point for ASCII terminal mode.
    *   It sets up the environment by creating necessary directories (`pieces/{fuzzpet,keyboard,master_ledger,display,world,clock}`).
    *   It initializes default state files for `pieces/world/map.txt` and various `pieces/<component>/piece.txt` files (e.g., for `fuzzpet`, `keyboard`, `master_ledger`, `display`, `world`, `clock`).
    *   It clears previous session data from `pieces/keyboard/history.txt` and `pieces/master_ledger/master_ledger.txt`.
    *   Finally, it executes the `+x/orchestrator.+x` binary, which takes over the management of the core game components.

*   **`run_gl_fuzzpet.sh`**: Entry point for GL graphical mode.
    *   Sets up the same environment as `run_fuzzpet.sh`.
    *   Executes `+x/orchestrator.+x` which launches all components including the GL renderer.
    *   Opens a GL window for graphical display alongside terminal output.

*   **`orchestrator.+x` (compiled from `plugins/orchestrator.c`)**:
    *   This component acts as a lightweight process manager.
    *   It launches the main game components as separate concurrent threads (implemented using `pthread_create` and `system()` calls to execute binaries):
        *   `+x/keyboard_input.+x` (terminal keyboard input)
        *   `+x/game_logic.+x` (game processing)
        *   `+x/renderer.+x` (ASCII terminal display)
        *   `+x/response_handler.+x` (event response feedback)
        *   `+x/gl_renderer.+x` (GL graphical display)
    *   It adheres to a "TPM +X" principle by having "no business logic" itself, focusing solely on orchestrating the execution of other components.

## 2. Keyboard Input Capture

User input is captured and logged through TWO components (both write to the same file):

*   **`keyboard_input.+x` (compiled from `plugins/keyboard_input.c`)** - Terminal Input:
    *   Sets the terminal into "raw mode" to capture individual key presses without requiring an Enter key.
    *   Continuously reads key presses from standard input (`STDIN_FILENO`).
    *   For each key press:
        *   It appends a timestamped entry (e.g., `[2026-02-17 12:34:56] KEY_PRESSED: 1000`) to `pieces/keyboard/history.txt`. This file acts as an event queue for keyboard input.
        *   It also logs the input event to `pieces/master_ledger/master_ledger.txt`, serving as a comprehensive audit trail for all system events.
        *   It provides immediate visual feedback by echoing the pressed key to the console where it runs.

*   **`gl_renderer.+x` (compiled from `plugins/gl_renderer.c`)** - GL Window Input:
    *   Uses GLUT keyboard callbacks (`glutKeyboardFunc`, `glutSpecialFunc`) to capture key presses in the GL window.
    *   For each key press:
        *   It appends a timestamped entry to `pieces/keyboard/history.txt` (SAME FILE as keyboard_input).
        *   It logs the input event to `pieces/master_ledger/master_ledger.txt` with source identifier `gl_keyboard_input`.
    *   This allows the game to be controlled from EITHER the terminal OR the GL window (or both simultaneously).

## 3. Game Logic Processing

The core game rules, state updates, and interactions are handled by the `game_logic` component.

*   **`game_logic.+x` (compiled from `plugins/game_logic.c`)**:
    *   **Initialization**: On startup, it loads the game world map from `pieces/world/map.txt` (creating a default one if it doesn't exist) and initializes the player's position.
    *   **Input Consumption**: It continuously monitors `pieces/keyboard/history.txt`. It reads new `KEY_PRESSED` entries, processing each command sequentially.
    *   **Command Processing (`process_key_command`)**:
        *   **Movement (w, a, s, d, arrow keys)**: Updates the player's coordinates (`player_x`, `player_y`) on an in-memory representation of the map. It checks for collisions with walls (`#`) and interactions with treasures (`T`). If the map changes (e.g., player moves), it calls `save_map_to_file("map.txt")` to persist the updated `pieces/world/map.txt`.
        *   **Pet Actions (1=feed, 2=play, 3=sleep, 4=status, 5=evolve)**: These commands trigger specific effects on the FuzzPet's attributes (hunger, happiness, energy, level). These attribute changes are managed by executing `system("./+x/piece_manager.+x fuzzpet set-state <attribute> <value>")` commands.
        *   **Time & Turn Advancement (6=end_turn, or movement)**:
            *   Pressing '6' executes `system("./+x/end_turn.+x")`.
            *   Any player movement also executes `system("./+x/increment_time.+x")` and `system("./+x/increment_turn.+x")` independently.
            *   These executables (covered below) modify the `turn` and `time` entries in `pieces/clock/state.txt`.
        *   **Master Ledger Logging**: Every significant action and state change (movement, pet actions, turn end) is logged to `pieces/master_ledger/master_ledger.txt` as `EventFire` entries. Crucially, it also logs `ResponseRequest` entries for pet actions, indicating a request for user feedback.
    *   **State Interaction via `piece_manager.+x`**: `game_logic.+x` extensively uses `system("./+x/piece_manager.+x ...")` calls to:
        *   `get-state <attribute>`: Retrieve current FuzzPet stats (e.g., hunger, happiness) or other piece states.
        *   `set-state <attribute> <value>`: Update FuzzPet stats.
    *   **Dashboard Preparation**: It constructs a comprehensive game dashboard by combining:
        *   Current FuzzPet stats (fetched via `piece_manager.+x`).
        *   Current turn and time (read from `pieces/clock/state.txt`).
        *   The current game map.
        *   Any `last_command_output` messages.
    *   **Display Output**: This prepared dashboard is written to `pieces/display/current_frame.txt`.
    *   **Frame Change Signaling**: After writing the frame, it appends a marker to `pieces/master_ledger/frame_changed.txt`:
        ```c
        FILE *marker = fopen("pieces/master_ledger/frame_changed.txt", "a");
        fprintf(marker, "X\n");
        fclose(marker);
        ```
        This append-only marker file allows renderers to detect frame changes reliably (by checking file SIZE, not mtime).

## 4. Clock Management

The game's internal clock (time and turn) is managed by dedicated utilities.

*   **`pieces/clock/state.txt`**: This file stores the current `turn` and `time` of the game (e.g., `turn 5`, `time 09:30:00`).
*   **`increment_time.+x` (compiled from `plugins/increment_time.c`)**: Reads `pieces/clock/state.txt`, increments the game time by 5 minutes, handles time overflows (hours, days), and writes the updated time back to the `state.txt` file.
*   **`increment_turn.+x` (compiled from `plugins/increment_turn.c`)**: Reads `pieces/clock/state.txt`, increments the game turn by 1, and writes the updated turn back to the `state.txt` file.
*   **`end_turn.+x` (compiled from `plugins/end_turn.c`)**: This utility combines the logic of `increment_time.+x` and `increment_turn.+x`, providing a single command to advance both the game time (by 5 minutes) and the turn count (by 1). It directly reads and writes to `pieces/clock/state.txt`.

## 5. Piece State Management Utility

The `piece_manager` acts as the standardized interface for interacting with the state of individual "pieces."

*   **`piece_manager.+x` (compiled from `plugins/piece_manager.c`)**:
    *   This is a command-line utility designed to be called by other components (like `game_logic.+x` and `response_handler.+x`).
    *   It parses `pieces/<piece_id>/piece.txt` files to extract state variables, responses, methods, and event listeners.
    *   **Key Actions**:
        *   `get-state <key>`: Retrieves the value of a specific state variable from a piece's `piece.txt` file.
        *   `set-state <key> <value>`: Updates a state variable in a piece's `piece.txt` file. After modification, it rewrites the entire `piece.txt` file to ensure persistence and logs the `StateChange` to `pieces/master_ledger/master_ledger.txt`.
        *   `get-response <event_type>`: Retrieves a predefined response string for a given event type from a piece's `piece.txt` file (e.g., "fed" -> "Yum yum~ Feeling so happy!").
    *   This centralized management of piece files ensures consistency and modularity in state handling.

## 6. Response Handling

User feedback based on game events is provided by the `response_handler`.

*   **`response_handler.+x` (compiled from `plugins/response_handler.c`)**:
    *   Continuously monitors `pieces/master_ledger/master_ledger.txt` for new entries.
    *   Specifically looks for `ResponseRequest` entries logged by `game_logic.+x`.
    *   Upon detecting a `ResponseRequest`, it parses the event type and piece ID.
    *   It then calls `system("./+x/piece_manager.+x <piece_id> get-response <event_type>")` to fetch the appropriate response message.
    *   The retrieved response message is printed directly to the main console, providing textual feedback to the user.
    *   It also logs a `ResponseTriggered` event to `pieces/master_ledger/master_ledger.txt`, documenting that the response was delivered.

## 7. Display Rendering (Dual Renderer Architecture)

The final visual output is handled by TWO renderer components that work in parallel:

### ASCII Terminal Renderer

*   **`renderer.+x` (compiled from `plugins/renderer.c`)**:
    *   Continuously monitors the SIZE of `pieces/master_ledger/frame_changed.txt` (append-only marker file).
    *   Runs at 60 FPS (checks every 16.667ms using `usleep(16667)`).
    *   When the marker file size changes (indicating a new frame):
        *   Reads the entire content of `pieces/display/current_frame.txt`.
        *   Prints it to standard output (`stdout`), displaying the game's current state.
    *   Uses file SIZE for change detection instead of mtime to avoid 1-second resolution issues.

### GL Graphical Renderer

*   **`gl_renderer.+x` (compiled from `plugins/gl_renderer.c`)**:
    *   Opens an OpenGL window (800x1200 pixels).
    *   Continuously monitors the SIZE of `pieces/master_ledger/frame_changed.txt` (SAME marker file as ASCII renderer).
    *   Runs at 60 FPS (timer callback every 16ms using `glutTimerFunc(16, ...)`).
    *   When the marker file size changes:
        *   Reads the entire content of `pieces/display/current_frame.txt` (SAME file as ASCII renderer).
        *   Parses the frame content line by line.
        *   Renders each line using GLUT bitmap font (`GLUT_BITMAP_HELVETICA_18`).
    *   Also handles keyboard input via GLUT callbacks, writing to `pieces/keyboard/history.txt`.
    *   Uses the same frame file and marker detection as ASCII renderer for synchronized output.

## Pipeline Summary

The FuzzPet system operates as a **file-driven, event-based pipeline** with dual rendering:

1.  **Input**: 
    *   `keyboard_input.+x` captures terminal keyboard input.
    *   `gl_renderer.+x` captures GL window keyboard input.
    *   Both write to `pieces/keyboard/history.txt` and `pieces/master_ledger/master_ledger.txt`.

2.  **Processing**:
    *   `game_logic.+x` reads input from `pieces/keyboard/history.txt`.
    *   It applies game rules, updates internal state (map), and manipulates piece states by interacting with `piece_manager.+x`.
    *   It updates the game clock via `increment_time.+x`, `increment_turn.+x`, or `end_turn.+x` (modifying `pieces/clock/state.txt`).
    *   All significant events and state changes are logged to `pieces/master_ledger/master_ledger.txt`.
    *   The complete visual state is written to `pieces/display/current_frame.txt`.
    *   A marker "X\n" is appended to `pieces/master_ledger/frame_changed.txt` to signal frame update.

3.  **Output & Feedback**:
    *   `renderer.+x` monitors `frame_changed.txt` size and prints `current_frame.txt` to terminal.
    *   `gl_renderer.+x` monitors `frame_changed.txt` size and renders `current_frame.txt` in GL window.
    *   `response_handler.+x` monitors `master_ledger.txt` for `ResponseRequest` entries and prints responses.

## Key Enhancements (vs Original Design)

### Frame Change Detection
- **Original**: Used file modification time (mtime) with 1-second resolution.
- **Enhanced**: Uses append-only marker file size for reliable change detection.
- **Benefit**: Detects every frame update, even multiple updates within the same second.

### Dual Input Sources
- **Original**: Only terminal keyboard input.
- **Enhanced**: Both terminal AND GL window keyboard input write to the same history file.
- **Benefit**: User can control the game from either interface simultaneously.

### Dual Rendering
- **Original**: Only ASCII terminal renderer.
- **Enhanced**: Both ASCII terminal AND GL graphical renderer work in parallel.
- **Benefit**: Same game state displayed in both interfaces, synchronized via shared marker file.

### 60 FPS Update Rate
- **Original**: 10 FPS (100ms check interval).
- **Enhanced**: 60 FPS (16-17ms check interval) for both renderers.
- **Benefit**: Smoother, more responsive visual updates.

### Shared Frame File
- **Original**: Separate frame files for different renderers.
- **Enhanced**: Both renderers read from `pieces/display/current_frame.txt`.
- **Benefit**: Guaranteed synchronized output, no divergence between displays.

---
**To make modifications or fix things:**

*   **Identify the relevant component**: Determine which part of the pipeline (input, logic, display, clock, piece management, response) is responsible for the functionality you want to change.
*   **Locate the source file**: Find the corresponding `.c` file in the `plugins/` directory.
*   **Understand file interactions**: Pay attention to which `.txt` files in `pieces/` the component reads from and writes to, and which `+x/` executables it calls via `system()`.
*   **Key files for rendering**:
    *   `pieces/display/current_frame.txt` - Frame content (read by both renderers)
    *   `pieces/master_ledger/frame_changed.txt` - Frame change marker (append-only, checked by both renderers)
    *   `pieces/keyboard/history.txt` - Keyboard input (written by both input sources)
*   **Modify code**: Make your changes in the `.c` file.
*   **Recompile**: Recompile the modified `.c` file into its `+x` executable using `./compile_all.sh`.
*   **Test**: Run `./run_fuzzpet.sh` (ASCII) or `./run_gl_fuzzpet.sh` (GL) to test your changes.
*   **Debugging**: Examine the various `.txt` files in `pieces/` to trace data flow. The `master_ledger.txt` is particularly useful for auditing all events. The `frame_changed.txt` size should grow with every frame update.
