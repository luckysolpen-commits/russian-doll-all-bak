Piece Definition Language OpenSource (PDLO)
Refactoring Guide + Architecture Specification

1. What Is PDLO?

PDLO is a formal interface contract layer for TPM.

TPM defines ontology:

Everything is a Piece.

PDLO defines interface:

A Piece may formally declare its contract in a deterministic, plaintext, spreadsheet-readable format.

PDLO:

Does not replace piece.txt

Does not replace the master ledger

Does not introduce JSON

Does not introduce a privileged runtime

Does not violate file sovereignty

Is optional

Is itself a Piece

PDLO is a contract declaration, not an execution engine.

7. Refactoring Strategy (Step-by-Step)
Phase 1 — Passive Declaration

Create pdlo.txt inside each piece.

Mirror existing:

state keys

methods

event listeners

emitted events

Do NOT enforce anything yet.

No plugin changes required.

Goal:
PDLO exists as documentation.

Phase 2 — Validation Layer

Create:

pieces/pdlo_validator/
    piece.txt
    pdlo_validator.+x


Validator responsibilities:

Parse pdlo.txt

Parse piece.txt

Compare:

state keys match

methods declared

event_listeners declared

Log violations to master ledger

NO mutation.
Audit only.

This preserves TPM purity.

Phase 3 — Plugin Alignment

Now adjust plugins to:

Never mutate undeclared state keys.

Never emit undeclared events.

Never expose undeclared methods.

Add runtime check:

Before method execution:

Verify method declared in PDLO.

If not, log violation and abort.

This prevents rogue behavior.

Phase 4 — Determinism Enforcement (Optional)

If determinism=strict:

Plugin must:

Log all randomness inputs.

Log external system calls.

Log clock reads.

Replay must match.

If determinism=open:

No enforcement.

9. Backward Compatibility

If pdlo.txt missing:

System behaves exactly as before.

If present:

Validator may log warnings but not break execution.

Strict enforcement mode optional.

Backward compatibility principle:

PDLO never blocks legacy execution unless explicitly enabled.

10. Versioning Strategy

META section must contain:

META | version | 1.0


When updating:

Add minor version for backward-compatible changes.

Increment major version for breaking schema changes.

Validator may check:

Required state fields missing.

Deprecated methods called.
