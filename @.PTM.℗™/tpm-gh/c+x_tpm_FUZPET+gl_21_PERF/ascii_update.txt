# ASCII CLI Renderer Update Mechanism

## How ASCII Renderer Knows When to Update

The ASCII renderer (`plugins/renderer.c`) uses **file modification time (mtime)** to detect changes.

### Code Flow

```c
// Global: Track last known modification time
time_t last_current_frame_time = 0;

// Main loop - runs forever
while (1) {
    // Get current modification time of the frame file
    current_time = get_file_mod_time("pieces/display/current_frame.txt");
    
    // Compare with last known time
    if (current_time != last_current_frame_time) {
        // File changed! Render new frame
        render_display();
        last_current_frame_time = current_time;
    }
    
    usleep(100000);  // Wait 100ms before next check
}
```

### Key Points

1. **Check Interval**: Every 100ms (10 times per second)

2. **Detection Method**: `stat().st_mtime` - filesystem modification timestamp

3. **Update Trigger**: mtime value changes

4. **Render Action**: Prints frame to terminal stdout (appends to scrollback)

### Why This Works for ASCII

- Terminal output is **append-only** - each render adds to scrollback
- No "blanking" issue because old frames stay in terminal history
- User sees a history of all frames as a log

### The mtime Resolution Problem

File modification time has ~1 second resolution on many filesystems.

**Scenario where this fails**:
```
T=0.0s: game_logic writes frame, mtime=1000
T=0.1s: ASCII renderer checks, mtime=1000, NO UPDATE
T=0.2s: ASCII renderer checks, mtime=1000, NO UPDATE
...
T=0.9s: ASCII renderer checks, mtime=1000, NO UPDATE
T=1.0s: Next key press, mtime=1001, UPDATE (shows both changes)
```

**Why ASCII still looks responsive**:
- Even with missed updates, the terminal scrollback shows ALL rendered frames
- When an update finally happens, you see the cumulative change
- The "--- FRAME UPDATE ---" header makes it look like continuous activity

## GL Renderer vs ASCII Renderer

| Aspect | ASCII Renderer | GL Renderer (Original) | GL Renderer (Fixed) |
|--------|---------------|------------------------|---------------------|
| Detection | mtime | mtime | frame_changed.txt size |
| Check Interval | 100ms | 100ms | 41ms (~24fps) |
| Render Target | Terminal stdout | OpenGL window | OpenGL window |
| Old Frames | Preserved in scrollback | Overwritten | Overwritten |
| Blanking Issue | N/A (scrollback) | Yes | Fixed |

## Why GL Renderer Had Issues

### Problem 1: mtime Resolution
Same as ASCII - misses updates within same second.

### Problem 2: No Scrollback
GL window shows only current frame. If update is missed, screen appears blank/stuck.

### Problem 3: Same Content Size
When player moves, frame file has same size (just different character positions). Size-based detection wouldn't help.

## The Solution: frame_changed.txt

Instead of trying to detect changes in the frame file itself, `game_logic` now appends to a marker file:

```c
// In game_logic.c, after gl_render_print():
FILE *marker = fopen("pieces/gl_display/frame_changed.txt", "a");
if (marker) {
    fprintf(marker, "[%s] Frame updated\n", timestamp);
    fclose(marker);
}
```

**GL renderer checks**:
```c
// Check marker file SIZE (always grows, never shrinks)
struct stat st;
if (stat("pieces/gl_display/frame_changed.txt", &st) == 0) {
    if (st.st_size != last_marker_size) {
        parse_frame_file();  // Read new frame
        last_marker_size = st.st_size;
    }
}
glutPostRedisplay();  // Always redraw
```

### Why This Works Better

1. **Append-only file**: Size always increases, never stays same
2. **No mtime issues**: We check size, not timestamp
3. **Every frame logged**: Even if content is identical, marker grows
4. **24fps polling**: Faster than ASCII's 10fps

## Summary

**ASCII Renderer**:
- Uses mtime of `pieces/display/current_frame.txt`
- 100ms check interval
- Terminal scrollback preserves history
- Looks continuous due to "--- FRAME UPDATE ---" headers

**GL Renderer (Fixed)**:
- Uses size of `pieces/gl_display/frame_changed.txt`
- 41ms check interval (~24fps)
- Keeps last valid frame in memory
- Marker file appended by `game_logic` on every frame update

The key insight: **Don't detect changes in the frame file - have the writer explicitly signal changes**.
