# GL Renderer vs ASCII Renderer - Comparison

## File Comparison: `gl_renderer.c` vs `renderer.c`

### ASCII Renderer (`renderer.c`)

```c
// Global variables
time_t last_current_frame_time = 0;
off_t last_marker_size = 0;

// Main loop (runs forever in while(1))
while (1) {
    // Check marker file SIZE for reliable change detection
    if (stat("pieces/master_ledger/frame_changed.txt", &st) == 0) {
        if (st.st_size != last_marker_size) {
            render_display();
            last_marker_size = st.st_size;
        }
    }
    usleep(16667); // ~60 FPS (16.667ms)
}
```

**Key characteristics:**
- Runs in a **tight while(1) loop**
- Checks marker file **every 16.667ms**
- Calls `render_display()` **immediately** when change detected
- Uses `usleep()` for timing

### GL Renderer (`gl_renderer.c`)

```c
// Global variables
time_t last_frame_time = 0;  // UNUSED!
off_t last_marker_size = 0;

// Timer callback (called by GLUT)
void timer(int value) {
    struct stat st;
    off_t current_size = 0;
    if (stat("pieces/master_ledger/frame_changed.txt", &st) == 0) {
        current_size = st.st_size;
        if (st.st_size != last_marker_size) {
            parse_frame_file();
            last_marker_size = st.st_size;
        }
    } else {
        fprintf(stderr, "[GL DEBUG] Cannot stat marker file\n");
    }
    glutPostRedisplay();
    glutTimerFunc(16, timer, 0);
}

// In main():
glutTimerFunc(100, timer, 0);  // <-- FIRST TIMER CALL AFTER 100ms!
```

**Key characteristics:**
- Runs in **GLUT event loop** (glutMainLoop)
- Timer is **callback-based**, not a tight loop
- First timer call happens after **100ms** (then every 16ms)
- Uses `glutTimerFunc()` for timing
- `display()` callback renders via `glutPostRedisplay()`

## Critical Differences

### 1. Initial Timer Delay

**ASCII:**
```c
last_current_frame_time = get_file_mod_time(...);  // Immediate
// Then immediately enters while(1) loop
```

**GL:**
```c
glutTimerFunc(100, timer, 0);  // First call after 100ms!
```

**Impact:** GL renderer waits 100ms before first check, then 16ms thereafter.
ASCII starts checking immediately.

### 2. Loop Timing

**ASCII:**
```c
while (1) {
    // Check immediately
    // Then sleep 16.667ms
}
```
- **Deterministic:** Checks happen at T=0, T=16.667ms, T=33.334ms, etc.

**GL:**
```c
void timer(int value) {
    // Check
    glutTimerFunc(16, timer, 0);  // Schedule NEXT call
}
```
- **Non-deterministic:** GLUT schedules callback "after at least 16ms"
- GLUT may batch/delay callbacks based on event queue
- Timer resolution depends on GLUT implementation

### 3. Render Trigger

**ASCII:**
```c
if (st.st_size != last_marker_size) {
    render_display();  // Direct function call - immediate
    last_marker_size = st.st_size;
}
```

**GL:**
```c
if (st.st_size != last_marker_size) {
    parse_frame_file();  // Parse now
    last_marker_size = st.st_size;
}
glutPostRedisplay();  // Request render (happens later)
```

**Impact:** ASCII renders immediately in the same iteration.
GL separates parsing (now) from rendering (later via callback).

### 4. Unused Variable

**GL has:**
```c
time_t last_frame_time = 0;  // NEVER USED!
```

This was left over from the old mtime-based detection. Should be removed.

### 5. File Being Read

**ASCII:**
```c
render_display() {
    FILE *frame_file = fopen("pieces/display/current_frame.txt", "r");
    // ...
}
```

**GL:**
```c
parse_frame_file() {
    FILE *fp = fopen("pieces/gl_display/current_frame.txt", "r");
    // ...
}
```

**Impact:** They read from DIFFERENT frame files!
- ASCII reads `pieces/display/current_frame.txt`
- GL reads `pieces/gl_display/current_frame.txt`

Both are written by `game_logic.c`, but if there's any timing difference in writes, one might get stale data.

### 6. Event Loop Interference

**ASCII:**
- Owns the thread completely
- No other callbacks
- Pure polling loop

**GL:**
- Shares thread with GLUT event processing
- Keyboard callbacks, reshape callbacks, display callbacks
- Timer callback competes with other GLUT events
- `glutPostRedisplay()` doesn't render immediately - schedules for later

## Why GL Skips Frames

### Root Cause #1: Timer Callback Scheduling

GLUT's `glutTimerFunc(16, ...)` means "call this function after AT LEAST 16ms".
It doesn't guarantee exactly 16ms.

If GLUT is busy processing other events (keyboard, reshape, etc.), the timer callback may be delayed.

**Scenario:**
```
T=0ms:   Timer callback #1 - check marker, no change
T=16ms:  Timer callback #2 scheduled
T=20ms:  Timer callback #2 actually runs (4ms delay)
T=36ms:  Timer callback #3 scheduled
T=42ms:  Timer callback #3 actually runs (6ms delay)
...
```

Meanwhile, ASCII checks happen at exact 16.667ms intervals.

### Root Cause #2: Separate Parse and Render

GL separates parsing from rendering:
```c
timer() {
    if (marker changed) {
        parse_frame_file();  // Parse NOW
    }
    glutPostRedisplay();  // Render LATER
}

display() {
    render_frame();  // Uses whatever frame_text has
}
```

If marker changes rapidly:
```
T=0ms:  Timer: marker=100, parse, glutPostRedisplay()
T=16ms: Timer: marker=101, parse (overwrites!), glutPostRedisplay()
T=20ms: Display callback runs - shows frame from marker=101
        (frame from marker=100 was never displayed!)
```

ASCII doesn't have this issue because parse+render happen atomically:
```c
if (marker changed) {
    render_display();  // Parse AND render immediately
}
```

### Root Cause #3: Different Frame Files

If `game_logic.c` writes to `display/current_frame.txt` and `gl_display/current_frame.txt` at slightly different times, one renderer might see stale data.

## Fixes

### Fix 1: Remove Unused Variable
```c
// REMOVE THIS:
time_t last_frame_time = 0;
```

### Fix 2: Reduce Initial Timer Delay
```c
// In main():
glutTimerFunc(16, timer, 0);  // Was 100
```

### Fix 3: Use Same Frame File
Either:
- Have GL read from `pieces/display/current_frame.txt` (same as ASCII)
- Or ensure `game_logic.c` writes both files atomically

### Fix 4: Faster Timer Interval
```c
glutTimerFunc(8, timer, 0);  // ~120 FPS instead of 60
```

Compensates for GLUT scheduling delays.

### Fix 5: Atomic Parse+Render (Optional)
```c
void timer(int value) {
    struct stat st;
    if (stat("pieces/master_ledger/frame_changed.txt", &st) == 0) {
        if (st.st_size != last_marker_size) {
            parse_frame_file();
            last_marker_size = st.st_size;
            render_frame();  // Render immediately too
        }
    }
    glutPostRedisplay();
    glutTimerFunc(16, timer, 0);
}
```

## Summary Table

| Aspect | ASCII Renderer | GL Renderer | Impact |
|--------|---------------|-------------|--------|
| Loop type | while(1) polling | GLUT timer callback | GL has scheduling overhead |
| First check | Immediate | After 100ms | GL misses first ~6 frames |
| Timer interval | 16.667ms (usleep) | 16ms (glutTimerFunc) | Similar, but GLUT less precise |
| Parse+Render | Atomic (same function) | Separate (timer + display callback) | GL can skip frames |
| Frame file | `display/current_frame.txt` | `gl_display/current_frame.txt` | Potential sync issues |
| Unused vars | None | `last_frame_time` | Code smell |

## Recommended Fix Order

1. **Change initial timer from 100ms to 16ms** (immediate impact)
2. **Remove unused `last_frame_time` variable** (cleanup)
3. **Have GL read from same file as ASCII** (`pieces/display/current_frame.txt`)
4. **Consider reducing timer interval to 8ms** (compensate for GLUT delays)
