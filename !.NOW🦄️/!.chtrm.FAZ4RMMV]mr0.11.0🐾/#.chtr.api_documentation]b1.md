# CHTM Framework API Documentation

## Overview
CHTM (Canvas HyperText Markup) is a terminal-based GUI framework that implements MVC architecture for creating CLI applications and games. The framework separates GUI logic (View) from business logic (Model-Controller) using a module system.

## Architecture

### MVC Pattern
- **Model (M)**: Data storage and management handled by the renderer
- **View (V)**: Display rendering handled by the renderer
- **Controller (C)**: Input processing and application logic handled by modules

### Communication
- Modules communicate with the main renderer via IPC (Inter-Process Communication)
- Uses a batched variable format: `VARS:name1=value1,name2=value2;`
- Modules send updates in batches to prevent multiple rendering cycles per action

## CHTM File Format

### Basic Elements
```html
<panel>          <!-- Container element -->
<text label="Text to display" />    <!-- Text element -->
<link label="Link Text" href="chtm/page2.chtm" key="1"/>  <!-- Link element -->
<button label="Button Text" key="2" onClick="SOME_ACTION"/>  <!-- Button element -->
<canvas id="my_canvas" width="8" height="8" label="Canvas Label" color="#FF0000"/>  <!-- Canvas element -->
<cli_io label="Command: ${command_input}" />  <!-- CLI input element -->
```

### Variables
- Use `${variable_name}` syntax to reference variables defined in modules
- Variables are automatically updated when modules send batched updates

### Module Tag
```html
<module>path/to/module_executable</module>
```
Place this at the bottom of your CHTM file to connect it with your business logic.

## Module Development Guide

### Creating a Module
1. Create a C program that communicates with the renderer via stdin/stdout
2. Implement handlers for various commands:
   - `DISPLAY` - Render canvas content
   - `KEY:code` - Handle keyboard input
   - `SIZE:width,height` - Handle canvas size changes
   - `CLI_INPUT:text` - Handle CLI input
   - `QUIT` - Handle program exit

### Variable Batch Format
Use the following macros to send variable updates:

```c
// Initialize buffer system
char var_buffer[4096];
int buf_pos = 0;

// For integers
ADD_VAR_INT(name, value);

// For characters  
ADD_VAR_CHAR(name, value);

// For strings
ADD_VAR_STR(name, value);

// Send all variables as a batch
printf("VARS:%s;\n", var_buffer);
```

### Module Communication Example
```c
// In your module's main loop
if (strcmp(input, "PLACE_OBJECT") == 0) {
    // Update internal state
    place_object_at(player_x, player_y);
    
    // Send batched updates
    buf_pos = 0;
    ADD_VAR_INT(frame_count, frame_count++);
    ADD_VAR_STR(last_action, "placed");
    ADD_VAR_CHAR(last_placed_symbol, 'X');
    ADD_VAR_INT(last_placed_at_x, player_x);
    ADD_VAR_INT(last_placed_at_y, player_y);
    ADD_VAR_INT(player_x, player_x);
    ADD_VAR_INT(player_y, player_y);
    
    printf("VARS:%s;\n", var_buffer);
    fflush(stdout);
}
```

## Canvas Module Architecture

### Layered Canvas System
- Layer 1: `original_canvas` - Static elements (base layer)
- Layer 2: `placed_objects` - Placed objects from palette
- Layer 3: `player_layer` - Player position and other dynamic elements

### Key Constants
```c
#define MAX_CANVAS_WIDTH 20
#define MAX_CANVAS_HEIGHT 20
#define PALETTE_SIZE 8
```

## Common Commands

### From Renderer to Module
- `DISPLAY[:ELEMENT_ID:id]` - Request canvas display with optional element ID
- `KEY:code[:ELEMENT_ID:id]` - Key press with optional element targeting
- `SIZE:width,height` - Canvas size change
- `CLI_INPUT:text` - CLI command input
- `QUIT` - Request module exit

### Handling Element-Specific DISPLAY Commands
When handling DISPLAY commands with ELEMENT_ID, ensure each element has proper initialization:

```c
if (strncmp(input, "DISPLAY", 7) == 0) {  // Check if command starts with "DISPLAY"
    // Parse for canvas ID if present in format "DISPLAY:ELEMENT_ID:canvas_id"
    char canvas_id[256] = {0};
    char *element_id_part = strstr(input, ":ELEMENT_ID:");
    
    if (element_id_part != NULL) {
        // Extract the canvas ID after :ELEMENT_ID:
        char *id_start = element_id_part + 12; // Length of ":ELEMENT_ID:"
        strncpy(canvas_id, id_start, sizeof(canvas_id) - 1);
        canvas_id[sizeof(canvas_id) - 1] = '\0';
    }
    
    // Handle based on canvas ID
    if (strlen(canvas_id) > 0 && strcmp(canvas_id, "emoji_palette") == 0) {
        // Handle emoji palette - send data to be stored in model by renderer
        // Use palette_symbols array directly to ensure correct symbols are sent
        for (int c = 0; c < PALETTE_SIZE; c++) {
            if (c == selected_palette_index) {
                putchar('>'); // Draw '>' in place of selected emoji
            } else {
                putchar(palette_symbols[c]); // Draw the actual emoji symbol directly from symbols array
            }
            putchar(' '); // Add a single space to separate items
        }
        putchar('\n');
        
        // Send associated variables
        // ... variable update code ...
    } else {
        // Handle regular game canvas - send composite canvas
        display_canvas();
    }
    fflush(stdout);
}
```

- `VARS:name1=value1,name2=value2;` - Batched variable updates
- Canvas grid data - For display elements

## Best Practices

### For Performance
- Always batch variable updates using the VARS format
- Only send variables that have changed
- Minimize the number of render-triggering operations per action

### For Responsive Input Handling
- **Never update display content directly from input handlers** - only update state variables
- **Only send variable updates from input handlers** - display updates should come from `DISPLAY` commands
- **Use the layered architecture pattern** for smooth, responsive movement:
  - Layer 1: Static elements (walls, floors)
  - Layer 2: Placed objects (items, etc.)
  - Layer 3: Dynamic elements (player, enemies)
- **Keep movement functions focused** on updating the player layer and sending variables
- **Never call display functions from key handling code** - this causes input lag

### For State Management and Initialization
- **Always ensure all canvas elements receive initial data** when the module starts, not just the primary canvas
- **Synchronize all related data structures** after loading state (e.g., keep `palette_canvas` in sync with `palette_symbols`)
- **Send initialization data for ALL elements** to prevent renderer confusion between different canvas elements
- **Call display functions for all canvases** after state loading to ensure renderer has correct initial state

### For Maintainability
- Use clear variable names
- Separate GUI logic from business logic
- Use the layered canvas architecture for complex displays
- Implement dedicated synchronization functions for related data structures

## Common Pitfalls and Troubleshooting

### Canvas Element State Confusion
**Problem**: When multiple canvas elements exist, one canvas may display content from another canvas initially, but work correctly after interaction/activation.

**Cause**: The renderer may not have initial state for all canvas elements, causing it to associate wrong data with canvas IDs until proper activation occurs.

**Solution**: 
1. Always send initial display data for ALL canvas elements during module initialization
2. Ensure all related data structures are properly synchronized after state loading
3. Call display functions for each canvas element to establish correct renderer state

**Example**:
```c
// After loading state or initializing, ensure ALL canvas elements 
// have their initial data sent to the renderer
display_canvas();    // For main game canvas
display_palette();   // For emoji palette or other canvases
```

### Data Structure Synchronization
**Problem**: Arrays or data structures that mirror each other become out of sync after state loading.

**Cause**: Only primary state is loaded, but derived or mirrored data structures aren't updated.

**Solution**: Create synchronization functions and call them after state operations:
```c
void sync_palette_canvas() {
    for (int c = 0; c < PALETTE_SIZE; c++) {
        palette_canvas[0][c] = palette_symbols[c];
    }
}
```

### Input Handling and Display Separation
**Problem**: Input commands (like arrow key movement) feel laggy or unresponsive.

**Cause**: Display functions are called directly from input handlers instead of being handled separately by `DISPLAY` commands.

**Solution**: Properly separate input handling from display updates:
1. Input handlers should only update state (player position in the player layer) and send variables
2. Display updates should only happen in response to `DISPLAY` commands from the renderer
3. Use the layered architecture to keep the display logic separated

**Correct Movement Function Pattern**:
```c
// Move player based on command - updates player layer only
void move_player(char direction) {
    // Clear the player from the current position in player layer
    player_layer[player_y][player_x] = ' ';
    
    // Update position based on direction
    switch (direction) {
        case 'U': // Up
            if (player_y > 0) player_y--;
            break;
        case 'D': // Down
            if (player_y < canvas_height - 1) player_y++;
            break;
        case 'L': // Left
            if (player_x > 0) player_x--;
            break;
        case 'R': // Right
            if (player_x < canvas_width - 1) player_x++;
            break;
    }
    
    // Place player at new position in player layer
    player_layer[player_y][player_x] = '@';
    
    // Send current state variables in batch
    buf_pos = 0;
    ADD_VAR_INT(player_x, player_x);
    ADD_VAR_INT(player_y, player_y);
    
    // Output batched variables
    printf("VARS:%s;\n", var_buffer);
    
    fflush(stdout);  // Important: flush immediately for responsiveness
}
```

## Getting Started Template

### 1. Create your CHTM file (e.g., `chtm/myapp.chtm`):
```html
<panel>
    <text label="My App Title" />
    <canvas id="main_canvas" width="10" height="10" label="Main Canvas" color="#00FF00"/>
    <text label="Status: ${status}" />
    <text label="Position: (${player_x}, ${player_y})" />
    <link key="1" label="Go to Help" href="chtm/help.chtm" />
</panel>
<module>modules/+x/myapp_module.+x</module>
```

### 2. Create your module (e.g., `modules/myapp/myapp_module.c`):
```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <stdbool.h>
#include <unistd.h>

#define VAR_BUFFER_SIZE 4096
char var_buffer[VAR_BUFFER_SIZE];
int buf_pos = 0;

#define ADD_VAR_INT(name, value) \
    buf_pos += snprintf(var_buffer + buf_pos, sizeof(var_buffer) - buf_pos, \
    "%s=%d,", #name, value)

#define ADD_VAR_CHAR(name, value) \
    buf_pos += snprintf(var_buffer + buf_pos, sizeof(var_buffer) - buf_pos, \
    "%s=%c,", #name, value)

#define ADD_VAR_STR(name, value) \
    buf_pos += snprintf(var_buffer + buf_pos, sizeof(var_buffer) - buf_pos, \
    "%s=%s,", #name, value)

int main() {
    // Initialize variables
    int player_x = 0, player_y = 0;
    char status[64] = "ready";
    
    // Send initial variables
    buf_pos = 0;
    ADD_VAR_INT(player_x, player_x);
    ADD_VAR_INT(player_y, player_y);
    ADD_VAR_STR(status, status);
    printf("VARS:%s;\n", var_buffer);
    fflush(stdout);
    
    char input[256];
    while (fgets(input, sizeof(input), stdin)) {
        input[strcspn(input, "\n")] = 0;
        
        if (strncmp(input, "KEY:", 4) == 0) {
            int key_code = atoi(input + 4);
            
            // Process key input
            switch (key_code) {
                case 1000: // ARROW_LEFT
                    player_x = (player_x > 0) ? player_x - 1 : player_x;
                    break;
                case 1001: // ARROW_RIGHT
                    player_x = (player_x < 9) ? player_x + 1 : player_x;
                    break;
                case 1002: // ARROW_UP
                    player_y = (player_y > 0) ? player_y - 1 : player_y;
                    break;
                case 1003: // ARROW_DOWN
                    player_y = (player_y < 9) ? player_y + 1 : player_y;
                    break;
            }
            
            // Send updated variables in batch
            buf_pos = 0;
            ADD_VAR_INT(player_x, player_x);
            ADD_VAR_INT(player_y, player_y);
            ADD_VAR_STR(status, "moved");
            printf("VARS:%s;\n", var_buffer);
            fflush(stdout);
        }
        else if (strcmp(input, "DISPLAY") == 0) {
            // Render canvas content
            for (int r = 0; r < 10; r++) {
                for (int c = 0; c < 10; c++) {
                    if (r == player_y && c == player_x) {
                        printf("P ");
                    } else {
                        printf(". ");
                    }
                }
                printf("\n");
            }
        }
        
        if (strcmp(input, "QUIT") == 0) break;
    }
    
    return 0;
}
```

### 3. Compile and run:
```bash
gcc -o modules/+x/myapp_module.+x modules/myapp/myapp_module.c
./chtml_d6.c chtm/myapp.chtm
```

## Platform Orchestrator

### Using the Platform Orchestrator
The platform orchestrator can run multiple command simultaneously and now supports specifying a custom CHTM file:

```bash
./+x/0.platform.orb.args]🪄️🔮️2gl📲️]e6.+x ./+x/22.kilQLL]ESCFIX.+x ./+x/chtml_d6.+x chtm/myapp.chtm
```

The last argument is treated as the CHTM file path if it contains ".chtm".
If no CHTM file is specified, it defaults to `chtm/index.chtm`.

The orchestrator will output a message indicating which CHTM file is being used.
