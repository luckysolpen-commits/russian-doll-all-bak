Here is a breakdown of the fundamental differences between the terminal (CLI) version and the OpenGL (GL) version, and a proposed solution to refactor the CLI version.

---
### 1. Core Architecture & Interaction Model
---

*   **CLI Version:**
    *   **Rendering:** Renders text with ANSI color codes directly to the terminal. The entire UI is a single stream of characters.
    *   **Interaction:** Keyboard-driven. The user presses a number key (e.g., '7') that is visually present in the text to select an option.

*   **GL Version:**
    *   **Rendering:** Renders graphics (shapes, text, etc.) in a dedicated window using OpenGL. It's a graphical user interface (GUI).
    *   **Interaction:** Mouse-driven. The user clicks on visual elements. The system identifies the clicked element by its coordinates.

---
### 2. Parsing & Data Structure
---

This is the most significant difference and the root cause of the confusion.

*   **CLI Version:**
    *   **Method:** Uses basic, repeated string searching (`strstr`) and replacement. It does not build a true representation of the document.
    *   **Data Structure:** It flattens the document's link structure into a simple, separate global array (`global_href_targets`). The link's target (`href`) is completely disconnected from its display text and its menu number, associated only by a fragile array index.

*   **GL Version:**
    *   **Method:** Implements a proper parser that reads the CHTM and builds a structured, in-memory representation of the document, much like a web browser's Document Object Model (DOM).
    *   **Data Structure:** Creates a tree of `UIElement` objects. Each tag in the CHTM file becomes an object. An `<a href="...">` tag becomes a `UIElement` object where the `href` and the display text are stored as properties *of that specific object*. The data is hierarchical and self-contained.

---
### 3. Link Handling & Navigation
---

*   **CLI Version:**
    *   **Association:** The link between the user's input ('7') and the navigation target is indirect and brittle. It relies on the view correctly parsing the number `7` from the text and using it as an index for the `global_href_targets` array. This can easily break if the CHTM formatting changes.
    *   **Navigation:** The controller takes the user's input number, uses it as an index to look up the `href` in the global array, and then triggers the page change.

*   **GL Version:**
    *   **Association:** The link is direct and robust. A mouse click targets a specific `UIElement` object. If that object is a link, its `href` property is read directly from it. There is no separate lookup table or fragile indexing scheme.
    *   **Navigation:** The controller identifies the clicked `UIElement`, gets the `href` property from it, and triggers the page change.

---
### 4. Proposed Solution: Refactor the CLI Version
---

To make the CLI version as clear and flexible as the GL version, you should adopt its data-oriented architecture. This separates parsing from rendering and makes the logic much more robust.

**Step 1: Introduce a `UIElement` Struct**
In your CLI application (e.g., in `model.c` or a new `ui.h` file), define a struct to represent a UI element. This will be your "DOM" object.

```c
// In a shared header file like model.h or a new ui.h
typedef struct {
    char type[20];      // "link", "text", etc.
    char label[256];    // The display text, e.g., "File"
    char href[512];     // The target path, e.g., "chtm/file.chtm"
    int key;            // The menu number, e.g., 7
} UIElement;

// Make a global array of these elements accessible
extern UIElement document_elements[100];
extern int document_element_count;
```

**Step 2: Refactor the Parser in `view.c`**
Change `print_ascii_grid` to be a two-stage process: parse first, then render.

1.  **Parse:** Read the CHTM file line-by-line. Instead of replacing strings on the fly, populate the `document_elements` array. When you find a line like `7. <a href="chtm/file.chtm">File</a>`, create a `UIElement` struct:
    *   `element.key = 7;`
    *   `element.type = "link";`
    *   `element.label = "File";`
    *   `element.href = "chtm/file.chtm";`

2.  **Render:** After the file is fully parsed into the `document_elements` array, loop through this array to print the UI to the terminal. For a "link" element, you would print: `printf("%d. [%s]", element.key, element.label);`.

**Step 3: Refactor the `controller.c`**
Update the controller to use the new `document_elements` structure. The confusing `global_href_targets` array can be completely removed.

```c
// In process_input() when a number key is pressed
bool process_input(int ch) {
    if (ch >= '0' && ch <= '9') {
        int selected_key = ch - '0';

        // Find the UI element with the matching key
        for (int i = 0; i < document_element_count; i++) {
            if (document_elements[i].key == selected_key && strcmp(document_elements[i].type, "link") == 0) {
                
                // Found it. Navigate to its href.
                switch_to_page(document_elements[i].href);
                set_status_message("Navigating...");
                return true; // Indicate that a screen update is needed
            }
        }
    }
    // ... other input handling ...
}
```

By making these changes, you separate the data (your UI structure) from the presentation (the text printed to the terminal), which is the core strength of the GL version's design. Your code will be cleaner, less error-prone, and much easier to extend in the future.
