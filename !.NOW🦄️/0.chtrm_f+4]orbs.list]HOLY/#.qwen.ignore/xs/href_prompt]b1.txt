
### Prompt: Foundational Refactoring of C-MVC for CHTM Specification Alignment

**Preamble for the Language Model:**
This is a foundational refactoring task. The goal is to overhaul the UI parsing and rendering layer of a simple C-based terminal MVC application. The current implementation is brittle and fails to correctly handle `href` navigation. The objective is to replace it with a robust, extensible architecture inspired by a more advanced graphical (OpenGL) reference version of the same framework. You should leverage your large context and reasoning abilities to implement this new architecture thoroughly, following the detailed plan below.

---
### Section 1: The Overarching Goal

The primary goal is to **replace the current string-replacement rendering in `view.c` with a robust, two-stage "Parse then Render" architecture.**

This new architecture is the cornerstone of the reference GL application and is essential for building a reliable and feature-rich terminal UI. The process will be:
1.  **Parse:** Read the entire CHTM document and transform it into a structured, in-memory "DOM tree" (represented as a flat array of `UIElement` structs with parent-child relationships).
2.  **Render:** Separately, iterate through the completed DOM tree and render the UI to the terminal.

---
### Section 2: The Core Problem (Why the Current CLI Fails)

The current `href` navigation is broken because the CLI's parser is naive. It uses line-based string scanning to *guess* the association between a menu number (e.g., `7.`) and an `<a>` tag. This fails because it makes incorrect assumptions about the CHTM file's layout.

The GL version succeeds because its parser builds a reliable, structured model of the UI, directly linking an `href` attribute to its specific UI element. We will now implement this superior model.

---
### Section 3: The New, Expanded `UIElement` Struct

To support a richer UI, we must first expand the `UIElement` struct. This definition should be placed in `view.c` and duplicated in `controller.c` to maintain the project's convention of not using new header files.

**New `UIElement` Definition:**
```c
typedef struct {
    char type[20];          // E.g., "link", "button", "text", "panel"
    char label[256];        // Display text for links, buttons, text elements
    char href[512];         // Target path for <link> tags
    char onClick[256];      // Command string for <button> tags (e.g., "SAVE_FILE")
    char module_path[512];  // Path for <module> tags
    int key;                // The numeric key (1-9) for CLI selection
    int parent_index;       // The array index of the parent element (-1 for root elements)
    char id[50];            // The element's ID attribute
} UIElement;
```

---
### Section 4: The Implementation Plan: A Detailed Walkthrough

This is a major refactoring of `view.c` and `controller.c`.

#### **Part A: The New Parser (in `view.c`)**

**Goal:** Create a robust, token-aware parser that builds the `document_elements` DOM array.

**Logic:**
1.  The new parser must replace the entire body of the `print_ascii_grid` function.
2.  It should loop through the CHTM source string, identifying tags (`<...>`) and plain text content between them.
3.  **Tag Identification:** When a tag is found (e.g., `<link ...>`), it must parse the tag name ("link").
4.  **Attribute Parsing:** It must be able to parse an arbitrary number of key-value attributes within a tag, such as `href="..."`, `label="..."`, `key="..."`, and `onClick="..."`.
5.  **DOM Creation:** For each recognized CHTM element, it will create a `UIElement` struct and populate its fields from the parsed tag and attributes.
6.  **Structural Parsing (Parenting):** The parser must handle container tags (`<panel>`, `<header>`). It should use a stack to keep track of the current parent container. When a new element is parsed, its `parent_index` should be set to the index of the element currently on top of the stack. When a closing tag (e.g., `</panel>`) is found, the corresponding element is popped from the stack.

#### **Part B: The New Renderer (in `view.c`)**

**Goal:** Render the UI based *only* on the `document_elements` array created by the parser.

**Logic:**
1.  This rendering logic will come *after* the parsing logic within the `print_ascii_grid` function.
2.  It will iterate through the `document_elements` array from `i = 0` to `document_element_count`.
3.  **Rendering Rules:** It will use `printf` to generate output based on the `type` of each element.
    *   `link`: `printf("%d. [%s]\n", el->key, el->label);
    *   `button`: `printf("%d. [%s]\n", el->key, el->label);
    *   `text`: `printf("%s\n", el->label);
    *   `panel`/`header`: For now, these do not need to render anything visually, but the parser must still recognize them to build the correct parent-child structure.
4.  **Deprecation:** This entire "Parse then Render" process completely replaces the old `replace_tag_content` and `strip_html_tags` functions. Those helper functions should be deleted. The dynamic `generate_*_string` functions (for canvas, palettes, etc.) should be kept, as they can be called by the renderer when it encounters special tags like `<canvas />`.

#### **Part C: The Updated Controller (in `controller.c`)**

**Goal:** Adapt the controller to use the new, richer `UIElement` model for dispatching actions.

**Logic:**
1.  The `process_input` function will continue to use the "select with number, confirm with Enter" flow.
2.  When the user presses Enter, the controller will find the `UIElement` in the `document_elements` array where `key == menu_selection`.
3.  **Action Dispatching:** Based on the element's `type`, it will perform a different action:
    *   If `type` is **"link"**, it calls `switch_to_page(el->href)`.
    *   If `type` is **"button"**, it must process the `el->onClick` string. This introduces a new command dispatch system. For example, if `el->onClick` is `"SAVE_FILE"`, the controller should contain logic to handle a save operation. For now, it can just print a status message like `"Button clicked: SAVE_FILE"`.

---
### Section 5: CHTM Features to Implement in This Pass

The new parser should be built to handle the following tags and attributes from the CHTM specification, adapted for the CLI:

*   **`<link>`:** Must parse `href`, `label`, and the CLI-specific `key` attribute.
*   **`<button>`:** Must parse `label`, `onClick` (as a command string), and `key`.
*   **`<text>`:** Must parse `label` or `value` as the display text.
*   **`<panel>`, `<header>`:** Must be recognized as structural container elements to enable parent-child relationships, even if they don't render visually yet.
*   **`<module>`:** Must be updated to parse the module path from the content between the tags (e.g., `<module>path/to/module</module>`), matching the GL spec.
*   **Special Tags:** The parser should also recognize the self-closing placeholder tags like `<canvas />`, `<status_message />`, etc., and create corresponding `UIElement`s so the renderer knows when to call the `generate_*_string` helper functions.
