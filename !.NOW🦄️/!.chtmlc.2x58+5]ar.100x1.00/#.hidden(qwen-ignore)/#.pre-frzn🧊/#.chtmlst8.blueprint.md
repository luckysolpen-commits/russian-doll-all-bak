# CHTML (C-HTML) Framework - Architecture Blueprint

## Overview
CHTML is a custom GUI framework that renders HTML-like markup in OpenGL for creating interactive desktop applications. It uses an MVC (Model-View-Controller) architecture to separate concerns and maintain clean, modular code.

## Core Architecture

### MVC Components

#### Model (`2.model.c`)
- Handles data and business logic
- Manages external modules via IPC (Inter-Process Communication)
- Maintains state for shapes, variables, and arrays
- Communicates with external executables using stdin/stdout pipes
- Supports communication protocol with messages like:
  - `CLEAR_SHAPES`, `SHAPE;type;x;y;width;height;r;g;b;a`
  - `VAR;label;type;value`, `ARRAY;label;type;count;val1,val2,...`

#### View (`3.view.c`) 
- Responsible for UI rendering and CHTML parsing
- **NEW**: Implements complete DOM tree functionality
- Parses CHTML files into a hierarchical DOM structure during parsing
- Maintains `UIElement` array for OpenGL rendering
- Supports rich UI elements: buttons, textfields, sliders, canvas, menus, etc.
- Handles emoji rendering with FreeType integration

#### Controller (`4.controller.c`)
- Handles user input and event processing
- **NEW**: Dynamic event system sending events to external modules
- Processes mouse, keyboard, and special key events
- Coordinates interaction between model and view
- Implements DOM-aware event handling

## NEW: DOM Tree System

### DOM Node Structure
```c
typedef struct DOMNode {
    char* tag_name;           // Element tag name
    char* id;                 // Element ID attribute
    char* class_name;         // Element class attribute  
    char* text_content;       // Element text content
    DOMAttribute attributes[MAX_ATTRIBUTES]; // HTML attributes
    struct DOMNode* parent;   // Parent node reference
    struct DOMNode* children[MAX_CHILDREN]; // Child nodes
    // Event handling and other properties...
} DOMNode;
```

### DOM Capabilities
- **Dynamic Parsing**: Builds DOM tree during CHTML parsing automatically
- **Element Querying**: `getElementById()`, `getElementsByTagName()`, `getElementsByClassName()` equivalents
- **Hierarchical Structure**: Maintains parent-child relationships accurately
- **Attribute Access**: Complete attribute management system
- **No Hardcoding**: Handles arbitrary tags and event handlers without registration

### Event System
- **Dynamic Event Handling**: Passes event names directly to modules via `EVENT:<handler>` format
- **Module Integration**: Events like `button_increment_handler` are forwarded to external modules
- **UI Events**: Special UI-only handlers (view switching) handled in controller
- **JavaScript-like Bubbling**: Events propagate to modules for processing

## CHTML Markup Support

### Supported Elements
- `<window>`, `<header>`, `<text>`, `<button>`, `<panel>`
- `<textfield>`, `<textarea>`, `<checkbox>`, `<slider>`
- `<canvas>`, `<menu>`, `<menuitem>`, `<dirlist>`, `<link>`, `<module>`

### Communication Protocol
#### From UI to Module:
- `SLIDER:<id>:<value>` (slider changes)
- `CANVAS:<x>,<y>` (canvas clicks)  
- `EVENT:<handler_name>` (general events)

#### From Module to UI:
- `CLEAR_SHAPES`, `SHAPE;type;label;x;y;z;width;height;depth;r;g;b;a`
- `VAR;label;type;value`, `ARRAY;label;type;count;val1,val2,...`

## Main Application Entry Point (`1.main_prototype_a1.c`)

### Core Responsibilities
- **Application Initialization**: Sets up GLUT window and OpenGL context
- **MVC Component Initialization**: Orchestrates the initialization of model, view, and controller
- **Module Discovery**: Parses module tags from CHTML and initializes external modules
- **Event Loop Management**: Implements the main GLUT event loop with optimized rendering
- **Navigation System**: Handles file-based page navigation and reload requests

### Key Functions
- `main()`: Entry point that takes CHTML file path as argument and initializes the application
- `idle_func()`: Optimized rendering loop that updates model state and checks navigation requests (~60 FPS)
- `reload_view_with_file()`: Handles dynamic reloading of different CHTML pages
- `check_navigation_requests()`: Monitors nav_request.tmp file for navigation commands
- `init_multiple_modules()`: Initializes all modules specified in the CHTML file

### Navigation System
- Uses `nav_request.tmp` coordination file for cross-process navigation
- Supports both relative and absolute path navigation via href attributes
- Implements reload mechanism for page transitions
- Maintains current file path state for relative path resolution

### Performance Features
- Optimized idle loop with 16ms sleep for ~60 FPS
- Efficient update checking before UI refresh
- Proper cleanup of view state during page transitions

## Benefits of the New Architecture

1. **Flexibility**: Can handle arbitrary CHTML tags without recompilation
2. **Maintainability**: Clean separation of concerns with DOM layer
3. **Extensibility**: Easy to add new element types and query capabilities  
4. **Performance**: DOM built during parsing, no duplicate work
5. **LLM Compatibility**: Modular design allows focused context prompting
6. **Backward Compatibility**: All existing functionality preserved

## Navigation System
- File-based page navigation using href attributes
- Coordinate file for navigation requests
- Page caching system with history
- Support for relative and absolute paths

## Module System
- External executable integration via fork/pipe
- Multiple module support per page
- Real-time communication protocol
- Dynamic content generation from modules

This architecture creates a robust, flexible framework that bridges traditional HTML concepts with custom C-based rendering, providing both the power of external module processing and the flexibility of dynamic DOM manipulation.
