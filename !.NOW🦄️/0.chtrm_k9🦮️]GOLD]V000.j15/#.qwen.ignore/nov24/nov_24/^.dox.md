# CHTM Renderer Documentation

## Overview
This document explains the architecture and functionality of the CHTM renderer, particularly focusing on input handling and canvas rendering mechanisms.

## Architecture: MVC Pattern

The system follows a Model-View-Controller (MVC) pattern:

- **Model (M)**: Handles data storage and management, including canvas data
- **View (V)**: Responsible for display rendering 
- **Controller (C)**: Manages input processing and coordinates between Model and View

## Input Handling and IPC Communication

### User Input Flow

1. **Input Capture**: The system reads keyboard input from `log.txt` instead of direct terminal input
2. **Input Processing**: The `process_input()` function handles different key types:
   - Arrow keys, Enter, ESC, printable characters, etc.
3. **IPC Communication**: Input is sent to modules via named pipes (IPC)

### IPC Mechanism

The renderer establishes bidirectional communication with external modules via pipes:

```c
int parent_to_child[2];  // Pipe for sending commands to module
int child_to_parent[2];  // Pipe for receiving responses from module
```

### Key Command Mapping

When elements are active [^], input gets forwarded to the appropriate module:

- **Canvas elements**: Arrow keys → "MOVE:U", "MOVE:D", "MOVE:L", "MOVE:R"
- **Canvas spacebar**: → "32" (space character)  
- **CLI elements**: Printable input → "CLI_INPUT:text"
- **All elements**: Enter key → "INPUT:10"

## Canvas Rendering and Player Movement

### How Arrow Keys Move the "p" Player

The arrow key movement system works as follows:

1. **Input Forwarding**: When a canvas element is active and arrow keys are pressed, commands like "MOVE:U" are sent via IPC to the module
2. **Module Processing**: The external module receives the command and updates its internal state (moves the "p" player)
3. **Canvas State Update**: The module sends back updated canvas data line by line
4. **Renderer Update**: The renderer receives the updated canvas lines via IPC and displays them

### Canvas Data Flow

#### Model Layer (Data Storage)
- `CanvasData` structure stores canvas grid data:
  ```c
  typedef struct {
      char grid[MAX_CANVAS_HEIGHT][MAX_CANVAS_WIDTH * 2]; // Canvas character grid
      int height;
      int width;
      bool valid; // Whether data is current
      char id[MAX_ATTR_LEN]; // Canvas element ID
  } CanvasData;
  ```
- `model_canvas_data[]` array stores canvas data for each canvas element
- Functions like `model_update_canvas_data()` update the stored grid data

#### View Layer (Display)
- When rendering canvas elements, the system:
  1. Sends "DISPLAY" command to module via `send_ipc_command()`
  2. Reads updated canvas data from module via `read_ipc_response()`
  3. Updates internal model with `model_update_canvas_data()`
  4. Renders the canvas grid to screen

#### Variable Updates
- Modules can send variable updates in format "VAR:name=value"
- These update the renderer's variable system, which can be referenced in CHTM files via ${var_name} syntax

## Current Architecture Analysis

### Pure Module Implementation (Desired)
✓ **Current State**: Player movement is handled entirely within the module
- Arrow key commands ("MOVE:U", "MOVE:D", etc.) are sent to the module
- The module processes movement logic internally
- Updated canvas state is sent back to renderer
- No reliance on CHTM framework variables for movement

### Legacy Scaffolding (Undesired)
⚠️ **Potential Issues**: Some variable-based interactions exist:
- Module can send "VAR:" responses to update renderer variables
- Renderer supports variable substitution in CHTM labels via ${var_name}
- These are available for modules to use but not required for basic canvas rendering

## API Available to Modules

### Input Commands (Processed from log.txt via Controller)
- `SIZE:width,height` - Canvas dimensions initialization
- `KEY:n:ELEMENT_ID:id` - Raw ASCII key codes with element targeting for active canvas/CLI modules (e.g., `KEY:1000:ELEMENT_ID:emoji_palette` for arrow keys)
- `CLI_INPUT:text` - Text input for CLI elements (from log.txt via controller)
- `DISPLAY` - Request to display canvas
- `DISPLAY_PALETTE` - Request to display palette

### Key Handling Architecture
- **Arrow keys**: Sent to modules as `KEY:1000-1003` with element targeting to distinguish between game canvas and emoji palette
- **Number keys**: Sent to modules as `KEY:48-57` for palette selection
- **Enter key (10)**: Sent to modules with element targeting for custom behavior
- **Spacebar (32)**: Sent to modules with element targeting for custom behavior
- **Element Targeting**: Commands include `ELEMENT_ID:id` to direct input to specific elements

### Character Placement Issue & Next Steps

The `place_selected_symbol()` function in the canvas module is currently causing rendering corruption issues when called. The function has been commented out to prevent these issues, but character placement functionality will be needed for the next phase of development.

**Known Issue**: Calling `place_selected_symbol()` causes rendering corruption in the display system, though the exact cause requires investigation.

**Next Steps**:
1. Debug the `place_selected_symbol()` function to identify the root cause of rendering corruption
2. Determine if the issue is related to:
   - Race conditions between multiple canvas updates
   - Conflicts between player movement and symbol placement logic
   - Synchronization issues between renderer and module state
   - File I/O operations (like updating map_from_module.txt) that interfere with display
3. Implement a stable character placement system as a follow-up feature

### Refactoring Achieved: Generic ASCII Input System
The framework has been successfully refactored to send raw ASCII codes directly to modules:

- **Previous**: Controller interpreted keys and sent semantic commands like `MOVE:U`
- **Current**: Controller sends raw ASCII codes like `KEY:259` for arrow up, `KEY:10` for enter
- **Benefit**: Modules have complete control over input interpretation, enabling more diverse applications
- **Implementation**: `process_input()` now forwards raw key codes instead of semantic commands for most inputs

#### Implementation Details

**Controller Changes (`process_input` function)**:
- Arrow keys now send `KEY:n` format with their raw ASCII codes instead of semantic `MOVE:U` commands
- Enter key now sends `KEY:10` instead of `INPUT:10` for canvas elements
- All printable characters send as `KEY:n` to active canvas elements
- The controller still handles:
  - Element activation/deactivation (at UI level)
  - Focus navigation when no element is active
  - File switching for links
  - CLI text input directly (not as key codes)

**Module Interface Changes**:
- Modules now receive `KEY:code` instead of semantic commands like `MOVE:U`
- Modules are responsible for interpreting the raw key codes according to their needs
- Special commands like `SIZE:`, `DISPLAY`, `VAR:` remain for essential renderer-module communication

**Benefits**:
- Modules can implement custom key mappings and behaviors
- Applications can respond to keys in ways not anticipated by the renderer
- Framework becomes more generic and flexible
- Enables development of various types of applications beyond canvas games

### Output Format (Sent to Renderer)
- **Canvas data**: Multiple lines of character grid data
  - Format: "x x x x ..." where x represents individual cell values with spaces between
  - Each line corresponds to one row of the canvas
- **Variable updates**: "VAR:name=value" to update renderer variables
- **Module responses**: Any other text for logging/debugging purposes

## Path Forward & Recommendations

### Maintain Pure Module Architecture
✅ **Recommended**: Continue with pure module-based processing:
- Keep player movement logic inside modules
- Use IPC only for input/output communication
- Avoid dependency on renderer variables for core functionality

### Module Best Practices
- Process all game logic internally within the module
- Send only updated visual state to renderer
- Use "VAR:" messages only for information that needs to appear in CHTM labels
- Consider canvas dimensions sent via "SIZE:" command as definitive

### Renderer Enhancements
- Focus on reliable IPC communication
- Maintain consistent canvas data model
- Ensure proper error handling when modules don't respond
- Support multiple canvas types (game canvas, emoji palette, etc.)

### Security Considerations
- The current design isolates module logic from renderer
- Use structured command format to prevent injection attacks
- Validate canvas dimensions and data before rendering
