# Modular Menu Refactor Plan - Revised

## Overview
Create a completely modular system where all functionality (including canvas) is separated into individual modules, managed by a central orchestrator application.

## New Architecture
```
[orchestrator] - Main application that manages everything
├── [canvas] - Handles canvas display and interaction
├── [emoji_palette] - Handles emoji selection
├── [color_palette] - Handles color selection
├── [tool] - Handles tool selection
├── [maps] - Handles maps (placeholder)
├── [events] - Handles events
└── [player] - Handles player on/off state
```

## Implementation Approach

### Phase 1: Create Orchestrator Application
- Acts as the main entry point
- Manages which module is currently active
- Handles the main menu display
- Handles inter-module communication
- Provides common functionality (status messages, etc.)

### Phase 2: Create Individual Modules
- Each module becomes its own executable in hydrgn.modules/+x/
- Each module handles its specific functionality
- Modules communicate state back to orchestrator
- Modules can be built independently

### Phase 3: Communication Protocol
- Define how modules communicate with orchestrator
- Define how modules share state (like canvas state)
- Handle persistence between module calls

### Phase 4: Migration and Testing
- Migrate functionality from existing app to modules
- Test each module individually
- Test the orchestrator with modules
- Ensure all features work as before

## Key Benefits
- Each feature can be included/excluded in builds independently
- Modules can be developed and tested separately
- Easier to maintain and extend
- Can create different builds for different use cases (headless, with/without specific features)

## Implementation Order
1. Create orchestrator application
2. Create canvas module
3. Create emoji_palette module
4. Create color_palette module
5. Create tool module
6. Create maps module
7. Create events module (update existing)
8. Create player module
