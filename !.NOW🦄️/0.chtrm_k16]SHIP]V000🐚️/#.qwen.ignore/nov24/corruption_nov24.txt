Venn Diagram Analysis: Character Placement vs Arrow Key Movement
=================================================================

SET A: Arrow Key Movement (Working)
SET B: Character Placement (Corrupted)

Intersection: Common Elements That Should Work Similarly
-------------------------------------------------------

BOTH HAVE:
- [X] Canvas state updates (modifies canvas[player_y][player_x])  
- [X] Variable updates sent to renderer (VAR:player_x, VAR:player_y)
- [X] File I/O operations (updating map_from_module.txt)
- [X] fflush(stdout) calls for IPC communication
- [X] Same canvas grid structure access (2D char array)
- [X] Renderer-module IPC communication

UNION: All Combined Elements
---------------------------

A UNION B:
- Canvas state updates
- Variable updates sent to renderer  
- File I/O operations
- fflush(stdout) calls
- Same canvas grid structure access
- Renderer-module IPC communication
- Player position tracking
- Original canvas restoration
- Display synchronization

SET A: Arrow Key Movement (Working) ONLY
----------------------------------------

- move_player() function
- Simple position changes (player_x/y updates)
- Backup of original character at old position
- Restore original character at old position
- Set player 'p' at new position 
- No additional symbol placement logic
- Minimal canvas character changes (just move 'p' from A to B)
- Direct position adjustments without value changes

SET B: Character Placement (Corrupted) ONLY
------------------------------------------

- place_selected_symbol() function
- Complex canvas modification (changes original_canvas AND canvas)
- Multiple file writes (both map_from_module.txt and map.txt)
- Double variable updates (placement + position)
- Changes to original_canvas[player_y][player_x] 
- Changes to canvas[player_y][player_x] 
- Additional logic for selected symbol handling
- Potential race conditions with display updates
- Two different file targets (map_from_module.txt vs map.txt)
- More complex state management
- Multiple fprintf() calls to files during single operation

DEEP DIVE: Why Character Placement Causes Corruption
----------------------------------------------------

ROOT CAUSES:

1. FILE CONFLICTS:
   - Arrow movement: Only updates map_from_module.txt 
   - Character placement: Updates BOTH map_from_module.txt AND map.txt
   - This creates race conditions when renderer tries to read files

2. DOUBLE VARIABLE UPDATES:
   - Arrow movement: Updates minimal variables (player_x, player_y)
   - Character placement: Updates extensive variables (last_action, last_placed_symbol, etc.)
   - Too many variable updates too quickly creates IPC buffer issues

3. STATE CONSISTENCY:
   - Arrow movement: Maintains state consistency with original_canvas restoration
   - Character placement: Changes both canvas and original_canvas simultaneously 
   - This can create inconsistencies between renderer and module views

4. SYNCHRONIZATION ISSUES:
   - Arrow movement: Single, clean state transition
   - Character placement: Multiple state transitions in rapid succession
   - Renderer may receive partial updates at wrong times

5. CANVAS UPDATE PATTERNS:
   - Arrow movement: Only moves player ('p') symbol, doesn't change underlying character
   - Character placement: Changes underlying character AND maintains player position
   - Different complexity levels of canvas manipulation

SOLUTION PATHS:
1. Unify file output to single file (preferably map_from_module.txt)
2. Reduce variable update frequency or batch them
3. Implement proper synchronization with renderer
4. Simplify the character placement logic to match movement pattern
5. Add delays or locks to prevent race conditions

WORKING HYPOTHESIS:
The corruption occurs not from the canvas manipulation itself, but from the rapid-fire file I/O and variable updates that happen during character placement, which overwhelms the renderer's ability to process updates cleanly, unlike the simple single-file, minimal-variable approach of arrow key movement.
