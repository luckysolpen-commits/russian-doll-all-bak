INNERHTML FUNCTIONALITY SPECIFICATION

GOAL: Implement JavaScript-like innerHTML functionality for the CHTM framework that allows dynamic content replacement and DOM manipulation.

SPEC REQUIREMENTS:

1. COMMAND FORMAT:
   - INNER_HTML:ELEMENT_ID:CHTM_CONTENT
   - Example: INNER_HTML:projects_container:<button onClick="LOAD:project1">Project 1</button><br/><button onClick="LOAD:project2">Project 2</button>

2. ELEMENT IDENTIFICATION:
   - Find existing element by its ID attribute
   - If ID not found, log error and return

3. CONTENT REPLACEMENT LOGIC:
   - Remove all existing children of the target element
   - Parse the new CHTM content string
   - Create new UIElement structures from the parsed content
   - Establish parent-child relationships with the target element as parent
   - Add new elements to the elements array while respecting MAX_ELEMENTS limit

4. PARSING REQUIREMENTS:
   - Use the existing tokenize_chtm() and parse_chtm_file() logic
   - But adapt it to parse content fragments instead of full files
   - Handle nested elements, attributes, and self-closing tags
   - Maintain the same attribute parsing rules (label, onClick, etc.)

5. INTERACTIVITY GUARANTEE:
   - All newly created elements must be fully interactive
   - They must respond to navigation (arrow keys, enter)
   - They must be clickable and respond to onClick handlers
   - They must support focus states ([>], [^])

6. DOM-INTEGRATION:
   - New elements become part of the elements array
   - Parent-child relationships are properly set
   - Navigation index calculations include new elements
   - The render function displays new elements correctly

7. SAFETY REQUIREMENTS:
   - Respect MAX_ELEMENTS limit
   - Validate CHTM content before parsing
   - Handle malformed content gracefully
   - Prevent infinite loops or stack overflows

8. PERFORMANCE:
   - Efficient element addition/removal
   - Minimal memory allocation/deallocation
   - Fast rendering updates after content change

9. BACKWARD COMPATIBILITY:
   - Existing functionality must remain unchanged
   - Old CHTM files continue to work
   - Navigation and interaction patterns remain the same

10. TESTING SCENARIOS:
   - Empty content replacement
   - Adding single elements
   - Adding multiple elements
   - Adding nested elements
   - Removing and re-adding content
   - Maximum element count handling
   - Elements with all different types (button, link, text, etc.)

TECHNICAL APPROACH:
- Create helper function to parse CHTM fragment and add to existing DOM
- Modify element removal logic to handle dynamic content
- Update all functions that iterate through elements to handle dynamic changes
- Ensure render_elements() correctly displays dynamically added content
