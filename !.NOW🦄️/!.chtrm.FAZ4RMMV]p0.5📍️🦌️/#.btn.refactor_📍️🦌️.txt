# Button Refactor Prompt: Unifying Button and Link Execution Logic

## Problem Statement
The current architecture has inconsistent execution logic between buttons and links:
- **Links**: Execute immediately when active (`switch_to_file(el->href);`)
- **Buttons**: Have a two-stage activation process requiring multiple Enter presses

This inconsistency creates a confusing user experience where buttons behave differently than other interactive elements.

## Current Button Logic (Problematic)
```c
// For buttons, check if this is the button that should execute on this Enter press
if (execute_on_next_enter == active_interaction_element && strlen(el->onClick) > 0) {
    // Execute on second Enter press
    send_ipc_command(el->onClick);
    execute_on_next_enter = -1;
} else if (execute_on_next_enter != active_interaction_element) {
    // Mark on first Enter press
    execute_on_next_enter = active_interaction_element;
}
```

## Current Link Logic (Desired Pattern)
```c
// For links, execute the action (navigate to href) and keep element active
switch_to_file(el->href);
```

## Menu Child Button Complexity
Menu child buttons have additional complexity because:
1. When a child button is in a menu, the parent menu should remain active even after the child executes
2. Parent menu maintains its state while child executes
3. Child buttons are treated as 'sudo active' to allow parent to stay active

## Solution Requirements

### 1. Mirror Link Functionality for Buttons
- Buttons should execute immediately when active (like links do)
- No more two-stage execution (mark then execute)
- Direct execution on Enter press when active

### 2. Preserve Menu Child Nuance
- When a button is a child of a menu, executing it should not deactivate the parent menu
- Parent menu should remain active after child button execution
- Handle navigation between child elements within the same menu

### 3. Maintain Consistent UX
- All interactive elements should respond uniformly to Enter key
- No special cases for button activation vs other elements
- Menu navigation should work consistently (arrow keys to navigate, Enter to execute)

## Expected Behavior After Refactor

### Top-level Buttons
- Enter when focused -> activates button [^] 
- Enter when active -> executes onClick immediately (like links)

### Menu Child Buttons  
- Enter when focused in menu -> highlights child [^]
- Enter when child is active -> executes child onClick but keeps parent menu active
- Parent menu state maintained after child execution

## Technical Implementation Notes

### Current execute_on_next_enter Pattern
The global `execute_on_next_enter` variable creates the two-stage process that's inconsistent with other elements. This may need to be restructured or eliminated.

### Menu State Preservation
Care must be taken to preserve the parent menu's active state when child buttons execute, unlike standalone elements which may lose focus after execution.

### IPC Command Processing
The button's onClick handling (including INNER_HTML commands) should occur immediately when active, with proper response processing and DOM updates.
