INNERHTML TIMING OPTIMIZATION PROMPT

CURRENT STATE:
- InnerHTML functionality is working correctly
- Dynamic content is successfully added to DOM and rendered
- Issue: Visual update has noticeable delay after button execution
- Requires 3 key presses to see results instead of expected 2

PROBLEM DESCRIPTION:
When a user presses Enter to execute a button that triggers INNER_HTML, the content update doesn't appear immediately. The user must press additional keys for the visual update to become apparent. This indicates a timing issue in the response processing pipeline.

TECHNICAL FLOW:
1. User presses Enter on button (already activated as [^])
2. send_ipc_command() sends request to module  
3. process_module_responses() is called immediately after
4. Module processes command and sends INNER_HTML response
5. Main loop eventually processes INNER_HTML response and updates display
6. Issue: Step 5 doesn't happen immediately, causing visible delay

ROOT CAUSE ANALYSIS:
The delay occurs because module responses arrive asynchronously. Even though process_module_responses() is called immediately after send_ipc_command(), the module may not have finished processing and sending the response back yet. Responses are processed in the next available cycle of the main loop, creating a perceptible lag.

DESIRED OUTCOME:
Visual content updates should appear immediately after button execution with minimal perceptible delay, requiring only 2 key presses total (1st to activate [^], 2nd to execute and see results).

REQUEST:
Optimize the IPC response processing pipeline to minimize delay between button execution and visual content update. Consider:
- Polling strategies for faster response detection
- Synchronization mechanisms between command sending and response processing
- Buffering techniques to ensure immediate processing of expected responses
- Alternative approaches to achieve near-immediate visual feedback after INNER_HTML commands
