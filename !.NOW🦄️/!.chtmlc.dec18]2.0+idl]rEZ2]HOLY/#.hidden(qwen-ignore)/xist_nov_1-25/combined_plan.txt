# Shared Memory Module System - Next Iteration Prompt

## Current State Analysis
- Successfully implemented shared memory infrastructure in the chtml model (2.model_v0.01.c)
- Added support for protocol="shared" in CHTML module tags
- Created foundation with ModuleSharedMem structure, semaphores, and communication functions
- Maintained full backward compatibility with pipe-based modules
- System can now handle both pipe-based and shared memory protocols simultaneously

## Core Issue Identified
The current implementation creates shared memory infrastructure in the model, but external module executables (like shared_player_movement.+x) are still running as separate forked processes using stdin/stdout pipes rather than actually using the shared memory infrastructure that was created for them.

## Desired Architecture
Need to modify the module execution system so that:
1. When protocol="shared" is specified, the module directly uses the shared memory infrastructure instead of stdin/stdout pipes
2. Modules execute within the same process space or have direct access to the created shared memory segments
3. Eliminate the pipe-based communication for shared memory modules, using only shared memory + semaphores

## Implementation Requirements
1. **Change module execution flow**: Instead of fork/exec with pipes, modules with protocol="shared" should use the shared memory segments created by the model
2. **Module-Infrastructure Integration**: Either integrate module logic directly into the main process or ensure the module executable can properly connect to pre-existing shared memory segments
3. **Timing Coordination**: Ensure shared memory segments exist before the module tries to access them
4. **Communication Channel Replacement**: For shared protocol, replace stdin/stdout pipe communication with shared memory + semaphore communication

## Success Criteria
- When using `<module protocol="shared">`, the module communicates exclusively via shared memory segments (not pipes)
- Significant performance improvement over pipe-based communication for shared memory modules
- Maintain backward compatibility for protocol="pipe" modules
- Demonstrate the 3D/2D player movement demo using true shared memory communication

## Next Steps
Modify the module initialization and communication system in 2.model_v0.01.c and related files to ensure that when protocol="shared" is specified, the module uses the shared memory infrastructure instead of stdin/stdout pipe communication.

# Shared Memory Module System Blueprint

## Overview
This blueprint outlines the plan to replace the current IPC (pipe-based) communication system with a shared memory system for better performance.

## Current Architecture
- Main application (model) communicates with external modules via pipes
- Uses fork() to create child processes
- Pipes for bidirectional communication (parent_to_child_pipe, child_to_parent_pipe)
- Messages formatted as "VAR;label;type;value", "SHAPE;...", etc.
- Non-blocking I/O with fcntl() to set O_NONBLOCK

## New Architecture (Shared Memory)
- Replace pipes with shared memory segments
- Use semaphores for synchronization
- Maintain backward compatibility during transition

## Phase 1: Shared Memory Infrastructure

### 1.1 Define shared memory structures
```c
// Shared memory block for module communication
typedef struct {
    int module_id;
    int active;                    // Whether this module is active
    int command_ready;             // Flag: 1 if command is ready for module
    int response_ready;            // Flag: 1 if response is ready for main
    char command_buffer[4096];     // Command from main to module
    char response_buffer[4096];    // Response from module to main
    int last_command_timestamp;    // For detecting stale commands
    int last_response_timestamp;   // For detecting stale responses
} ModuleSharedMem;
```

### 1.2 Update model to use shared memory
- Replace pipe creation with shared memory segment creation
- Update `init_model()` and `init_multiple_modules()` functions
- Create shared memory segments using `shmget()` and `shmat()`

### 1.3 Update modules to use shared memory
- Modify external modules to connect to shared memory instead of using stdin/stdout
- Add semaphore-based synchronization
- Maintain same message format for compatibility

## Phase 2: Backward Compatibility Layer

### 2.1 Hybrid approach
- Allow both pipe-based and shared-memory modules to coexist
- Add a configuration flag to specify communication method
- Maintain all existing API functions with same signatures

### 2.2 Wrapper functions
- Create `send_to_module_shared()` and `read_from_module_shared()` functions
- Create `send_to_module_pipe()` and `read_from_module_pipe()` for backward compatibility
- Add detection mechanism to determine communication method

## Phase 3: Migration Path

### 3.1 Module registration
- Add a "protocol" attribute in CHTML modules that specifies communication method:
  `<module protocol="shared">path/to/module</module>`
  `<module protocol="pipe">path/to/module</module>` (default for backward compatibility)

### 3.2 Incremental migration
- Start by creating new shared memory implementations alongside existing pipe code
- Test with simple modules first
- Gradually replace pipe-based modules with shared memory versions

## Implementation Steps

### Step 1: Define shared memory data structures
- Define header files with shared memory structures
- Define shared memory key generation functions
- Create semaphore management utilities

### Step 2: Implement shared memory model functions
- `init_shared_module()` - initialize shared memory communication
- `send_to_shared_module()` - send command to module via shared memory
- `read_from_shared_module()` - read response from module via shared memory
- Update `update_model()` to handle shared memory modules

### Step 3: Update external modules
- Modify modules to connect to shared memory instead of stdin/stdout
- Implement loop to check command_ready flag and process commands
- Implement response writing to response_buffer

### Step 4: Add configuration system
- Add protocol parameter to module specification in CHTML
- Add runtime detection of communication method
- Create connection factory based on protocol

### Step 5: Testing and verification
- Verify shared memory modules work correctly
- Verify mixed (shared+pipe) modules work correctly
- Verify performance improvements
- Ensure graceful fallback to pipe-based modules

## Key Benefits
- Reduced I/O overhead compared to pipes
- Lower latency for communication between main app and modules
- Better performance for high-frequency updates
- Reduced system call overhead

## Key Challenges
- Synchronization complexity (semaphores/mutexes)
- Memory management (proper cleanup of shared segments)
- Process termination handling (orphaned shared memory)
- Maintaining backward compatibility

## Safety Measures
- Implement timeout mechanisms for synchronization
- Add checks for shared memory validity
- Ensure proper cleanup of shared memory segments
- Add fallback to pipe-based communication if shared memory fails

## Testing Strategy
1. Test single shared memory module
2. Test multiple shared memory modules
3. Test mixed pipe and shared memory modules
4. Test module startup/teardown with shared memory
5. Test edge cases (module crashes, shared memory corruption, etc.)
