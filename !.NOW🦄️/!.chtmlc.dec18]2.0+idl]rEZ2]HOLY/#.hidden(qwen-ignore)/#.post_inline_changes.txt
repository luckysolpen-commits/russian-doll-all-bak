# Post-Implementation Changes Documentation: Inline CHTML Rendering

## Overview
This document describes the changes made to implement inline CHTML rendering capability in the CHTML framework, allowing modules to pass and render CHTML elements via shared memory (IPC), similar to JavaScript's inline HTML feature.

## Changes Made

### 1. Model Module (2.model_v0.01.c)

#### A. Enhanced Shared Memory Structure
- **Before**: `ModuleSharedMem` structure contained fields for basic command/response communication
  ```
  typedef struct {
      int module_id;
      int active;
      int command_ready;
      int response_ready;
      char command_buffer[4096];
      char response_buffer[4096];
      int last_command_timestamp;
      int last_response_timestamp;
      int sem_id;
  } ModuleSharedMem;
  ```

- **After**: Added fields for CHTML content communication
  ```
  typedef struct {
      int module_id;
      int active;
      int command_ready;
      int response_ready;
      char command_buffer[4096];
      char response_buffer[4096];
      int last_command_timestamp;
      int last_response_timestamp;
      int sem_id;
      // New fields for CHTML content via shared memory
      int chtml_ready;               // Flag: 1 if CHTML content is ready for main
      int chtml_content_size;        // Size of CHTML content
      char chtml_content[4096];      // CHTML content from module to main
  } ModuleSharedMem;
  ```

#### B. Added CHTML Communication Functions
- **New Functions Added**:
  - `send_chtml_to_main(int module_idx, const char* chtml_content)` - For modules to send CHTML content
  - `receive_chtml_from_module(char* buffer, int buffer_size, int module_idx)` - For main to receive CHTML content
  - `is_chtml_ready(int module_idx)` - To check if CHTML content is available

#### C. Updated update_model() Function
- **Before**: Only processed regular command/response communication
- **After**: Added CHTML content checking with the following logic:
  ```c
  // First, check if there's CHTML content available from this module
  char chtml_buffer[4096];
  int chtml_len = receive_chtml_from_module(chtml_buffer, sizeof(chtml_buffer), mod_idx);
  if (chtml_len > 0) {
      printf("Received CHTML content from shared memory module %d:\n%s\n", mod_idx, chtml_buffer);
      
      // Check if this is a special command (e.g., LOAD_FILE:filename:width:height)
      if (strncmp(chtml_buffer, "LOAD_FILE:", 10) == 0) {
          // Parse the command: LOAD_FILE:filename:width:height
          char filename[256];
          int width, height;
          if (sscanf(chtml_buffer, "LOAD_FILE:%255[^:]:%d:%d", filename, &width, &height) == 3) {
              printf("Module requesting to load CHTML file: %s with dimensions %dx%d\n", filename, width, height);
              
              // Call the view function to load the external CHTML file
              int window_id = load_chtml_file_to_window(filename, width, height);
              printf("Created window (ID: %d) with external CHTML from: %s\n", window_id, filename);
              any_updated = 1; // Mark that content was updated
          } else {
              printf("Error: Could not parse LOAD_FILE command: %s\n", chtml_buffer);
          }
      } else {
          // Regular CHTML content - add to DOM
          int elements_added = add_chtml_to_dom(chtml_buffer);
          printf("Added %d elements to DOM from module %d\n", elements_added, mod_idx);
          any_updated = 1; // Mark that content was updated
      }
  }
  ```

#### D. Added Forward Declarations
- Added declarations for view module functions:
  - `extern int add_chtml_to_dom(const char* chtml_content)`
  - `extern int view_needs_refresh()`
  - `extern int load_chtml_file_to_window(const char* filename, int width, int height)`

## Communication Method Used

### Current Implementation: Shared Memory Only
The implementation exclusively uses **shared memory** communication protocol:
- Modules with `protocol="shared"` attribute use shared memory segments
- CHTML content is transferred via the enhanced `ModuleSharedMem` structure
- Communication is asynchronous through flags (`chtml_ready`) and semaphores
- Higher performance than pipe-based communication for frequent data transfers

### Pipe-Based IPC Method (Not Implemented)
The original framework also supported pipe-based communication:
- Modules with `protocol="pipe"` (or no protocol specified) use stdin/stdout pipes
- Communication is synchronous line-by-line text
- Would require different implementation approach if added

## What Would Be Required for Pipe-Based Implementation

If pipe-based communication was desired for inline CHTML, the following would be needed:

### A. Model Module Changes:
- Parse CHTML content from pipe responses in the pipe-based module handling section (instead of shared memory section)
- Update the command parsing logic to handle CHTML content from stdin/stdout pipes
- Modify `update_model()` function's pipe handling section similar to the shared memory section

### B. Message Format:
- Define a specific message format for sending CHTML content via pipes (e.g., `CHTML:CONTENT_LENGTH:content...`)
- Handle multi-line CHTML content through proper message framing

### C. Performance Considerations:
- Pipe-based communication would be slower than shared memory for large CHTML content
- Would require more system calls for reading/writing
- Might need buffering mechanisms for large content

## Usage Examples

### 1. Module-to-Application Communication:
```c
// Module sends: "LOAD_FILE:example.chtml:400:300"
// Application receives and creates window with example.chtml content
```

### 2. Button Click Event:
```html
<button onClick="createWindow('Title', 'file.chtml', 400, 300)"/>
```

### 3. CHTML File Structure:
```html
<module path="./module/+x/example_file_loader.+x" protocol="shared"/>
```

## Summary
The implementation successfully adds inline CHTML rendering capability while maintaining full backwards compatibility with existing functionality. The system now supports dynamic loading of CHTML content from external files via shared memory communication, matching the JavaScript inline HTML feature as requested.
