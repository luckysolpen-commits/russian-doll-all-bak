# chtml Framework API Documentation

## Overview

The chtml framework is a custom GUI rendering system built on OpenGL that uses an MVC architecture to separate GUI elements from the main application logic. The framework combines HTML-like markup with module integration to create dynamic user interfaces. This document describes how to use the current features and how to extend the framework with new capabilities.

## chtml Markup Language

The chtml framework uses an XML/HTML-like markup language for defining user interfaces. The markup is parsed and rendered by the framework to create GUI elements.

### Document Structure
Each chtml document must have a root `<window>` element that defines the overall window properties:

```xml
<window width="800" height="600" color="#222222">
    <!-- GUI elements go here -->
</window>
```

### Supported Elements

#### Root Elements
- `<window>`: Root container element with overall window properties
  - Attributes: `width`, `height`, `color`

#### Layout Elements
- `<header>`: Header section
  - Attributes: `x`, `y`, `width`, `height`, `color`
- `<panel>`: General container panel
  - Attributes: `x`, `y`, `width`, `height`, `color`
- `<menu>`: Menu container
  - Attributes: `id`, `x`, `y`, `width`, `height`, `label`, `color`, `onClick`
- `<menuitem>`: Menu item
  - Attributes: `x`, `y`, `width`, `height`, `label`, `color`, `onClick`

#### Interactive Elements
- `<button>`: Clickable button
  - Attributes: `id`, `x`, `y`, `width`, `height`, `label`, `color`, `onClick`
- `<checkbox>`: Checkable item
  - Attributes: `id`, `x`, `y`, `width`, `height`, `label`, `checked`
- `<slider>`: Slider control
  - Attributes: `id`, `x`, `y`, `width`, `height`, `min`, `max`, `value`, `step`
- `<textfield>`: Single-line text input
  - Attributes: `id`, `x`, `y`, `width`, `height`, `value`
- `<textarea>`: Multi-line text input
  - Attributes: `id`, `x`, `y`, `width`, `height`, `value`
- `<link>`: Hyperlink to other chtml files
  - Attributes: `x`, `y`, `width`, `height`, `href`, `label`

#### Display Elements
- `<text>`: Text display element
  - Attributes: `id`, `x`, `y`, `value`, `color`
- `<canvas>`: Canvas area for dynamic content
  - Attributes: `id`, `x`, `y`, `width`, `height`, `label`, `color`

#### Module Integration
- `<module>`: Embeds an external module within the interface
  - Content: Path to the module executable

### Example Document
```xml
<window width="800" height="600" color="#222222">
    <header y="0" width="800" height="50" color="#333333">
        <text x="10" y="10" value="My Application" color="#FFFFFF"/>
    </header>
    
    <panel x="10" y="60" width="780" height="530" color="#444444">
        <button id="btn1" x="20" y="20" width="100" height="30" 
                label="Click Me" onClick="button_click_handler" color="#FF0000"/>
        <text id="status_text" x="20" y="60" value="Ready" color="#FFFFFF"/>
        
        <!-- Embed a module -->
        <module>module/+x/enhanced_game_ipc.+x</module>
    </panel>
</window>
```

### Module Element Integration
The `<module>` element allows you to embed external modules within your chtml interface. The content of the module tag should be the path to the module executable that follows the chtml IPC protocol.

When the framework encounters a `<module>` element:
1. It launches the specified executable as a child process
2. Establishes pipes for communication using the standard chtml message format
3. The module can send SHAPE, VAR, and ARRAY messages to update the display
4. User interactions can trigger commands sent to the module

## Flexible Sizing System

The chtml framework supports both fixed and flexible sizing options, similar to CSS percentage-based layouts.

### Fixed Sizing (Pixels)
```xml
<button x="10" y="10" width="100" height="30" label="Fixed Button"/>
```

### Flexible Sizing (Percentage-based)
The framework also supports percentage-based sizing using special values:

```xml
<!-- Full width of parent container -->
<panel x="0" y="0" width="100%" height="50" color="#444444"/>

<!-- Half height of parent container -->
<canvas x="0" y="50" width="800" height="50%" label="Flexible Canvas"/>

<!-- Full window size -->
<panel x="0" y="0" width="100vw" height="100vh" color="#666666"/>
```

### Supported Flexible Units
- `100%`: Percentage of parent container's dimensions
- `vw`: Percentage of viewport (window) width (1vw = 1% of window width)
- `vh`: Percentage of viewport (window) height (1vh = 1% of window height)

### Flexible Sizing Examples
```xml
<!-- Full-width header -->
<header y="0" width="100%" height="50" color="#333333">
    <text x="10" y="10" value="My App" color="#FFFFFF"/>
</header>

<!-- Main content panel taking remaining space -->
<panel x="0" y="50" width="100%" height="calc(100% - 50)" color="#444444">
    <button x="10" y="10" width="20%" height="40" label="Responsive Button"/>
    <canvas x="10" y="60" width="calc(100% - 20)" height="300" color="#CCCCCC"/>
</panel>

<!-- Full screen canvas -->
<canvas id="game_canvas" x="0" y="0" width="100vw" height="100vh" color="#000000"/>
```

This flexible sizing system allows chtml interfaces to adapt to different screen sizes and window dimensions, similar to responsive web design.

## Current Features

### Shape System

The framework supports various 2D and 3D shapes that can be dynamically modified. Shapes are sent from modules to the main application via standardized messages. The system supports both fixed and flexible sizing.

#### Shape Message Format
```
SHAPE;TYPE;ID;X;Y;Z;WIDTH;HEIGHT;DEPTH;R;G;B;ALPHA
```

**Parameters:**
- `TYPE`: Shape type (e.g., "SQUARE", "RECT", "CIRCLE", "TRIANGLE", "TEXT")
- `ID`: Unique identifier for the shape
- `X, Y, Z`: Position coordinates (3D)
- `WIDTH, HEIGHT, DEPTH`: Dimensions (can be fixed values or percentage-based)
- `R, G, B`: RGB color values (0.0-1.0)
- `ALPHA`: Transparency value (0.0-1.0)

**Flexible Sizing in Module Messages:**
Modules can specify dimensions using percentage values that are calculated relative to the canvas or viewport size:

```
SHAPE;SQUARE;player;50.0;100.0;0.0;10%w;10%h;5.0;0.0;0.0;1.0;0.8
SHAPE;CIRCLE;target;100.0;150.0;0.0;5%vw;5%vh;5.0;1.0;0.0;0.0;0.9
```

Where:
- `10%w` = 10% of the parent container width
- `10%h` = 10% of the parent container height
- `5%vw` = 5% of the viewport width
- `5%vh` = 5% of the viewport height

**Supported Shapes:**
- SQUARE: Square shape
- RECT: Rectangle shape
- CIRCLE: Circular shape
- TRIANGLE: Triangular shape
- TEXT: Text rendering

#### Example
```
SHAPE;SQUARE;player;50.0;100.0;0.0;20.0;20.0;5.0;0.0;0.0;1.0;0.8
```

### Variable System

Modules can send variable updates to the main application for display or state management.

#### Variable Message Format
```
VAR;LABEL;TYPE;VALUE
```

**Parameters:**
- `LABEL`: Variable identifier
- `TYPE`: Data type ("int", "float", "string")
- `VALUE`: Variable value

#### Example
```
VAR;score;int;150
VAR;health;float;87.5
VAR;status;string;Game in progress
```

### Array System

Modules can send array data to the main application.

#### Array Message Format
```
ARRAY;LABEL;TYPE;COUNT;VALUES
```

**Parameters:**
- `LABEL`: Array identifier
- `TYPE`: Element data type ("int", "float")
- `COUNT`: Number of elements
- `VALUES`: Comma-separated values

#### Example
```
ARRAY;positions;float;5;10.0,20.0,30.0,40.0,50.0
```

## Future Features Implementation Guide

### 3D Models (.obj + Textures)

To add 3D model support, you'll need to implement the following components:

#### 1. Model Loading in Model Layer (2.model_v0.01.c)

```c
typedef struct {
    int id;
    char model_path[256];        // Path to .obj file
    char texture_path[256];      // Path to texture file
    float x, y, z;              // Position
    float rotation_x, rotation_y, rotation_z;  // Rotation angles
    float scale_x, scale_y, scale_z;          // Scale factors
    char label[64];
    int active;
} Model3D;

// Add function to load .obj files
Model3D* load_obj_model(const char* path);
```

#### 2. View Layer Integration (3.view_v0.01.c)

```c
// Add OpenGL model rendering code
void render_3d_model(Model3D* model);
void load_texture(const char* texture_path);
```

#### 3. Message Format for 3D Models

```
SHAPE;MODEL3D;ID;X;Y;Z;SCALE_X;SCALE_Y;SCALE_Z;ROT_X;ROT_Y;ROT_Z;TEXTURE_PATH;MODEL_PATH
```

#### 4. Module Implementation

Modules would send model commands like:
```
SHAPE;MODEL3D;spaceship;100.0;150.0;0.0;1.0;1.0;1.0;0.0;45.0;0.0;/path/texture.png;/path/spaceship.obj
```

### Audio System

To add audio support, implement both synthesis and playback capabilities.

#### 1. Audio Data Structures (2.model_v0.01.c)

```c
typedef struct {
    int id;
    char type[32];              // "SYNTH" or "FILE"
    char filepath[256];         // For file-based audio
    float frequency;            // For synthesis
    float duration;             // In seconds
    char label[64];
    int active;
} AudioSource;

// Add audio message handling
void play_audio_source(AudioSource* source);
```

#### 2. Audio Message Format

```
AUDIO;TYPE;ID;PARAMS
```

**For synthesized audio:**
```
AUDIO;SYNTH;buzzer;frequency:440.0;duration:1.0;waveform:sine
```

**For file-based audio:**
```
AUDIO;FILE;music;/path/sound.mp3;volume:0.8
```

#### 3. View Layer Integration (3.view_v0.01.c)

```c
#include <AL/al.h>  // OpenAL for audio
#include <AL/alc.h>

void init_audio_system();
void play_sound(const char* filepath, float volume);
void generate_tone(float frequency, float duration, const char* waveform);
```

### Video/GIF/Media Elements

To support video and animated media, implement the following:

#### 1. Media Data Structure (2.model_v0.01.c)

```c
typedef struct {
    int id;
    char media_type[32];        // "VIDEO", "GIF", "IMAGE"
    char filepath[256];
    float x, y, z;
    float width, height;
    int loop;                   // Whether to loop
    float playback_rate;        // For video
    char label[64];
    int active;
} MediaElement;
```

#### 2. Media Message Format

```
MEDIA;TYPE;ID;X;Y;WIDTH;HEIGHT;LOOP;PARAMS;FILEPATH
```

**Examples:**
```
MEDIA;GIF;loading;200.0;200.0;100.0;100.0;1;rate:1.0;/path/anim.gif
MEDIA;VIDEO;intro;0.0;0.0;800.0;600.0;0;volume:0.5;/path/intro.mp4
MEDIA;IMAGE;bg;/path/background.jpg;0.0;0.0;800.0;600.0;0;alpha:0.8
```

#### 3. View Integration (3.view_v0.01.c)

```c
#include <libavformat/avformat.h>  // For video
#include <gif_lib.h>               // For GIF support

void render_video_frame(MediaElement* media);
void render_gif_frame(MediaElement* media);
void render_image(MediaElement* media);
```

## Internal Developer Guide: Extending the Framework

### Adding New Shape Types

1. **Update Shape Structure (2.model_v0.01.c)**
   - Add any new fields to the Shape structure if needed

2. **Update Message Parser (4.controller_v0.01.c)**
   - Add parsing logic for new message formats
   - Ensure proper validation of parameters

3. **Update Renderer (3.view_v0.01.c)**
   - Add OpenGL rendering code for the new shape type
   - Implement proper resource management

4. **Update Module API**
   - Document the new message format
   - Provide example usage in module templates

### Adding New Message Types

To add a new category of messages (like AUDIO, MEDIA, etc.):

1. **Define New Message Format**
   - Use consistent CSV format: `CATEGORY;SUBTYPE;PARAMS...`
   - Include error handling for malformed messages

2. **Update Model Layer (2.model_v0.01.c)**
   - Add data structures to hold the new data
   - Add functions to process/validate the data

3. **Update Controller Layer (4.controller_v0.01.c)**
   - Add parsing logic to handle the new message type
   - Update main message dispatch loop

4. **Update View Layer (3.view_v0.01.c)**
   - Add rendering/processing logic for the new features
   - Ensure proper resource cleanup

### Best Practices for Extensions

1. **Consistent Message Format**
   - Always use semicolon-separated values
   - Maintain consistent parameter order
   - Include error checking for malformed messages

2. **Resource Management**
   - Implement proper cleanup functions
   - Handle resource limits gracefully
   - Use reference counting where appropriate

3. **Performance Considerations**
   - Minimize allocations in render loops
   - Use batch rendering where possible
   - Consider level-of-detail for complex elements

4. **Backwards Compatibility**
   - Don't remove existing message formats
   - Use optional parameters where possible
   - Provide fallbacks for missing features

5. **Flexible Sizing Implementation**
   - When implementing features that require sizing, support both fixed and relative values
   - Parse percentage values in format "XX%w", "XX%h", "XX%vw", "XX%vh"
   - Calculate dimensions relative to parent containers or viewport
   - Implement a sizing calculator function that can convert percentage values to absolute values
   - Example sizing calculator implementation:
   ```c
   float calculate_dimension(const char* value_str, float parent_size, float viewport_size) {
       if (strchr(value_str, '%')) {
           // Handle percentage-based values
           float percent = atof(value_str);
           if (strstr(value_str, "%w")) {
               return parent_size * percent / 100.0f;
           } else if (strstr(value_str, "%h")) {
               return parent_size * percent / 100.0f;
           } else if (strstr(value_str, "%vw")) {
               return viewport_size * percent / 100.0f;
           } else if (strstr(value_str, "%vh")) {
               return viewport_size * percent / 100.0f;
           }
       }
       // Handle fixed values
       return atof(value_str);
   }
   ```

### Module Development Guidelines

When creating modules that use the chtml framework:

1. **Communication Protocol**
   - Send updates at consistent intervals
   - Use buffering to avoid overwhelming the main application
   - Implement proper error handling

2. **State Management**
   - Maintain internal state independently
   - Send only necessary updates to the main application
   - Use the variable system for persistent data

3. **Flexible Sizing Support**
   - Modules can send percentage-based dimensions in SHAPE messages
   - Use formats like "10%w" for percentage of parent width
   - Use formats like "5%vw" for percentage of viewport width
   - The main application will calculate actual pixel dimensions

4. **Performance**
   - Minimize computation in the module
   - Use efficient algorithms for game logic
   - Consider separate threads for intensive operations
