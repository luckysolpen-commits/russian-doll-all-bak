# CHTML Image Support Implementation Plan

## Problem Identified
Previous image implementation broke UI interactivity (buttons, sliders, etc. not responding to clicks) due to several issues:
1. Texture loading happening during render cycle in draw_element function
2. OpenGL state changes during rendering interfering with UI detection
3. Adding new fields to UIElement struct causing memory layout issues
4. Complex texture management during rendering process

## Correct Approach - Safe Implementation

### Phase 1: Minimal UIElement Extension
- Add only essential image fields to UIElement struct ensuring backward compatibility
- Keep texture loading separate from rendering cycle
- Maintain original memory layout as much as possible

### Phase 2: Proper Texture Loading Architecture
- Implement texture preloading during parse phase (not render phase)
- Load all image textures when CHTML is parsed initially
- Store textures in element data during parsing, not during rendering

### Phase 3: Safe Rendering Implementation
- Only bind and draw already-loaded textures during rendering
- Never attempt to load textures during draw_element
- Properly manage OpenGL state saves/restores to avoid interfering with UI element detection

### Phase 4: Image Tag Integration
- Add "img" and "image" element types to parser safely
- Handle src attribute parsing without disrupting other element types
- Maintain proper parent-child stack relationships for nested elements

## Incorrect Approaches to Avoid

### DO NOT:
1. Load textures during the draw_element/rendering cycle - this causes timing issues with UI
2. Change UIElement struct significantly without ensuring all files (controller, model, view) are compatible
3. Modify OpenGL state permanently without proper save/restore (glPushAttrib/glPopAttrib)
4. Attempt texture operations inside the render loop rather than preloading
5. Call file I/O functions during rendering functions
6. Have conditional texture loading happening in draw functions
7. Change the sequence of element type checks in draw_element without maintaining original functionality

### DO:
1. Preload textures when the CHTML is parsed, before any rendering
2. Have draw_element only use preloaded, already-available textures
3. Use proper OpenGL state management with glPushAttrib/glPopAttrib
4. Test UI functionality after each change to image support
5. Maintain backward compatibility by keeping all original element rendering code intact
6. Add new functionality in a way that's isolated from existing working functionality

## Implementation Strategy

### Step 1: Extend UIElement Struct Safely
- Add minimal image-specific fields at the end of existing structure
- Ensure no disruption to existing field positions

### Step 2: Update Parser for Image Tags
- Add support for img/image tags in parse_chtml function
- Handle src attribute parsing specifically for image elements
- Set up parent-child relationships properly in stack logic

### Step 3: Implement Preloading System
- Add texture loading call after parsing completes
- Load all referenced images into memory and bind to texture IDs
- Stash texture IDs in the UIElement data during parsing

### Step 4: Safe Rendering
- In draw_element, only draw textures that are already loaded
- Use proper OpenGL state management
- Ensure all original element rendering logic is preserved

This approach separates loading from rendering, preventing the timing and resource conflicts that broke UI interactivity.
