The CHTML framework image support implementation has broken UI interactivity functionality. All UI elements (buttons, sliders, checkboxes, text fields, etc.) are no longer responding to mouse clicks, keyboard events, or other interactions, even though the canvas module interactions still work. The image rendering itself works correctly (images display properly and are right-side up).

After recent implementation attempts, the problem is likely related to:

1. Texture loading during the render cycle - calling load_texture_from_file in draw_element may cause timing issues that interfere with input handling
2. OpenGL context state - enabling textures/blend modes in draw_element might affect hit detection or coordinate calculations even with glPushAttrib/glPopAttrib
3. The placement of image rendering in the draw_element function - it may be affecting the overall rendering flow
4. Memory/resource allocation during rendering - the texture loading might be consuming resources needed for UI event processing

Focus on:
- Avoiding any file loading or heavy processing during the render loop
- Ensuring that OpenGL state changes needed for texturing do not interfere with UI element rendering or detection
- Potentially pre-loading textures during the parse phase rather than during rendering
- Checking that the addition of the image rendering section doesn't affect coordinate calculations for other UI elements
- Making sure that adding the image type handlers doesn't disrupt the flow of the draw_element function

The fix should restore full UI interactivity while keeping all the implemented image features. Consider implementing a preloading system that loads textures when the CHTML is parsed, rather than during rendering.
