Why Loss Goes to NaN

Division by Zero: expf(x - max_logit) in the softmax can underflow to 0 if x - max_logit is a huge negative number, making sum zero, then logits[i] /= sum becomes 0/0 = NaN.

Log of Zero/Negative: loss = -logf(logits[targets[seq_len - 1]] + 1e-10f) can hit NaN if logits[targets[seq_len - 1]] is negative or too close to zero (despite the 1e-10f guard).

Exploding Values: Large weights or inputs can make logits[i] explode (e.g., via expf), leading to infinity, then NaN in subsequent ops.

Numerical Instability: Floating-point ops like expf or sqrtf can overflow/underflow with extreme values, especially with LEARNING_RATE = 0.384f pushing weights fast.

Your vanilla code looks solid, but with LEARNING_RATE = 0.384f (pretty high) and no explicit guards beyond 1e-10f, itвҖҷs prone to these issues, especially on a small dataset where gradients might spike.

What is Mixture-of-Experts (MoE)?
In an MoE system:
Experts: Multiple sub-networks (e.g., neural nets), each specialized for a task or domain (in your case, one per subject like Mathematics, Physics, etc.).

Gating Mechanism: A lightweight network or function that decides which expert(s) to use for a given input, often outputting a probability distribution over experts.

Execution: The input is routed to one or more experts (top-k selection), and their outputs are combined (e.g., weighted sum).

Your current system has 10 expert models (one per subject in curricula/<subject>/), and ollama-api.c computes weights that llm.c uses via RL (epsilon-greedy) to pick one. We can evolve this into a DeepSeek3-like MoE by:

Optionally training the gating mechanism (though weвҖҷll start with a static version for simplicity).
