# Mars StreetRace & 3D Tactics Board Game Compatibility Report

## Overview
This report outlines the compatibility requirements and suggested modifications to ensure seamless synchronization between the Mars StreetRace text-based stock market simulator and the 3D Tactics Board Game featuring human "pieces" as employees/traders.

## Key Constraint: Sequential Operation
**Important Note**: For single-player and local multiplayer modes, these systems will never run simultaneously. Only one system will be active at any given time. This significantly simplifies our synchronization requirements, as we only need to ensure proper state persistence and restoration between sessions of different interfaces.

## Time File Usage Pattern
**Critical Clarification**: The `wsr_clock.txt` file is only read/written during save and load operations, not during continuous gameplay. When a user switches between interfaces:
- The current system writes the final game time to `wsr_clock.txt` during save/shutdown
- The new system reads the game time from `wsr_clock.txt` during load/startup
- During active gameplay, each system manages its own time advancement internally

## Recommended Modifications

### For Mars StreetRace (Text Version)

**1. Save/Load Integration**
```
- On save: Write current game time to wsr_clock.txt
- On load: Read initial game time from wsr_clock.txt
- Implement internal time advancement during active gameplay
```

**2. Time Advancement Logic**
```
- Advance time based on actual game actions (trades, decisions, etc.)
- Maintain time internally during active session
- Only update file during save operations
```

### For 3D Tactics Board Game

**1. Save/Load Integration**
```
- On load: Read starting game time from wsr_clock.txt
- On save: Write final game time to wsr_clock.txt
- Update UI to display current game time during active session
```

**2. Time Advancement Logic**
```
- Advance time based on game actions (piece movement, work, etc.)
- Maintain time internally during active session
- Only update file during save operations
```

**3. Turn Synchronization**
```
- Map 3D board actions to game turn advancement
- Ensure turn-based actions in 3D game trigger equivalent stock market events
- Synchronize current player/entity between systems
```

## Synchronization Protocols

### 1. Save/Load Protocol
```
- During save: Write complete time state to wsr_clock.txt
- During load: Validate and apply time state from wsr_clock.txt
- Include timestamp and system identifier in save data
```

### 2. Time Consistency
```
- Each system advances time according to the same rules during gameplay
- Time advancement rules must be identical between both systems
- Use same time speed settings (sec, min, hour, day) across systems
```

## Cross-System State Mapping

### Mars StreetRace State → 3D Board State (on save/switch)
- Player portfolios and cash: Transfer to 3D system for employee asset tracking
- Corporate stock prices and data: Maintain consistency for 3D system
- News events and economic state: Preserve for 3D system awareness
- Employee/piece positions and statuses: Update 3D system when changed via Management submenu

### 3D Board State → Mars StreetRace State (on save/switch)
- Employee status and positions: Reflect in market activity in text system
- In-game economic activities: May influence stock prices and market events
- Game time and progression: Maintain consistent timeline between systems

### Management Submenu Integration
- Create submenu under "15. Management" titled "Employee Pieces" or "Board Pieces"
- Display list of all pieces (employees) from the shared data file, populated from pieces on the 3D board
- For each piece, show current XYZ position, status, and available actions
- Actions should include: Update Position (X,Y,Z), Feed, Sleep, Work, Assign Task
- When position is updated in text-based system, update the 3D system's position file
- **Critical Feature**: When selecting a piece in the text menu, navigate to the 3D HTML board with the selector already positioned and active on the selected piece
- Consider how 3D pieces (soldiers → employees, water → resources/locations) map to economic entities

## Implementation Insights from 3D System Analysis

### 3D System Architecture
Based on analysis of the 3D Tactics Board Game, the system uses:
- CSV files (`pieces.csv`, `selector_state.csv`) as data storage
- PHP endpoints (`get_pieces.php`, `save_pieces.php`, `get_selector.php`, `save_selector.php`) for data operations
- Client-side Three.js for rendering and game logic
- File-based persistence between sessions

### Integration Recommendations

**1. Unified Data Storage**
- Extend the 3D system's CSV-based approach to include time data
- Consider using `game_state.csv` or similar to store all shared state including time
- This would be more consistent with the 3D system's architecture than continuing with the text-based wsr_clock.txt

**2. Shared Time Storage Format**
Instead of or in addition to `wsr_clock.txt`, consider implementing:
```
game_state.csv
- year, month, day, hour, minute, second
- current_player_index
- game_mode (text/3d)
- last_save_timestamp
```

**3. Data Structure Alignment**
- Ensure both systems can read/write the same data formats
- The 3D system uses type, x, y, z coordinates for pieces - these could map to employee/workplace locations in the economic simulation
- Consider how 3D pieces (soldiers, water) map to economic entities or employee types in Mars StreetRace

### File Operations Synchronization

The 3D system already implements:
- `get_pieces.php` and `save_pieces.php` for piece state persistence
- Similar functionality should be created for time and game state
- This approach aligns better with the 3D system's architecture than the current text-based file approach
