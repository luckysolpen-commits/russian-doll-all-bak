## BUG 1: XLECTOR renders as "?" on map, does not move
**Severity:** BLOCKER ‚Äî directly caused by our Phase 2 refactoring
**Status:** FIXED (2026-04-05)

### Symptom
The xlector piece renders as `?` instead of `X` on the game map. Movement commands fire (move_entity.+x) but the icon never updates.

### Root Cause
Two C source files map `type=` strings to icon characters. Both still check `type=selector` but not `type=xlector`:

1. **`pieces/apps/playrm/ops/src/render_map.c` line 237** ‚Äî `infer_icon_from_type()`:
   ```c
   else if (strcmp(type, "selector") == 0) snprintf(out_icon, out_sz, "X");
   ```
   Since xlector state.txt now has `type=xlector`, this comparison fails ‚Üí falls through to `else` ‚Üí `"?"`.

2. **`pieces/apps/playrm/ops/src/create_piece.c` lines 69 and 79** ‚Äî type-to-piece-ID and type-to-icon helpers:
   ```c
   if (strcmp(type, "X") == 0 || strcmp(type, "selector") == 0) return "selector";
   // ...
   if (strcmp(type, "X") == 0 || strcmp(type, "selector") == 0) return "X";
   ```

### Fix Applied (2026-04-05)
Clean cut ‚Äî changed `"selector"` ‚Üí `"xlector"` in:
- `render_map.c` ‚Äî type‚Üíicon mapping, piece_id comparisons, xlector_idx variable, render ordering
- `move_entity.c` ‚Äî collision bypass check
- `scan_op.c` ‚Äî piece state reads, directory skip
- `inspect_tile.c` ‚Äî directory skip
- `move_player.c` ‚Äî collision bypass
- `playrm_module.c` ‚Äî hotkey guard, '9' return-to-xlector

All recompiled. No backward compatibility.

## BUG 4: "No Methods" shown when xlector is active (used to show selector methods)
**Severity:** BLOCKER ‚Äî caused by Phase 2 refactoring
**Status:** OPEN

### Symptom
When active target is xlector, the UI shows `[No Methods]` instead of xlector's methods (scan, collect, place, inventory, etc.). The methods ARE in the PDL (`xlector.pdl`) but playrm_module isn't finding them.

### Root Cause
playrm_module reads methods from the PDL by piece_id. The piece_id changed from `selector` to `xlector` (both directory name and PDL `piece_id` field). The hotkey guard in `playrm_module.c` now checks `strcmp(active_target_id, "xlector") != 0` ‚Äî this correctly EXCLUDES xlector from method hotkeys (keys 2-8), which is the OLD behavior where selector was excluded because it's a cursor, not an entity with methods.

But the UI method list ("scan", "collect", etc.) is generated separately ‚Äî likely from the PDL or a hardcoded layout. The layout/chtpm may have method names that reference `selector` explicitly, or the method-fetching code reads PDL using `selector` as the piece_id.

### Files to Check
- `pieces/apps/playrm/layouts/game.chtpm` ‚Äî method buttons may reference `selector`
- `pieces/apps/playrm/plugins/playrm_module.c` ‚Äî `get_method_name()` or similar PDL reading
- `pieces/apps/playrm/ops/src/interact.c` or similar ‚Äî method dispatch for xlector

## BUG 7: Kill script regex broken ‚Äî old processes not killed
**Severity:** BLOCKER ‚Äî causes double-move, stale state, method inconsistency
**Status:** FIXED (2026-04-05)

### Root Cause
The ERE regex `pieces/.*/\+x/.*\.+x` in `run_orchestrator.sh` and `kill_all.sh` was broken:
- In ERE, `\.+x` means "one or more dots followed by x" (NOT "literal dot, literal plus, x")
- `/\+x/` was being interpreted as "one or more `/` then x/" (NOT "literal `/+x/`")
- **No binary paths matched the regex** ‚Äî old processes were never killed
- Result: multiple instances running simultaneously ‚Üí double-move, stale keys, method inconsistency

### Fix Applied (2026-04-05)
Changed regex to use character classes (avoids backslash escaping hell):
- `pieces/.*/[+]x/.*[.][+]x` ‚Äî correctly matches `pieces/*/+x/*.+x`
- `projects/.*/[+]x/.*[.][+]x` ‚Äî correctly matches `projects/*/+x/*.+x`
- Fixed in: `pieces/buttons/shared/run_orchestrator.sh` and `pieces/os/kill_all.sh`
- Added `zombie_ai` to `kill_all.sh` surgical kill list

## BUG 8: Methods show "No Methods" intermittently in fuzz-op
**Severity:** BLOCKER ‚Äî caused by Phase 2 refactoring
**Status:** FIXED (2026-04-05)

### Root Cause
`projects/fuzz-op/project.pdl` said `player_piece = fuzzpet`. The loader wrote `active_target_id=fuzzpet` to the manager state. The parser's first frame read `active_target_id=fuzzpet` and tried to load methods from `projects/fuzz-op/pieces/fuzzpet/piece.pdl` ‚Üí doesn't exist ‚Üí "No Methods". The manager then corrected to `xlector`, and subsequent frames showed methods correctly.

### Fix Applied (2026-04-05)
- Changed `player_piece` in `projects/fuzz-op/project.pdl` from `fuzzpet` to `xlector`
- Parser now reads correct `active_target_id=xlector` from frame 1
- Methods load correctly from `projects/fuzz-op/pieces/xlector/piece.pdl`

## BUG 3: gl-os 404 "Project Not Found"
**Severity:** MINOR ‚Äî pre-existing, not caused by xlector refactor
**Status:** OPEN (fix: remove from project loader)

### Symptom
Selecting "gl-os" in the project loader shows a 404 error page.

### Root Cause
`projects/gl-os/` directory does not exist. The project loader (`loader_module.c`) scans `projects/` for directories with `project.pdl` files. However, the stale `loader.pdl` (at `pieces/apps/playrm/loader/loader.pdl`) contains a hardcoded METHOD entry for `gl-os` from a previous scan ‚Äî likely from when `projects/gl-os/` existed. At startup, `update_loader_pdl()` should regenerate the file, but it only lists projects that have a `project.pdl` file.

Wait ‚Äî gl-os IS in the loader list because `update_loader_pdl()` generates it dynamically. That means there IS a `project.pdl` somewhere for gl-os, OR the loader isn't being regenerated. The stale `loader.pdl` shows: `METHOD | gl-os | LOAD_PROJECT:gl-os`.

### Fix (separate ticket)
- Remove gl-os from the project loader ‚Äî it's an app, accessed from main menu
- OR: Ensure `update_loader_pdl()` excludes gl-os (add a skip condition)
- OR: Create `projects/gl-os/project.pdl` that routes to `pieces/apps/gl_os/layouts/desktop.chtpm`

### User Note
gl-os IS accessible from the main CHTPM menu (option 6). Only the project loader shows a broken entry.
