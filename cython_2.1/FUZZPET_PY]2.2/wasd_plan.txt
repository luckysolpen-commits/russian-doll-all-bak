COMPLETE TPM REFACTOR PLAN - UNIFIED INPUT SYSTEM
===============================================

GOAL
----
Completely refactor the FuzzPet application to follow the TPM-compliant keyboard architecture from the reference implementation. Instead of having separate input systems, ALL input will flow through the unified piece-driven system:

1. All keyboard input (WASD, arrows, numbers, letters) goes to pieces/keyboard/history.txt
2. A piece-based system processes all input events 
3. Different pieces handle different types of input (mover for WASD, commands for number keys)
4. Eliminate the split between immediate input and enter-required input

KEY INSIGHTS FROM REFERENCE IMPLEMENTATION
----------------------------------------
1. UNIFIED INPUT PIPELINE: All input flows through pieces/keyboard/history.txt - no bifurcation between different input types

2. PIECE-DRIVEN PROCESSING: Different pieces (menu_control, keyboard, etc.) process appropriate inputs based on event listeners

3. PLUGIN ARCHITECTURE: Plugins handle different aspects (keyboard, control, display, piece_management) coordinated by the engine

4. HISTORY.TXT AS CENTRAL INPUT QUEUE: Input is written to history file, then processed by listening pieces

5. IMMEDIATE VS ENTER DISTINCTION: Done through piece logic, not different input systems

CURRENT PROBLEMS TO FIX
----------------------
1. BIFURCATED INPUT: Current system splits between Enter-requiring main loop and potential immediate input
2. MANUAL INPUT LOOP: Using input() instead of unified piece-driven system
3. MISSING KEYBOARD PIECE: No centralized keyboard management like reference
4. INCONSISTENT EVENT FLOW: Not all input follows the ledger-event pattern

INPUT FLOW IN NEW SYSTEM
-----------------------
1. User presses key (any key: WASD, 1-6, Enter, etc.)
2. Keyboard plugin detects and writes key code to pieces/keyboard/history.txt
3. Control plugin polls history.txt and processes new entries
4. Control plugin determines if key is movement (WASD/arrow) or command (1-6/letters) 
5. Appropriate piece (mover or command_processor) receives event
6. Piece method executes and updates state
7. State changes trigger further events if needed
8. All activity logged to master ledger for audit

BENEFITS OF THIS APPROACH
------------------------
1. FULL TPM COMPLIANCE: All input flows through pieces, following proper architecture
2. NO INPUT BIFURCATION: Unified system handles all input types consistently  
3. EXTENSIBILITY: Easy to add new input types or behaviors
4. MAINTAINABILITY: Cleaner separation of concerns
5. AUDITABILITY: All input processing traceable through ledger and piece states
6. SCALABILITY: Can easily add multiple input devices or remote control
7. CONSISTENCY: Same patterns used throughout system
