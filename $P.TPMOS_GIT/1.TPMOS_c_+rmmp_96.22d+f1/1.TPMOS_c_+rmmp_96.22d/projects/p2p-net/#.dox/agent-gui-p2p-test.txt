CONCEPT: EMAIL-STYLE MESSAGING ON LOCAL SUBNET
================================================================================

Instead of real-time chat, P2P-NET uses an email-like messaging system:
- Messages are stored per-subnet
- Users can have multiple accounts (wallets) on SAME TERMINAL
- Messages persist until read/deleted
- Works like email: compose, send, receive, inbox

KEY POINT: YOU CAN TEST THIS IN ONE TERMINAL!
────────────────────────────────────────────
You don't need two terminals or two machines.

Think of it like having TWO EMAIL ACCOUNTS:
  - Account 1: player1@gmail.com
  - Account 2: player2@gmail.com

You can send emails between them from the SAME computer.

In P2P-NET:
  - Account 1: Wallet "abc123..." (like player1@gmail.com)
  - Account 2: Wallet "def456..." (like player2@gmail.com)

Both wallets exist on your machine. You switch between them in the GUI.
Messages are stored in separate inbox files per wallet.

Why Email-Style?
- No need for both users online simultaneously
- Messages persist across sessions
- Can send to yourself (2 accounts on same machine) ← TESTING!
- Audit trail of all communications
- Fits TPM file-based architecture perfectly

================================================================================
HOW IT WORKS (Single Terminal, Two Accounts)
================================================================================

YOU (One User, One Terminal, Two Wallets):
──────────────────────────────────────────

Wallet 1: "abc123..." (player1)
Wallet 2: "def456..." (player2)

Both wallets are on the SAME "chat" subnet.

Step 1: Switch to Wallet 1 in GUI
  → Press [6] (Switch Account)
  → Select: abc123... (player1)
  → GUI shows: "Account: player1 (abc123...)"

Step 2: player1 sends message to player2
  → Press [1] (Compose)
  → Input: to=def456..., subject="Hello!", body="Test"
  → Message stored in: chat/inbox/def456.../msg_001.txt

Step 3: Switch to Wallet 2 in GUI
  → Press [6] (Switch Account)
  → Select: def456... (player2)
  → GUI shows: "Account: player2 (def456...)"

Step 4: player2 checks inbox
  → Press [5] (Check Inbox)
  → Sees 1 unread message from player1
  → Frame shows: "INBOX (1 unread / 1 total)"

Step 5: player2 reads message
  → Press [2] (Read Message)
  → Sees: "From: player1, Subject: Hello!, Body: Test"

Step 6: player2 replies to player1
  → Press [3] (Reply)
  → Input: body="Hi back!"
  → Message stored in: chat/inbox/abc123.../msg_001.txt

Step 7: Switch back to Wallet 1
  → Press [6] (Switch Account)
  → Select: abc123... (player1)
  → Press [5] (Check Inbox)
  → Sees reply from player2!

ALL DONE IN ONE TERMINAL! No second machine needed.

================================================================================
FILE STRUCTURE (Email Protocol)
================================================================================

projects/p2p-net/pieces/chat/
├── subnets/
│   ├── chat.txt              ← Subnet config
│   └── general.txt           ← Subnet config
├── inbox/
│   ├── <wallet_address>/     ← Each wallet has an inbox
│   │   ├── msg_001.txt       ← Message 1
│   │   ├── msg_002.txt       ← Message 2
│   │   └── status.txt        ← Unread count, last checked
│   └── <wallet_address>/     ← Another wallet's inbox
│       └── msg_001.txt
└── sent/
    └── <wallet_address>/     ← Sent messages archive
        ├── msg_001.txt
        └── status.txt

================================================================================
MESSAGE FILE FORMAT
================================================================================

msg_001.txt:
```
# P2P Message
msg_id=msg_001
from=abc123def456...
to=def456ghi789...
subnet=chat
timestamp=2026-04-02 12:34:56
subject=Hello!
read=0
body=Hi there! This is a test message.
     You can have multiple lines.
     Just like email!
```

status.txt:
```
unread_count=3
last_checked=2026-04-02 12:30:00
total_messages=15
```

HOW TO TEST (M18 Principles)
================================================================================

TEST 1: Send Message Between Two Accounts (Same Machine)
────────────────────────────────────────────────────────
Setup:
  - Two wallets: player1 (abc123...), player2 (def456...)
  - Both joined "chat" subnet

Steps:
  1. Launch p2p-net GUI
  2. Press [1] (Compose Message)
  3. Input: to=def456..., subject="Hello!", body="Test message"
  4. Press Enter to send

Expected Frame:
  [RESP]: Message sent to player2

State Assertions:
  ✓ File created: chat/inbox/def456.../msg_001.txt
  ✓ File created: chat/sent/abc123.../msg_001.txt
  ✓ player2 status.txt: unread_count=1

TEST 2: Check Inbox (Player2)
─────────────────────────────
Steps:
  1. Switch to player2 account (or launch second instance)
  2. Press [5] (Check Inbox)

Expected Frame:
  INBOX (1 unread / 1 total)
  [>] 1. From: player1       Subject: Hello!

State Assertions:
  ✓ chat/inbox/def456.../status.txt: unread_count=1
  ✓ Frame shows 1 message

TEST 3: Read Message
────────────────────
Steps:
  1. Press [2] (Read Message)
  2. Input: msg_id=msg_001

Expected Frame:
  MESSAGE PREVIEW
  From: player1 (abc123...)
  Subject: Hello!
  ─────────────────────────────────────────────────────
  Test message

State Assertions:
  ✓ msg_001.txt: read=1
  ✓ status.txt: unread_count=0

TEST 4: Reply to Message
────────────────────────
Steps:
  1. Press [3] (Reply)
  2. Input: body="Hi back!"

Expected Frame:
  [RESP]: Reply sent to player1

State Assertions:
  ✓ New file: chat/inbox/abc123.../msg_002.txt
  ✓ New file: chat/sent/def456.../msg_002.txt
  ✓ player1 status.txt: unread_count=1

FRAME HISTORY WALKTHROUGH (What You'll See)
================================================================================

When we reach GUI testing, I'll walk you through:

1. INITIAL FRAME
   - P2P Message Center layout
   - Shows your account, subnets, inbox count
   - Frame file: pieces/display/current_frame.txt

   - Frame updates with [RESP]: Message sent

   - Frame history shows state change

3. AFTER CHECKING INBOX
   - Frame shows message list
   - Unread count visible
   - Frame history shows navigation

4. AFTER READING MESSAGE
   - Frame shows full message content
   - Read status updated
   - Frame history shows message view

5. AFTER REPLY
   - Frame confirms reply sent
   - Original sender's inbox now has 1 unread
   - Frame history shows complete cycle

Each step will include:
- Key injection command used
- Frame before/after diff
- State file assertions
- Timestamp from frame history
